# 迭代精准性：抖音场景判断

> 第四步：结合产品定位和成本约束，判断抖音特效场景下"迭代修改"这件事应该怎么做、做到什么程度。

---

## 判断锚点

- 产品定位：抖音加号入口 AI 特效，普通用户快速产出社交内容
- 核心约束：**不收费**（每次迭代 = 额外一次推理成本，平台承担）
- 技术现实：现有特效 = 端侧实时渲染（参数调整零成本）；AI 特效 = 云端生成（每次修改都是一次完整推理）
- 用户画像：不懂 prompt、不会画 mask、耐心有限（3 次以内出片或放弃）
- 兄弟产品边界：即梦/剪映 = 多轮精修、精细控制；抖音特效 = 快速、自动化、轻量

---

## 核心判断

### 判断 1：抖音场景下"迭代"的定义不是多轮精修，是"不满意→换一个"

专业工具（即梦、Photoshop）里的迭代是：出图 → 调颜色 → 改构图 → 换元素 → 调光影 → 5-10 轮打磨出作品。这不是抖音用户的行为模式。

**抖音用户的"迭代"更接近**：
- 点一下 → 看结果 → 不喜欢 → 再点一下（换个版本）
- 大概满意 → 想微调一处 → 改完就发

本质区别：**专业用户的迭代是"收敛到心中标准"，抖音用户的迭代是"快速过滤到一个够好的"。**

产品含义：
- "重新生成"（换 seed 出变体）覆盖抖音 60-70% 的"迭代"需求
- 真正的"精准编辑"只在"大概满意但有一处想改"时才需要
- 产品的第一优先级是**提高首次出图满意度**（减少迭代需求），而不是优化迭代精度

### 判断 2："重新生成"比"精准编辑"更适合作为默认交互

| 维度 | 重新生成（换 seed） | 精准编辑（改局部） |
|------|---------------------|---------------------|
| 用户心智成本 | 零（点一个按钮） | 中（要说清改哪里） |
| 技术复杂度 | 低（同 prompt 换 seed） | 高（Kontext / inpainting） |
| 单次推理成本 | 等于首次生成 | 等于或高于首次生成 |
| 失败风险 | 低（每次都是完整新图） | 中（联动漂移、欠/过编辑） |
| 用户预期管理 | 容易（"换一个"不承诺保留什么） | 难（"改这里"隐含"其他不动"的预期） |

**结论**：默认交互应该是**"不满意？看看其他版本"（多版本选择）**，而不是"不满意？告诉我改哪里"（编辑模式）。

原因：
1. 重新生成不产生"改了 A 结果 B 也变了"的预期落差——用户知道每次都是新的
2. 多版本选择是零学习成本交互（用户只需要"选"不需要"描述"）
3. 成本可预测——每次就是一次标准推理，不存在多轮累积

**但这不意味着不做编辑。** 当用户已经从 4 个变体中选了一个，想做最后微调时，编辑能力决定了"选完到发布"之间的转化率。

### 判断 3：如果要支持编辑，技术路线的选择

三条路线在抖音场景下的适用性排序：

**首选：Token concat（Kontext 类）—— 覆盖主流编辑需求**

理由：
- 用户门槛最低（说人话即可）
- 多轮漂移最小（Kontext 论文专门优化）
- 架构天然支持"在这张图基础上继续"的用法
- 单次推理 3-5 秒，成本 ~0.05-0.1 元，字节算力下可承受

适合的编辑场景：换颜色、加/去元素、换表情、调风格

**次选：Auto-mask + Inpainting —— 覆盖精确移除/替换**

理由：
- "移除某个东西"用 token concat 会留痕——区域内需要完全重画
- SAM-2 + 语义理解可以做到"用户点一下就选中对象"，不需要手动画 mask
- 成本低（inpainting 模型通常比 12B Kontext 小）

适合的编辑场景：擦除水印/杂物、替换特定小区域

**不选：手动 mask + 参数调节**

理由：手机端画 mask 是反人性的，抖音用户不会用。这是即梦/剪映的领域。

**产品层的路由逻辑**（用户无感）：

```
用户说"改一下"
    │
    ├── MLLM 判断编辑类型
    │
    ├── "去掉 XX" / "擦掉 XX" → Auto-mask + Inpainting
    │
    └── 其他（改颜色/加东西/换风格/调氛围）→ Token concat
```

### 判断 4：迭代次数的成本控制——每轮迭代的边际成本决定产品形态

**数字算清楚**：

假设：
- 抖音 AI 特效日均使用 800 万次（1% 渗透）
- 平均每次使用产生 1.5 次"重新生成" + 0.5 次"编辑"
- 重新生成成本 ~0.03 元（蒸馏后 4 步推理）
- 编辑成本 ~0.08 元（Kontext 级 12B）

日均迭代成本：800 万 × (1.5 × 0.03 + 0.5 × 0.08) = **6800 万元/天**

这个数字**不可承受**。必须有产品层约束：

| 策略 | 效果 | 实现方式 |
|------|------|----------|
| 提高首次满意度 | 减少迭代次数 50%+ | 意图澄清、参数推导、一次出多版本 |
| 重新生成用蒸馏模型 | 单次降到 0.01-0.02 元 | 变体生成走轻量模型（质量可接受） |
| 编辑限频 | 每人每天 N 次精准编辑 | 免费额度 + 到达限制后只给"重新生成" |
| "选"代替"改" | 把编辑需求转化为选择需求 | 出 4 个变体让用户挑，减少进入编辑的比例 |

**待讨论**：上面的数字是粗估。实际单次成本取决于字节内部推理架构和 batching 效率，入职后需要拿到真实数字重新算。

### 判断 5：与端侧特效的体验落差——用户会拿"实时调参"的预期来衡量 AI 特效

现有抖音特效（大眼瘦脸、滤镜、AR 贴纸）的参数调整是**实时零成本**的：拖个滑块立刻看到变化，不满意接着调，拍一百次也不花一分钱。

AI 特效每次迭代要等 3-8 秒 + 花一次钱。**用户不会理解这个差异——他们只会觉得"这个特效怎么这么慢还不能调"。**

产品设计上的缓解策略：

1. **能实时预览的尽量实时预览**：如果编辑只涉及颜色/滤镜/强度，能不能做一个"预览级"的快速估算（低分辨率/低步数），确认后再出高清？（待讨论：技术可行性取决于 DiT 模型是否支持渐进式去噪预览）

2. **明确区分"端侧可调"和"需要重新生成"的参数**：有些调整（亮度、对比度、裁切、简单滤镜）可以在生成结果上端侧后处理，不需要重新跑模型。产品应该把这些"免费调整"和"需要重新生成的修改"在 UI 上区分开。

3. **管理预期**：进入 AI 特效前告诉用户"生成需要几秒"，不要让用户误以为和端侧特效一样是实时的。

### 判断 6：什么场景必须支持迭代 vs 什么场景"一次出结果"就够

**一次出结果就够的场景（占 70%+）：**
- 风格化滤镜（一键变漫画、一键变油画）
- 背景替换（选预设场景）
- 贴纸/元素叠加
- 整体氛围调整

共同特征：变化是全局性的、结果的可接受范围宽（"差不多就行"）、不涉及精确局部控制。

**必须支持迭代的场景（占 30%）：**
- 人物形象编辑（换装、换发型）——用户对自己的脸极其敏感
- 多元素组合创作（"我想要 A + B + C"）——一次很难全中
- 故事性内容（连续多帧需要一致）——前后帧的人物必须一致

共同特征：用户对"具体某一处"有明确预期，且对偏差零容忍。

**产品策略**：
- "一次就够"的场景：不开放编辑入口，减少选择焦虑，降低成本
- "必须迭代"的场景：提供编辑入口，但默认先给多版本选择，选完再开放精调
- 判断标准：**用户是否能说清"哪里不对"**——能说清的才需要编辑，说不清的给他重新生成

### 判断 7：多轮漂移的产品兜底——模型做不到的事，产品层必须接住

所有 AI 编辑模型都有多轮漂移问题（Kontext 减缓但没消除）。在抖音场景下：

**最低限度的产品兜底**：
1. 自动保存每一轮结果（用户无需手动保存）
2. 提供"回到上一版"的能力（一键回退）
3. 从任意历史版本"重新编辑"（不强制线性累加）

**理想状态**（待讨论可行性）：
- A/B 对比视图：当前版 vs 原图 / vs 上一版
- 一次编辑出 2 个变体，让用户选漂移更小的那个
- 第 3 轮编辑后主动提示"是否要从原图重新开始？"（避免用户在漂移后的图上继续累积）

**不做什么**：
- 不做树状分支编辑——太重、学习成本高，留给即梦
- 不做图层系统——这是 Photoshop 的领域

## 产品设计建议（汇总）

### 默认体验：多版本选择 > 精准编辑

```
用户选了一个 AI 特效
    │
    ├── 一次出 4 个变体（同 prompt 不同 seed）
    │
    ├── 用户选一个最喜欢的
    │
    ├── 选完后出现"微调"入口（不强推）
    │       │
    │       ├── 简单表达编辑意图（"换个颜色"/"去掉那个"）
    │       │
    │       ├── MLLM 路由 → Kontext 或 Inpainting
    │       │
    │       └── 出 2 个编辑结果 → 用户选
    │
    └── 满意 → 发布
```

### 编辑能力分档

| 层级 | 用户操作 | 底层实现 | 成本 |
|------|----------|----------|------|
| L0：重新生成 | "换一个" | 同 prompt 换 seed | 0.01-0.03 元 |
| L1：轻度编辑 | 选预设选项（"更亮"/"换背景"） | 预定义参数调整或端侧后处理 | 0-0.03 元 |
| L2：自由编辑 | 说一句话描述改什么 | Token concat (Kontext) | 0.05-0.1 元 |
| L3：精确编辑 | 点选区域 + 描述 | Auto-mask + Inpainting | 0.03-0.08 元 |

用户感知不到底层差异，只知道"我说了，它改了"。

### 和即梦/剪映的协作而非竞争

- 抖音特效做到 L2 为止（自由编辑），3 轮内出片
- 超过 3 轮仍不满意 → 引导"去即梦/剪映继续精修"
- 这不是功能阉割，是**正确的产品边界**——抖音特效的价值是"快速出片发布获得社交奖励"，不是"做出完美作品"

## 判断的不确定性

| 判断 | 确定度 | 不确定来源 |
|------|--------|------------|
| "重新生成"是抖音用户的主要迭代方式 | 高 | 符合用户画像和成本约束 |
| Token concat 是编辑的首选技术路线 | 高 | 行业共识 + 适合场景 |
| 编辑限频是必要的 | 高 | 成本数字决定 |
| 3 轮以内出片的假设 | 中 | 需要数据验证 |
| 首帧编辑 + 重生视频的策略 | 中 | 取决于视频生成的成本下降速度 |
| MLLM 自动路由的可靠性 | 待讨论 | 路由判断错误会导致体验断裂 |
| 多版本选择能覆盖 60-70% 迭代需求 | 待讨论 | 可能高估或低估 |
