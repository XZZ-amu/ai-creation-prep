# 迭代精准性：各家解法对比

> 面对"改 A 不动 B"的难题，不同技术路线和产品分别怎么解？

---

## 一句话总结

**行业从"用户画 mask 告诉模型改哪里"演进到"用户说人话、模型自己判断改哪里"，DiT 时代的 token concat 范式（Kontext、Step1X-Edit）是当前多轮迭代精准性的最优解，但没有单一方案能覆盖所有编辑场景——产品层需要组合多条路线。**

---

## 按技术路线分类

### 路线 1：Mask-based Inpainting（划区重绘）

**核心思路**：用户或模型圈出要改的区域，区域外像素物理锁定，区域内重新生成。

**代表方案**：
- SD Inpainting（Stability AI，2022）：UNet 额外通道输入 mask + 原图
- FLUX.1 Fill（Black Forest Labs，2025）：DiT 架构 inpainting，边界自然度大幅提升
- SAM-2 + Grounded-SAM（Meta，2024）：把"点击/文字描述"自动转成精确 mask

**优势**：
- 锁定精度最高——mask 外像素级不变
- 不依赖编辑指令的理解能力
- 成熟方案，工程稳定

**劣势**：
- 用户需要画 mask（手机端体验差）
- Mask 内从噪声重画，不保留原始结构（换颜色时形状也可能变）
- 边界处理是永恒难题
- 多轮迭代每次都要重画 mask，操作成本累积

**最适合的编辑类型**：移除物体、替换局部元素、修复瑕疵

**产品里的典型形态**：
- 画笔涂抹选区 → 填入新内容
- 点击物体自动选中（SAM）→ 替换/删除
- "橡皮擦"工具

---

### 路线 2：Attention 注入（Prompt-to-Prompt 系）

**核心思路**：生成原图时录制 attention map，生成编辑图时强行注入原 map，只让修改的 token 产生局部变化。

**代表方案**：
- Prompt-to-Prompt（Google Research，2022）：cross-attention map 注入
- Null-Text Inversion（Mokady et al., 2023）：改善 P2P 在真实图上的表现
- MasaCtrl（Cao et al., 2023）：mutual self-attention control

**优势**：
- 不需要 mask——只改 prompt 一个词
- 结构保留度极好（原图的布局、姿势完整保留）
- 连续控制（τ 参数）：可以精细调节"保结构"vs"保新词"的平衡

**劣势**：
- **只对模型自己生成的图有效**——真实图需要 inversion，效果打折
- 在 DiT 架构上适配不成熟（P2P 基于 UNet 的 cross-attention 设计）
- 需要两次前向推理（原图 + 新图），成本翻倍
- 超出"换词"的编辑能力有限（加新物体、改结构都做不好）

**最适合的编辑类型**：颜色替换、属性微调、风格强度调节

**产品里的典型形态**：
- "创意度"滑块（底层是 attention re-weighting）
- "保持度"滑块
- 适合做"变体生成"——同一构图不同风格

**DiT 时代的状态**：逐渐被 token concat 方案取代。DiT 的 attention 机制和 UNet 不同（self-attention 跨 image+text token），P2P 的注入方式需要重新设计。目前没有看到 DiT 上 P2P 级别的开源成熟方案——**这条路线正在退出主舞台**。

---

### 路线 3：指令式编辑（Instruction-based Editing）

**核心思路**：训练一个"看着原图 + 听指令 → 生成目标图"的端到端模型。用户只需要说人话。

**代表方案**：
- InstructPix2Pix（UC Berkeley，2023）：GPT-3 合成数据 + SD 微调
- MagicBrush（OSU，2023）：人工标注数据微调 IP2P
- Emu Edit（Meta，2024）：多任务训练 + 任务 token
- MGIE（Apple，2024）：MLLM 生成详细指令再编辑

**优势**：
- 用户体验最自然——说人话即可
- 不需要 mask，不需要懂 prompt 技巧
- 端到端推理，只要一次前向

**劣势**：
- 联动漂移严重——整张图都被重生成
- 多轮漂移严重——每轮输出当下轮输入，误差累积
- 训练数据的天花板——合成数据噪声大，人工数据量小
- 复杂指令（空间推理、计数、因果）普遍失败
- 中文指令效果差（训练数据以英文为主）

**最适合的编辑类型**：颜色/材质替换、添加/移除简单元素、风格变换

**产品里的典型形态**：
- 聊天框输入编辑指令
- "一键优化"按钮（底层是预设指令）

**进化方向**：这条路线正在被路线 4（DiT token concat）吸收——FLUX Kontext 和 Step1X-Edit 本质上就是"更强的指令式编辑"，但用了完全不同的架构来实现。

---

### 路线 4：Token Concat / In-context Editing（DiT 时代主力）

**核心思路**：把原图编码为 token 序列，和生成目标图的 token 序列拼在一起送入 transformer，让模型通过 self-attention 自动学会"参考原图"。

**代表方案**：
- FLUX.1 Kontext（Black Forest Labs，2025）：12B DiT + token concat + 3D RoPE
- Step1X-Edit（阶跃星辰，2025）：Qwen-VL 指令解析 + DiT token concat
- OmniGen（清华，2024）：统一 generation + editing 的 in-context 框架（待验证成熟度）

**优势**：
- **多轮迭代漂移最小**——Kontext 论文专门在 KontextBench 上展示
- 支持真实图（不需要 inversion）——原图直接编码为 token
- 架构天然支持多图参考（人物 ref + 风格 ref + 原图）
- 说人话即可——用户体验和指令式编辑一样
- 12B 级模型，单次推理 3-5 秒（1024x1024），成本可控

**劣势**：
- 仍然是端到端重生成——像素级锁定做不到
- 12B 参数模型的推理成本高于 inpainting（但低于视频生成）
- 编辑力度控制粒度不够——缺乏 P2P 那样精细的"强度旋钮"
- 复杂组合指令仍然会丢后半句

**最适合的编辑类型**：多轮迭代编辑、角色一致性编辑、风格迁移、属性修改

**产品里的典型形态**：
- 多轮对话式编辑（"改这里"→"再调那里"→"最后加个东西"）
- 角色换装/换场景（保持人物 identity）
- "在这张图基础上继续创作"

**这是 2025-2026 年行业方向的共识。** GPT-Image-1、Gemini Native Image Gen 本质上也在走类似路线（待验证具体架构差异）。

---

### 路线 5：Seed 控制 + 参数微调（工程兜底）

**核心思路**：不引入新模型，靠 img2img 的 strength 参数 + seed 锁定来做"轻度编辑"。

**代表方案**：
- img2img（SD/FLUX 标准 pipeline）
- ControlNet + img2img（结构锁定 + 轻度重绘）

**优势**：
- 零额外成本——用现有 t2i 模型即可
- 实现简单
- 配合 ControlNet 可以锁定构图/姿势

**劣势**：
- 全图微变——没有"不动"的区域
- strength 调参靠玄学
- 在 DiT 模型上效果不如 UNet 时代稳定
- 精度完全不够用于"精确局部编辑"

**最适合的编辑类型**：风格微调、整体氛围调整、生成变体

**产品里的典型形态**：
- "风格强度"滑块
- "创意度"滑块
- "随机变体"按钮（换 seed）

**定位**：辅助手段，不是主力编辑方案。适合做"同一个意图出多个版本让用户选"。

## 各产品的策略

### 可灵（快手）

| 维度 | 描述 |
|------|------|
| 图片编辑 | 局部重绘（mask-based inpainting）+ 文字描述替换内容 |
| 视频编辑 | 视频局部重绘（mask 跨帧传播），2025 年上线 |
| Mask 方式 | 手动画笔 + AI 智能选区（点击选中物体） |
| 多轮迭代 | 支持多轮编辑，但无公开的漂移优化技术文档 |
| 特色 | 视频编辑能力在国内领先，首帧编辑 + i2v 重生是主要流程 |
| 待验证 | 底层是否已切换到 DiT 架构的编辑方案 |

### 即梦（字节跳动）

| 维度 | 描述 |
|------|------|
| 图片编辑 | 局部重绘 + 擦除 + 扩图 + 指令编辑 |
| 视频编辑 | 首帧编辑 + 重新生成视频、局部擦除替换 |
| Mask 方式 | 画笔 + 智能选区 + 文字描述自动圈选 |
| 多轮迭代 | 支持编辑历史，可回退到任意步骤 |
| 特色 | 编辑工具链完整度高，面向专业/半专业用户 |
| 与抖音特效的差异 | 即梦提供精细控制，抖音特效应更自动化、更轻量 |

### Midjourney

| 维度 | 描述 |
|------|------|
| 图片编辑 | 2024 年推出 inpainting（"Vary Region"）：框选区域 + 描述新内容 |
| 多轮迭代 | 支持在同一张图上多次 Vary Region |
| Mask 方式 | 矩形/自由形状选区（Web 端） |
| 特色 | 美学质量极高，编辑后画面和谐度好 |
| 局限 | 无视频编辑能力；编辑控制粒度有限（无 strength 调节）；不开源 |
| 产品哲学 | 宁可生成质量高但控制粗糙，也不牺牲美学——"出图好看比精确控制更重要" |

### Runway

| 维度 | 描述 |
|------|------|
| 图片编辑 | Expand Image（扩图）、Erase & Replace |
| 视频编辑 | Gen-3 Turbo 支持视频 inpainting（mask + 重生） |
| Mask 方式 | 画笔 + 智能选区 |
| 多轮迭代 | 视频编辑支持多轮，但成本高、速度慢 |
| 特色 | 视频编辑是核心卖点，Gen-3 的时间一致性在业界领先 |
| 局限 | 价格贵（视频编辑按秒计费）；迭代成本高 |
| 产品哲学 | 面向专业创作者，接受高成本换高质量 |

### Pika

| 维度 | 描述 |
|------|------|
| 视频编辑 | "Modify Region"——在视频中选区域 + 描述变化 |
| 特色 | 视频局部编辑体验做得相对轻量 |
| 局限 | 精度和一致性不如 Runway；多轮迭代能力有限 |
| 产品哲学 | 面向消费级用户，追求"好玩"而非"精确" |

### Adobe Firefly / Photoshop

| 维度 | 描述 |
|------|------|
| 图片编辑 | Generative Fill（mask + prompt）、Generative Expand |
| Mask 方式 | Photoshop 的专业选区工具（魔棒、快速选择、钢笔路径、SAM） |
| 多轮迭代 | 完全支持——生成内容作为新图层，可无限迭代且不影响其他图层 |
| 特色 | **图层系统是杀手级兜底方案**——每次生成都是独立图层，不可能联动漂移 |
| 局限 | 非实时（3-8 秒/次）；移动端体验远不如桌面端；无视频编辑 |
| 产品哲学 | "AI 是画笔之一，不是整个画布"——嵌入专业工作流，不颠覆 |

### GPT-Image-1（OpenAI，2025）

| 维度 | 描述 |
|------|------|
| 图片编辑 | 对话式编辑——用户上传图 + 说指令，模型生成编辑结果 |
| 多轮迭代 | 天然多轮（对话上下文），但漂移报告较多 |
| 特色 | 指令理解能力强（GPT-4 级别 LLM）；用户体验最自然 |
| 局限 | 像素保留度不如 Kontext（FLUX Kontext 论文对比数据）；不开源；成本高 |
| 产品哲学 | "对话即创作"——模糊了生成和编辑的边界 |

## 组合策略：没有银弹，需要路线组合

实际产品中不会只用一条路线。以下是行业里已经验证的组合模式：

### 模式 A：MLLM 分流 + 双路线执行

```
用户输入指令
    │
    ├── MLLM 判断：这是"局部精确修改"还是"整体风格调整"？
    │
    ├── 局部精确 → 自动生成 mask（SAM-2）→ Inpainting
    │
    └── 整体/多轮 → Token concat 编辑（Kontext 类）
```

**采用者**：即梦（推测）、Adobe（Generative Fill vs Neural Filters 是两条路线）

### 模式 B：图层化兜底

```
每次编辑结果 = 独立图层
用户可以：
  - 调整图层透明度（控制编辑强度）
  - 删除某层回退
  - 合并后继续编辑
```

**采用者**：Adobe Photoshop（最成熟）、即梦（部分实现）

### 模式 C：多版本选择 + 迭代

```
每次编辑 → 生成 4 个变体（不同 seed）
用户选最好的 → 在这个基础上继续下一轮
```

**采用者**：Midjourney（V1-V4 选择）、可灵（部分实现）

### 模式 D：首帧精修 + 视频重生

```
用户在首帧上用图像编辑工具精修到满意
    │
    └── 用 i2v 模型基于精修首帧重生整段视频
```

**采用者**：可灵、Runway（视频编辑的主流流程）

## 对抖音场景的启发

### 1. 默认路线选择：Token concat（Kontext 类）+ Auto-mask inpainting

**理由**：
- 抖音用户是普通人，不会画 mask，不会调参数 → 必须"说人话"
- 多轮迭代是核心使用路径（"先出一个，再调"）→ 需要漂移最小的方案
- Token concat 范式同时满足：低门槛 + 多轮友好

**但 token concat 不够时**（移除物体、精确替换小区域）：
- 用 MLLM 自动判断 + SAM-2 自动圈选 → 走 inpainting
- 用户无感切换——不需要知道底层走了哪条路

### 2. "编辑强度"必须产品化

所有技术路线都有"力度"参数（strength / guidance / τ），但目前产品普遍没有把它暴露给用户。

**建议**：给用户一个简单的"改动幅度"滑块——
- 小幅 = 颜色/细节微调（高保留度）
- 大幅 = 结构性改变（低保留度，高创意度）

底层映射到不同参数组合甚至不同模型路线，但用户只看到一个旋钮。

### 3. 版本树 + 分支回退是必备 UI

多轮漂移不可能被模型完全消除。产品层必须提供：
- 每一轮的快照（自动保存）
- 从任意历史版本分支出去继续编辑
- 对比视图（A/B 看当前 vs 上一轮）

**不是"历史记录"那种线性的撤销——是树状的分支结构。** 用户可以从第 2 轮分出两条路试不同方向。

### 4. 视频编辑的产品策略：首帧精修 > 视频直接改

在当前算力约束下（见 product-foundation.md 的成本计算），视频逐帧编辑对抖音来说成本不可承受。

**更可行的产品路径**：
1. 用户在首帧上用图像编辑（Kontext / inpainting）精修到满意
2. 一键重新生成视频（i2v，用精修后首帧作为条件）
3. 不满意 → 回到首帧继续改

这样把视频编辑的成本降到"图像编辑成本 + 一次 i2v 生成成本"，而不是"N 帧 x 编辑成本"。

### 5. 算力成本估算

| 方案 | 单次推理成本（估算） | 抖音场景可行性 |
|------|---------------------|----------------|
| Inpainting（SD级） | ~0.01-0.02 元 | ✅ 完全可行 |
| FLUX Kontext（12B） | ~0.05-0.1 元 | ✅ 可行（字节算力优势） |
| Step1X-Edit（MLLM+DiT） | ~0.1-0.2 元 | ⚠️ 边界，需要蒸馏/量化 |
| 视频 inpainting（5秒） | ~1-3 元 | ❌ 不可承受（日活量级） |
| 首帧编辑 + i2v 重生 | ~0.1 + 0.5 = 0.6 元 | ⚠️ 勉强，需要精细化成本控制 |

> 以上成本数字基于 2025 年公开的 API 定价和硬件成本推算，**实际字节内部成本可能因自研芯片/集群利用率显著更低，标"待验证"。**

### 6. 和兄弟产品（即梦/剪映）的差异化

| | 即梦 | 抖音 AI 特效 |
|---|---|---|
| 用户画 mask | 是，精细控制 | 否，AI 自动或不需要 |
| 编辑轮次 | 多轮精修 | 1-3 轮快速定稿 |
| 编辑参数暴露 | 多（strength、guidance等） | 最多一个滑块 |
| 目标 | 作品级输出 | 社交内容级输出 |
| 容忍漂移程度 | 低（专业用户会注意到） | 中（普通用户容忍度更高） |

**核心启示**：抖音特效不需要像素级完美，但需要"3 轮内出片"——所以多轮漂移控制的优先级极高，比边界精度更重要。

### 7. 技术路线选型建议

```
优先级排序（抖音 AI 特效场景）：

P0：Token concat 编辑能力（FLUX Kontext / Step1X-Edit 级别）
    → 覆盖 80% 用户编辑需求（改颜色、加东西、换风格、角色一致性）
    → 多轮迭代漂移小
    → 用户门槛最低

P1：Auto-mask + Inpainting（FLUX.1 Fill + SAM-2）
    → 覆盖"移除 / 精确替换"需求
    → 和 P0 组合，由 MLLM 自动路由

P2：首帧精修 + i2v 的视频编辑流程
    → 视频场景的性价比最优路径

P3：版本树 UI + 分支回退
    → 产品层兜底，不依赖模型完美
```

## 行业趋势判断

1. **P2P / attention 注入类方案正在退出历史舞台**。DiT 架构的 attention 模式不同于 UNet，P2P 的适配成本高且效果不如 token concat。2025 年后的新产品不太可能基于 P2P 路线。

2. **指令式编辑和 token concat 正在融合**。Step1X-Edit 就是"MLLM 指令解析 + token concat 执行"的融合体。未来的编辑模型会越来越像"一个能看图听指令的多模态 agent"，而不是"一个带额外通道的生成器"。

3. **Auto-mask 正在消灭"用户画 mask"这个交互**。SAM-2 + Grounded-SAM 让"点击选中"和"文字描述选中"成为可能。2026 年的产品应该默认不让用户手动画 mask——除非用户主动选择"精确模式"。

4. **多轮迭代的评估标准正在建立**。KontextBench 是第一个专门测"多轮编辑一致性"的 benchmark。预计 2026 年会有更多标准化评估——这意味着产品团队可以定量追踪编辑精度的改善。

5. **视频编辑仍然是成本禁区**。即使 DiT 视频模型在质量上有突破，"每次编辑重生成整段视频"的成本结构没有根本改变。短期内（2025-2026），视频迭代编辑仍会被限制在"首帧改 + 重生"模式。

## 还没搞清楚的部分

- **字节内部是否已有 Kontext 级别的编辑模型**：如果有，抖音特效的编辑体验升级路径很清晰；如果没有，需要评估 Step1X-Edit（开源）的可用性或自研投入。待入职后确认。
- **可灵视频编辑的具体技术方案**：是 mask 跨帧传播 + 视频 inpainting，还是首帧改 + v2v？公开信息不足。
- **GPT-Image-1 的编辑架构**：OpenAI 没有发论文，只知道效果。Kontext 论文对比显示 FLUX Kontext 在保留度上优于 GPT-Image-1，但 GPT-Image-1 的指令理解力可能更强。
- **蒸馏/量化后 DiT 编辑模型的精度损失**：12B 的 Kontext 在手机端部署不现实，服务端推理成本也不低。蒸馏到 3-5B 后编辑精度能保留多少？没有公开数据。
