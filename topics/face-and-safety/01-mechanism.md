# 安全与人脸处理：用户上传一张自拍，到底要过多少道关

> 写作说明：本文是 v2 范式的延伸——沿用 [intent-understanding/01-mechanism-v2.md](../intent-understanding/01-mechanism-v2.md) 的"画室"比喻体系。如果还没读过 v2 和 [iteration-precision/01-mechanism.md](../iteration-precision/01-mechanism.md)，建议先读那两篇——本文里"翻译官 / 画家 / 草稿世界 / 骨架图 / 风格手册"五个角色不再重新解释。
>
> 本文要往画室里加两个新角色：**身份印章师**（face ID injection）和**安检员**（safety filter / watermark），他们决定"用户的脸能不能稳"和"做出来的东西能不能发出门"。

---

## 一句话总结

抖音 AI 特效的核心场景——**用户上传自拍，AI 把他变成古风武侠角色**——技术上是两件事拧在一起：
**（一）让"是他"这件事在风格剧烈变化下不丢；（二）让生成的内容不出事、能识别、能合规。**

第一件事靠**人脸识别向量 + 专门的注入通路**（不再是 CLIP 那种粗糙的图像 embedding）。第二件事靠**前置数据过滤 + 后置内容识别 + 不可见水印 + 合规元数据**——四层防线，每层都漏一点。

---

## 贯穿案例

整篇文档围绕这一个场景走完全程：

> 用户在抖音特效里**上传一张自拍**，选了一个特效叫"古风武侠"，prompt 大致是 `"a martial arts hero in ancient Chinese style, flowing robes, sword in hand, mountain backdrop"`。
> 生成结果要满足：**"还是他"**（朋友看一眼能认出来）+ **"风格够浓"**（不是只换了背景的自拍）+ **"不出事"**（不能 NSFW、不能侵犯名人肖像、不能让用户变成未成年的暧昧形象、出门要带"AI 生成"的标）。

每个机制讲完都回到这个场景——它对这张自拍做了什么、可能在哪里出错。

---

## 画室扩编：两个新角色

| 角色 | 对应概念 | 它在干什么 |
|---|---|---|
| **身份印章师** | 人脸 ID 注入（IP-Adapter FaceID / InstantID / PhotoMaker / PuLID / ConsisID） | 拿到自拍 → 提取一枚"身份印章"（face embedding）→ 在画家每一笔上盖一下，让"是他"这件事不被风格冲掉 |
| **安检员** | NSFW filter / 名人识别 / 未成年识别 / 水印 / C2PA | 在画室入口看输入、在画家身边盯生成、在出门口贴标签——分前置和后置两班 |

**这两个角色都不完美**，下面每个机制单独讲时会说"哪里会失手"。

> **和"修脸匠"的边界**（避免混淆 [quality-and-fallback](../quality-and-fallback/01-mechanism.md) 里的角色）：**身份印章师**管"是不是同一个人"——在生成前就给画家盖印章；**修脸匠（CodeFormer / GFPGAN）** 管"脸糊了重画清楚"——在生成后做修复，且**不保 identity**（修完可能更糊地不像本人）。两者经常配合使用，但解决的是两件不同的事。

## 第一部分：身份印章师——让"是他"不丢

### 为什么 v2 里的画家不会画"是他"

v2 里讲过：画家通过 cross-attention 听翻译官说的话。"一个英俊的中国男人"画家能画——但画的是"一个泛泛的英俊中国男人"，不是用户本人。

**人脸识别比类别识别精细一万倍**。`"a man"` 在翻译官的语义里只占几个 token；而"是这个人不是那个人"需要的是一个高维度的连续向量（512 或 768 维），这种信息塞不进文字 prompt。

更要命的是：v2 里的图像参考机制——**IP-Adapter（用 CLIP 编码参考图）**——也不行。CLIP 是为了"图文对齐"训出来的，它的图像 embedding 抓的是**"这是什么"**（构图、风格、颜色、大致是个亚洲男性），而**不是"这是谁"**。InstantID 论文原话："CLIP's inherent limitation lies in its training on weakly aligned data, which means its encoded features predominantly capture broad, ambiguous semantic information like composition, style, and colors."

所以人脸 ID 注入这件事，2023 年底以前是**靠 LoRA 一个人训一个模型**（DreamBooth/LoRA 路线，每个用户几十分钟训练），2024 年开始变成**靠"身份印章师"——单图 / 几张图、几秒钟、不训练**。

### 三种身份印章的"印章原料"差异

身份印章师手里的"印章"长什么样，决定了它能做什么。三种主要原料：

| 原料 | 来自哪里 | 抓的是什么 | 代价 |
|---|---|---|---|
| **CLIP 图像 embedding** | 通用图文模型（CLIP ViT-H/L） | 泛泛的"这是什么人"——年龄、性别、肤色、表情、构图 | 像素层面的相似度高，但"是他"这件事不准 |
| **人脸识别 embedding**（ArcFace 系，如 InsightFace 的 buffalo_l / antelopev2） | 在百万级人脸识别数据上训的专用模型 | "是他"——一个对姿态/光照/表情大致不变的身份向量 | 抓"身份"很准，但**不抓"长相结构"**——所以脸型、五官比例可能漂 |
| **混合**（ArcFace + CLIP） | 把两路拼起来 | 身份 + 结构 | IP-Adapter FaceID Plus、PuLID、ConsisID 都用这条路 |

**核心 tradeoff**：印章越精确（ArcFace 单路），prompt 越听话但"长相结构"越容易漂；印章越完整（混合），"长得像"但容易把参考图的光线、表情一起带过来，**风格手册（古风武侠）反而压不下去**——这就是为什么"用 IP-Adapter Plus 出来的脸总像 PS 上去的"。

### 机制 1：IP-Adapter FaceID（h94，2023 年底起）—— 身份印章师 1.0

**干什么**

IP-Adapter 原本是 v2 里的"风格手册"（递一张参考图给画家）。FaceID 变种把那本手册的封面换了：原本递的是 CLIP 算出来的图像 embedding，现在递的是 InsightFace 算出来的人脸识别 embedding，再加一个 LoRA 帮画家"记得这种人脸的画法"。

**变种家谱**

- **FaceID（基础版）**：只递人脸识别向量。脸的"身份"很稳，但**脸型/五官比例容易漂**——因为人脸识别 embedding 是为"在不同光线姿态下也能认出是同一个人"训的，本来就抹掉了一部分结构信息。
- **FaceID Plus**：人脸识别向量 + 一张对齐过的脸的 CLIP embedding。**结构跟住了**，但开始抢风格——"画成水墨"的指令变弱。
- **FaceID PlusV2**（最实用）：Plus 的基础上多了一个 `s_scale` 旋钮，可以**单独调"结构"那一路的强度**。低 `s_scale` ≈ 基础版（自由度高），高 `s_scale` ≈ Plus（长得像但风格被压）。
- **FaceID Portrait**：默认吃 5 张参考图，**取 ID embedding 的平均**——压低姿态/光照噪声，identity 更稳。代价是要用户传多张照片，门槛上去了。

**回到我们的古风武侠**

- 用户传一张自拍，FaceID 提取一枚 ArcFace 印章；
- 画家照着 prompt 画"古风武侠角色"，每一笔都被印章盖了一下；
- ✅ "是他"这件事大概率成立；
- ⚠️ 用基础版：脸型可能瘦了一圈、五官位置偏；用 Plus：结构对了但**风格手册（"水墨"、"古风"）效果变弱，看起来还是当代写实**；
- ⚠️ 这是"印章 vs 风格"的核心矛盾——**抖音特效里风格越夸张这个矛盾越明显**。

**比喻不准在哪**

把 ArcFace embedding 说成"一枚身份印章"是浪漫化了。它实际是一个 512 维向量——本身就有偏：**InsightFace 在不同人种、年龄、性别上的识别精度不一致**（这是公开学术界共识，模型卡承认 "generalization is limited due to limitations of the training data"）。这意味着同一个特效，**在不同肤色用户身上的"是他"成功率不一样**。这是一个产品体验的隐藏不公平。

**许可证警告**

InsightFace 预训练模型是 **non-commercial research only**——h94 的 IP-Adapter FaceID 模型卡明确说"inherits this restriction"。换句话说，**直接拿这个开源版本商用是不行的**。要商用要么换可商用的人脸识别 backbone 自己重训（百度、商汤、字节内部都有），要么从头训。

### 机制 2：InstantID（小红书 InstantX，2024 年初）—— 身份印章师 2.0

**干什么**

InstantID 把"人脸 ID 注入"从"风格手册"变成了"风格手册 + 骨架图"两路并发：

- **路 A：图像适配器**——和 IP-Adapter FaceID 类似，用 ArcFace embedding 做参考；
- **路 B：IdentityNet**——一个**专门训的 ControlNet**，骨架图不是姿态/边缘，而是**人脸的 5 个关键点**（两眼、鼻、两嘴角）。

为什么只用 5 个点不用 68 个？InstantID 论文原话：5 个点是"a more generalized constraint"，避免把"嘴张开还是闭合"、"脸型"这种细节绑死，**保留可编辑性**。

**核心权衡——两个独立旋钮**

- `ip_adapter_scale`：身份印章强度（默认 ~0.8）；
- `controlnet_conditioning_scale`：5 点骨架强度（默认 ~0.8）。

两个都拉高 → "是他"很稳，但生成偏写实、风格被压；都拉低 → 风格爆，但脸不像。**这两个旋钮是抖音特效产品调参的核心**——不同特效（写实变身 vs 极致风格化）需要不同的预设组合。

**回到我们的古风武侠**

- 自拍 → ArcFace 印章 + 5 点关键点；
- 5 点骨架告诉画家"眼鼻嘴该在哪"——这一步直接把"脸的位置和大致比例"按住；
- ArcFace 印章告诉画家"是他"；
- ✅ 比 FaceID Plus 更稳——结构由 5 点关键点提供，不用借 CLIP 的 face crop 那种"会污染风格"的信号；
- ⚠️ 论文里坦承：**只支持单人**（"only use the largest face as reference facial landmarks"）——多人合影直接 fallback 到只识别最大那张脸；
- ⚠️ ID embedding 自身耦合了"性别、年龄"——所以**用男孩的脸生成"古代女将军"，性别可能漂回男的**（这是 ArcFace 学到的统计耦合）。

**比 IP-Adapter FaceID 强在哪**

InstantID 论文里的对比图很清楚：IP-Adapter FaceID-Plus 在"古风/动漫/赛博朋克"这种**非写实风格**下，会出现"脸像 PS 贴上去的"——身份保住了但和背景割裂。InstantID 因为 ControlNet 那一路只递了 5 点稀疏信号，画家有空间把脸"溶进"风格里。**对抖音"特效"场景这是关键能力**——抖音特效十有八九要风格化，不是写实换装。

### 机制 3：PhotoMaker（腾讯 ARC，2024）—— 多照片求平均的另一条路

**干什么**

PhotoMaker 没用 ArcFace，反而**重新拥抱 CLIP**——但做了两件事让 CLIP 也能扛 ID：

1. 接受**多张参考照片**（同一个人不同角度），通过 CLIP image encoder 各算一组 embedding；
2. 把这些 embedding 与 prompt 里的"class token"（如 "man"、"woman"）**叠加合并**（"stacked ID embedding"），得到一个增强的语义条件，喂给 UNet。

它的卖点是**编辑性强**——身份 embedding 是叠在 class token 上的，所以 prompt 改 "old man → young man" 还能听话。代价是**单图 ID 准确度不如 InstantID**，需要 3-4 张参考。

**回到我们的古风武侠**

- 用户得传 3-4 张不同角度的自拍；
- ✅ Prompt editability 强（"古风、瘦削、长须" 这些 prompt 修饰真的会作用到生成的脸上）；
- ⚠️ 3-4 张照片对抖音"加号入口快速出特效"的场景**用户路径太长**——产品上很可能只在"专业模式"或"多图体验"里出现，不会是默认。

### 机制 4：PuLID（字节 ByteDance，2024）—— 用对比学习压"污染"

**干什么**

PuLID（Pure and Lightning ID Customization，作者来自字节内部）解决的是另一个问题：**"插入身份会污染原模型行为"**——前面三种方法都有，IP-Adapter FaceID、InstantID 也跑不掉。"污染"具体表现：

- 同一个 prompt + 同一个 seed，**没插身份**生成 A 图，**插了身份**生成 B 图，按理应该 A 和 B 除了脸都一样，但实际上：背景、光线、风格、构图全变了；
- 越是"身份保得准"的方法，污染越重——身份信号把整张图都拉向训练时的"真人写实分布"。

PuLID 的招数：

- 训练时多开一条"Lightning T2I 分支"（4 步快速采样的子模型）；
- 在这条分支上同时跑两路——**有身份 / 无身份**——其他都一样；
- 强行让两路在 cross-attention 的注意力分布上对齐（contrastive alignment loss）；
- 这等于在训练阶段就告诉模型："插身份只能改身份，**不许改其他**"。

外加一个"准确 ID 损失"——在快速采样后的清晰图像上算 ArcFace 距离（不像 InstantID 在去噪中间步算，那时图还很糊，loss 不准）。

**回到我们的古风武侠**

- ✅ "古风武侠"的风格手册不被身份信号压制——按论文图，PuLID 的"插身份后构图/风格/光线一致性"显著高于 InstantID 和 IP-Adapter FaceID；
- ✅ 字节自家方案，**没有 InsightFace 那种 non-commercial 许可证 issue**（PuLID 完全在字节训练数据上训）；
- ⚠️ 风格化能力上仍然不是无敌——极端 stylization（动漫、3D 卡通）下身份还是会有损耗；
- ⚠️ 推理时多一条 Lightning 分支用，**速度比 InstantID 慢一点**——抖音侧需要权衡延时。

### 机制 5：ConsisID（北大袁源组 + ByteDance，2024）—— 视频里的身份印章师

**干什么**

前面四种都是**图像**任务。视频更难——一段 5 秒视频里，**每一帧的脸都要是同一个人**。如果按图像方法逐帧来，**帧间会闪烁**（每帧的 ArcFace 信号干扰 cross-attention 不稳定，导致脸在帧间微抖）。

ConsisID 把人脸 embedding 拆成两段频率：

- **低频**（脸的整体形状、轮廓、比例）→ 与噪声 latent **直接 concat**，喂到 transformer 浅层（DiT 没有 UNet 的 skip connection，浅层需要硬塞）；
- **高频**（眼睛纹理、嘴唇细节这些"是他"的强标记）→ 通过 **cross-attention 在每一层注入**，因为 transformer 对高频不敏感，要专门照顾。

外加一个"动态跨帧损失"：训练时**用片段外的另一帧**作为参考图（带轻微高斯噪声），强迫模型不要走"复制粘贴"的捷径——必须真的学到"这个人在不同帧的不同表情"。

**回到我们的古风武侠**（假设 5 秒视频版）

- ✅ 49 帧（约 6 秒 @ 8fps）跨帧身份一致；
- ⚠️ 论文给的训练分辨率上限是 480×720——抖音手机端虽然能接受，但**对竖屏长视频不够**；
- ⚠️ 推理时 ~44 GB 显存——单卡 H100 / A100 跑得动，**手机端不可能本地跑**，必须走云端；
- ⚠️ 身份保住了但**面部表情多样性下降**——cross-face loss 在压"复制粘贴"的同时也压了一部分自然表情漂移。

### 三件事的取舍——产品视角的核心关卡

到这里把人脸注入的本质讲完了。这件事永远是**三件事的零和博弈**：

```
       身份保留度 ──┐
                   │
                   │  抖音特效要的是这三角形里的某个点
                   ↓
  风格转换强度 ←──── ── ── ── ── ──→ 表情/动作自然度
```

**三角形上的四个产品决策点**：

1. **写实换装类特效**（"穿越成 1950 年代上海"）：身份 ★★★★★、风格 ★★★、自然度 ★★★★ → InstantID 默认参数
2. **极致风格化特效**（"变成皮克斯卡通角色"）：身份 ★★★、风格 ★★★★★、自然度 ★★★★ → PuLID 或 InstantID 低 scale
3. **专业化高保真**（"专业摄影写真"）：身份 ★★★★★、风格 ★★★、自然度 ★★★★★ → PhotoMaker 多图模式
4. **视频特效**：以上全部 + 帧间一致 → ConsisID 类方案，**算力贵 5-10x**

**多人脸场景**：InstantID 论文坦承"only support the largest face"。要支持多人合影变身，业界做法是**先 SAM2 / Grounded-SAM 把每个人分割出来 → 单独跑身份注入 → 拼回去**。这条路工程量大、闪烁多、容易"互相污染"（用户 A 的脸跑到用户 B 身上）——这是抖音特效里**还没解决好的硬骨头**。

### 不同 backbone 的人脸稳定性差距（直观感受）

| Backbone | 人脸稳定性（基于公开 benchmark + 社区共识） | 备注 |
|---|---|---|
| **SD 1.5** | 中——人脸 ID 工具最早就是为它做的，生态最成熟 | IP-Adapter FaceID、PhotoMaker SD1.5 版都在 |
| **SDXL** | 较高——分辨率上去之后细节丰富，但 **CFG 调高容易脸变形** | 是 InstantID、PuLID、PhotoMaker 主战场 |
| **FLUX (DiT)** | 最高质量，但**人脸生态滞后**——直到 2024 下半年 PuLID-FLUX、FLUX.1 Kontext 才有像样的人脸方案 | DiT 架构和 UNet 的 attention 模式不同，UNet 时代的方案不能直接搬 |
| **HunyuanVideo / CogVideoX**（视频） | ConsisID 类方案在 5-6 秒短视频上稳定；更长片段未见公开结论 | 抖音特效短视频时长（< 15 秒）刚好踩在 sweet spot |

## 第二部分：变形容忍度——这是产品判断不是技术判断

技术给的是**"能压多准"**，但**"该压多准"是产品判断**。

抖音特效用户上传自拍的心理预期不是"100% 像我"——是**"一眼能看出是我但又不完全是我"**：

- **0% 像**：朋友看不出是谁，用户不会想分享——无法构成"是我做的特效"的社交资本；
- **50% 像**：朋友看一眼觉得"哎这是 X X X 吧？"——这是甜点，**用户最愿意发**；
- **100% 像**：等于把自己的脸贴到一个新背景上——失去"特效"的趣味性，同时**法律风险陡增**（详见第三部分）。

所以 InstantID `ip_adapter_scale = 0.8` 这种默认参数，**产品上其实应该再压低一点**——大约 0.5-0.7，让"风格"赢一点点。即梦/可灵给用户的"创意度滑块"本质上就是这个 scale 旋钮的封装。

**几个产品上必须列出的失败模式**：

| 失败模式 | 用户视角 | 根因 |
|---|---|---|
| 完全不像我 | "这特效太假了" | 印章弱（scale 低）或 backbone 在该肤色/性别上识别精度差 |
| 像得不舒服 | "怎么把我贴到画里去了，没融进去" | 印章太强（scale 高），脸不能融入风格 |
| 像但变性别 / 变年龄 | "怎么把我变成女的（我是男的）" | ID embedding 性别耦合，prompt 里"少女"等词压过了 ID 信号 |
| 不同肤色用户成功率不一 | （隐藏）"我朋友用一样的特效更像他" | 人脸识别 backbone 在不同人种上精度不平等 |
| 多人合影乱串 | "怎么我女儿的脸长在我老公身上了" | 多脸场景没分割好，identity 互相污染 |

## 第三部分：安检员——前置 / 后置 / 水印 / 元数据

四道关。每一关都漏一点。

### 关 1：训练数据过滤（前置阻断）

**干什么**

模型训练时就把"危险数据"摘掉——色情、暴力、未成年人有性意味的图、明确的明星个人数据集。Stable Diffusion 2.x 之后官方训练就用 LAION 的 NSFW 标签过滤了一波。开源社区里很多"不安全"的能力实际是**用未过滤数据微调出来的**（civitai 上的某些 NSFW LoRA 就是这种来源）。

**比喻不准在哪**

"前置阻断"听起来像把脏数据彻底拒之门外。**实际是分类器有假阴性**——LAION 的 NSFW 分类是 CLIP 系，会漏一部分隐晦的、艺术化的、跨文化语境的有问题内容。所以即使是"过滤后"训出来的模型，在某些 prompt 下还是能生成 NSFW——只是概率低了。

**对抖音的意义**

抖音特效模型如果是基于开源 backbone（如 FLUX、SDXL）+ 自己微调，**底层的 NSFW 倾向会从开源模型继承下来**——换言之，**单靠"我没在 NSFW 数据上微调"是不够的，必须在前面再加一层后置过滤**。

### 关 2：NSFW Safety Checker（后置过滤）—— 现成工具的局限

**机制（CompVis Safety Checker，最普及的版本）**

- 用 CLIP ViT-L/14 给生成图算一个 image embedding；
- 和**预先训好的 17 个"坏概念" embedding**（具体内容是 obfuscated 的，但社区已知大致是 NSFW 概念）算余弦相似度；
- 超过阈值 → **替换成纯黑图**返回给用户。

**比喻不准在哪**

"安检员"听起来很专业。实际上**这个安检员的眼睛是 CLIP 的眼睛，CLIP 学到的偏见它全继承了**：

- **对深肤色用户的误报率显著高于浅肤色**——CLIP 模型卡自己承认"significant disparities with respect to race and gender"；
- 把**艺术裸体、医学图、运动健身、游泳照**经常误判为 NSFW；
- 同时，**轻微对抗扰动就能绕过**——生成图加一点噪声、改色调，embedding 就跨过阈值。

**绕过它的代价是 0**——`safety_checker=None` 一行参数关掉。这意味着开源生态里**实际上没有强制 NSFW 防线**——任何接入开源模型的应用都得自己再做一遍。

**对抖音的意义**

抖音侧不可能用 CompVis 的开源 safety checker——
1. 误报率太高（深肤色误判会变成产品事故）；
2. 概念太英文中心——"NSFW"在中国大陆合规框架里只是其中一个维度，还有政治敏感、暴恐、宗教敏感、未成年人保护等多个独立维度；
3. **对抗鲁棒性差**。

字节内部肯定有自己的多模态内容审核栈（图像识别 + 文本识别 + 多模态判别），抖音特效的生成结果**必须经过这一层才能让用户保存/发布**。这是入职后才能看到具体细节的环节，**这里只标"待查"**。

### 关 3：明星 / 未成年人识别（专项防线）

**这是抖音/字节这种平台必须重点投入的方向，原因很简单：**
- 用户上传的照片**不一定是自己的**——有可能是明星、朋友、暧昧对象、随便从网上扒的；
- 输出的脸**不能撞知名公众人物**——否则就是 deepfake；
- **未成年人的脸不能进入任何带有 sexualization 倾向的特效**——这是大陆和全球都不可触碰的红线。

**业界做法（公开信息可见的）**：

- **明星识别**：在用户上传图片时跑一遍**内置的明星人脸库**对比；命中则拒绝特效或要求用户证明本人。Lensa、即梦、Midjourney 都做了这个（当年 Lensa 翻车就是没做）。字节肯定有最大的中文明星库——**这是壁垒之一**；
- **未成年识别**：用年龄估计模型（公开如 InsightFace 自带）+ **保守阈值**（任何疑似未成年都拒绝部分特效）；过滤要主动而非反应式；
- **生成结果的二次识别**：生成完成后再跑一遍明星识别，万一模型自己"脑补"出了像某明星的脸，要拦截。

**比喻不准在哪**

"专项防线"听起来很专业。实际上：
- 中文明星库的覆盖永远滞后于网红更新速度——昨天爆红的网红今天还没进库；
- 年龄估计在不同人种、化妆/未化妆、戴眼镜/不戴上**误差 ±5 岁**——保守阈值定多严决定了"误伤多少正常成年用户"；
- "生成结果像某明星"是模型本身的训练数据问题，下游识别只能补救，**根本上需要训练阶段做明星照片去重**。

### 关 4：水印 / C2PA（出门贴标）

**两条独立的技术路径**：

#### a. 不可见水印（SynthID 系）

- Google DeepMind 的 SynthID：在生成的像素 / token 概率分布里嵌入一个**人眼不可见但检测器能识别**的信号；
- 关键属性：对**轻度修改鲁棒**——小幅裁剪、压缩、滤镜、加噪声后水印仍能检出；
- 局限：**重度修改可破坏**（大幅裁剪、二次生成、极强滤镜）；
- 公开信息里 SynthID 已应用于 Imagen、Lyria、Veo、部分 LLM 输出。SynthID-Text 开源了参考实现，图像/视频版的具体算法**未完整公开**。

#### b. 显式签名元数据（C2PA）

- C2PA = "Coalition for Content Provenance and Authenticity"——一个开放标准；
- 在媒体文件里附加一个**密码学签名的清单（manifest）**，记录"谁创建的、用什么工具、做了哪些编辑、签名者身份"；
- 任何能读 manifest 的检测器都能验真伪；
- 局限：**manifest 可以被剥离**（保存时去掉元数据），剥离后无法事后恢复；
- 主要采纳者：Adobe、Microsoft、OpenAI（DALL·E 3、Sora 输出附带）、相机厂商（Sony / Leica）。

**两者通常配合用**——C2PA 在文件里贴显式标签，SynthID 在像素里藏冗余信号；前者被剥后者还在，后者被破坏前者还在。

**对抖音的意义**

抖音是中国大陆的产品，必须遵守**《互联网信息服务深度合成管理规定》（2023 年起施行，国家网信办）和《生成式人工智能服务管理暂行办法》（2023 年 8 月起施行）**——其中明确要求"对生成的图片、视频等内容进行标识"。

我**不能从公开 fetch 拿到完整法条原文**（域名限制），但根据公开报道的合规要点：
- **必须显式标识**——"显著位置"显示生成内容是 AI 生成的（这条是显式可见水印，不只是不可见水印）；
- **元数据中也要带标识**（C2PA 类）；
- **不得用于深度伪造他人**——尤其是政治人物、公众人物。

抖音的现实做法是：**生成视频右下角小字"AI 生成"标识**——这是合规最低线。是否同时叠加不可见水印（字节内部的 BeyondAI 类技术）属于内部细节，**待查**。

## 第四部分：抖音特效场景的特殊性

把前三部分的所有约束放到"抖音特效"的具体语境里。

### 1. 用户上传自拍是主流入口 → 隐私风险

特效产品的入口是**让用户授权打开摄像头/相册**——这意味着平台拿到了用户面部数据。

**关键风险点**：
- 用户自拍**会不会被用于训练后续模型**？这是大陆《个人信息保护法》和欧盟 GDPR 都管的事；合规做法是**用完即删**或明确告知并征得同意；
- **自拍上传后是否在云端持久化**？合规做法是用临时签名 URL，处理完销毁；
- **多个特效之间能不能复用同一张自拍 / 同一份 face embedding**？这是体验和隐私的平衡——复用提速，但意味着 embedding 在云端存了。

### 2. 平台风险：用户用 AI 生成他人面孔

抖音特效的根本风险点 ≠ 只是 NSFW，而是**有人拿别人的脸做特效后发出去当真**。

**业界标准做法**（综合 Lensa、Midjourney v6、字节自己已上线的特效经验）：
- 上传**自拍**时强制做**活体检测**（不是任何静态照片都行）——用前置摄像头实时拍；
- 强制识别明星 → 拒绝；
- 强制识别未成年 → 拒绝部分特效；
- 输出的视频**强制带"AI 生成"水印**——这条是合规要求也是产品自保。

### 3. 大陆合规

国家网信办出台的两份核心文件（**网信办深度合成规定 + 生成式 AI 暂行办法**）合计要求：

- **算法备案**：模型上线前要在网信办系统做算法备案——**字节肯定已经备过了**，这是上线前提；
- **训练数据合法**：训练数据要合法获得（这条对开源 backbone 是个有趣问题——LAION 的合规性本身有争议，**字节侧的实际做法是用自有数据 recaption / 微调**绕开这个问题，但 backbone 的"原罪"无法消除）；
- **生成内容标识**：包括显式标识 + 元数据标识；
- **不得违反公序良俗、违法乱纪、伤害他人合法权益等**——这部分映射到产品就是 NSFW filter + 政治敏感 + 暴恐识别，比海外严苛一档；
- **用户实名 / 防青少年模式**——防止未成年人滥用。

我没有办法直接 fetch 网信办原文（域名限制 + 中文政府网站不在白名单），所以**法条具体条文标"待查"**——产品决策时必须以法务确认的最新版本为准。

## 失败模式清单（产品上一定会碰到的）

| 失败模式 | 表现 | 根源 | 兜底方案 |
|---|---|---|---|
| **身份漂移**（不像我） | 朋友看不出是谁 | scale 太低 / backbone 对该用户人种识别精度差 | 让用户传多张自拍走 Portrait 模式 |
| **风格被压**（像但僵硬） | 看起来是把脸 P 在风格图上 | scale 太高 / IP-Adapter 信号过强 | 默认 scale 调低，给用户"创意度滑块" |
| **性别 / 年龄漂移** | 男变女 / 大人变小孩 | ID embedding 耦合人口属性 + prompt 里有冲突词 | 检测 prompt 与上传图的属性冲突，提前提醒 |
| **多人合影互串** | 一家人合影变身后五官混乱 | 多人场景没分割 | SAM2 分割 + 单独注入 + 拼接（贵） |
| **NSFW 漏网** | 用户用 AI 生成擦边内容 | 后置 filter 假阴性 | 多模态多模型审核栈，前置 prompt 检查 + 后置图像/视频检查 |
| **NSFW 误报** | 正常运动 / 泳装照被拦 | CLIP 安检员对深肤色 / 暴露皮肤敏感 | 自训分类器替代 CLIP 系，分多档（成人 vs 未成年）阈值 |
| **生成像某明星** | 模型自己脑补出明星脸 | 训练数据没去重明星 | 生成后跑明星识别二次检测 |
| **未成年用户敏感特效** | 未成年用户跑成人化特效 | 上传图年龄估计有误差 | 保守阈值 + 强制实名 / 青少年模式 |
| **水印被剥离** | 用户截图发出去后看不到 AI 生成标 | 视频元数据不跨平台、显式水印可裁掉 | 多层水印（显式 + 不可见）+ 平台间互信协议 |
| **视频帧间脸抖** | 6 秒视频里脸闪烁 | 视频 ID 注入算力不够，每帧独立采样 | ConsisID 类方案，算力 5-10x |

## 产品设计师的落地洞察

1. **"是他"的精度是用户体验的核心，但不是越高越好**。技术上可以做到 95% 像，但产品上 60-70% 是甜点——用户要的是"能识别出是我"+"风格够强让朋友惊讶"，不是"高保真换脸"。**默认 scale 应该调到偏向风格那一侧，再给一个滑块让用户拉**。这件事的判断是**用户心理 + 法律风险 + 技术能力**三方约束的综合。

2. **明星识别 / 未成年识别不是"加分项"，是上线门槛**。Lensa 当年的崩盘案例（2022 年底它没做明星识别，被发现可以生成明星 NSFW）是教科书级警告。抖音侧投入的不是"要不要做"——是"覆盖率有多高、误报率有多低、对抗鲁棒性多强"。**这是产品上线前法务和审核团队联合定线的事，产品设计要从一开始就把这些拒绝路径设计进流程**。

3. **隐私上要把"用完即删"做成产品承诺**。自拍是用户最敏感的数据之一。每一次特效生成，**清晰告诉用户"这张照片只在生成时使用，不会被用于训练"**——这是合规要求也是用户信任基石。如果业务上必须存（比如让用户复用同一张自拍跨多个特效），要清晰告知保存时长 + 提供"立即删除"按钮。

4. **合规标识不要做得遮遮掩掩**。"AI 生成"标识在视频右下角的字号、对比度、位置，是法务定的最低线。**作为产品设计师，争取的不是"能不能小一点"——是"能不能更清晰地告知"**——研究表明清晰的 AI 标识反而提升用户对生成内容的接受度（"这是有趣的特效"vs"这是冒充真实"）。监管会越来越严，提前往严的方向走总比被动改要好。

5. **"变形容忍度"是抖音特效区别于即梦/剪映的核心选择**。即梦/剪映是专业用户做精细化产品——他们需要 InstantID 那种 0.8 默认 scale 的高保真；抖音特效是普通用户社交内容——需要 0.5-0.6 的"是我又不完全是我"。这不是技术 bug，是**产品定位差异决定了参数预设差异**。从这件事可以延伸出一整套抖音特效的"参数哲学"——**永远偏向"惊喜感"而不是"技术正确"**。

6. **多人脸场景是抖音特效尚未真正解决的硬骨头，要么投入要么明确不做**。情侣合影、一家人合影、班级合照——这些是抖音用户最爱发的场景，也是 InstantID/PuLID 这种单脸方案做不好的。要么投入做"分割 + 单独注入 + 拼接"的工程化方案（贵、慢），要么在产品上**明确只支持单人特效**（用 SAM 自动检测多人时给提示"该特效仅支持单人")——把模糊地带变清晰。

## 还没搞清楚的部分（诚实标记）

- **网信办法规原文**：《生成式人工智能服务管理暂行办法》《互联网信息服务深度合成管理规定》原文条款无法在受限的 fetch 域名里直接拿。本文里的"必须显式标识"、"算法备案"、"训练数据合法"是基于公开报道的概括——**入职后第一时间向法务团队拿现行版本**。
- **字节内部的人脸 backbone 是哪个**：公开 PuLID 论文用的是 antelopev2（InsightFace 系），但字节肯定有自研的。商汤、百度、字节都有自研人脸识别——**抖音特效产品用哪个、识别精度多少、对深肤色/亚洲不同人种的精度差距是多少，待查**。
- **抖音特效具体的算力预算**：单次生成允许多少秒、多少 GPU/秒——这个预算决定了能用 ConsisID 还是只能用单帧拼接。**待入职后看团队 OKR**。
- **明星库的覆盖范围和更新频率**：中文/英文明星、网红、政治人物的库怎么维护——是否有跨业务复用（抖音、TikTok、剪映、即梦共享一套）。**待查**。
- **多脸场景的现行方案**：抖音侧目前的多人特效是怎么做的（拒绝？降级？分割？）。**待入职**。
- **未成年识别的具体阈值**：业界普遍用 18 岁作为硬阈值，但年龄估计误差 ±5 岁意味着 13-23 岁这一档都会被拦/不拦影响巨大——抖音侧的具体策略**待查**。
- **DeepFake 法律边界**：什么情况下属于"恶意 deepfake"是司法实践决定的，不是产品文档能定的。**结合具体案例咨询法务**。
