# 人脸与安全：抖音场景判断

---

## 1. 判断锚点

人脸是抖音特效的第一素材——超过 80% 的特效使用场景涉及用户的脸。**"60-70% 像 + 风格惊喜"是抖音的正确产品答案，不是技术能力不足，是产品选择。** 安全不是"加一层就行"的事，它直接影响生成延迟和用户体验，必须和产品体验一起设计。

---

## 2. 核心判断

### 判断 1：保真度应该偏向风格而非写实——"能认出是我 + 好看的惊喜"是正确答案

**理由**：
- 抖音特效用户的需求是"变身"——如果变完还长得和自拍一模一样，那不叫特效，叫美颜
- "60-70% 像"恰好在"朋友能认出是你"的阈值之上，同时留出足够的风格化空间
- 对比美图秀秀（90%+ 保真）：美图用户要"精修的我"，抖音用户要"有趣的我"
- 过度保真（>85%）反而会让风格手册失效——看起来像是真人照片做了个轻度滤镜，缺乏惊喜感

**对设计的影响**：
- ID injection scale 预设在 0.5-0.7 区间（偏风格），不做用户可调
- 质量评估时"风格表达强度"和"身份保真度"是两个独立维度——不能为了保住身份牺牲风格
- 用户如果反馈"不像"——先排查是不是模型问题；如果不是，就接受"这是产品定位决定的"

### 判断 2：PuLID 是抖音人脸注入的首选方案——字节自研 + 不压风格

**理由**：
- PuLID 的核心创新就是"注入身份但不干扰画风"——用对比对齐损失把 ID 信号限制在人脸区域
- 这恰好解决了抖音特效的核心矛盾：既要认出是本人，又要风格化效果足够强
- PuLID 是字节自研——内部可以拿到最好的版本、最快的迭代
- 对比 InstantID：保真度略高但会压风格，不适合"风格惊喜"定位

**对设计的影响**：
- 技术路线明确：人脸注入 = PuLID（或其内部演进版本）
- 关注 PuLID 的已知短板：大角度侧脸精度下降、不同人种表现差异——产品侧需要对应的兜底
- 如果 FLUX Kontext 的人脸保持效果接近 PuLID 且不需要额外模块，评估是否简化技术栈

### 判断 3：安全审核延迟必须控制在用户无感的范围内——<500ms

**理由**：
- 抖音特效的体验是"点了就出"——如果安全审核加 1-2 秒延迟，用户会感知到"卡了"
- 但安全审核不能省——字节作为平台对内容安全的要求是刚性的
- 解法是"审核和生成并行"或"审核极速化"，而不是"审核串行在生成后面"

**对设计的影响**：
- 输入层审核（人脸活体检测、明星识别、年龄估计）在生成**之前**完成——阻止问题输入进入生成流程
- 输出层审核和后处理并行跑——不是"审核完了再修脸"，而是同时进行
- 合规被拒后的重抽必须对用户透明——用户只看到"等待中"，不看到"审核失败"
- 总延迟预算：输入审核 <200ms + 生成 3-5s + 输出审核并行 <300ms = 用户感知 3-5 秒（和纯生成一样）

### 判断 4：活体检测是抖音独有的前置安全能力——应该充分利用

**理由**：
- 抖音可以要求用户实时拍摄（而不只是上传照片）——活体检测天然可做
- 活体检测能解决"拿别人照片做换脸"的核心风险——从源头杜绝 deepfake
- 这比其他平台的"输出层检测"更靠前、更安全、且对用户体验的影响更小
- 但限制：如果特效只支持"实时拍摄"就丢掉了"上传照片"的场景（相册里的照片、老照片变身）

**对设计的影响**：
- "实时拍摄"场景：默认开启活体检测，安全策略可以更宽松（允许更大的 ID scale）
- "上传照片"场景：不能做活体检测，需要更严格的输出层审核 + 明星识别
- 产品上可以做差异化策略：实时拍摄的特效开放更多玩法（因为安全有保障），上传照片的特效限制风格范围（避免生成真实感太强的内容）

### 判断 5：多人场景应该拒绝或降级，不应该硬做

**理由**：
- 多人脸同框的 ID 注入技术不成熟——两张脸同时注入容易"串味"
- 多人场景的安全风险翻倍——需要验证每个人的身份/合规性
- 多人场景的计算成本翻倍——两个 ID 注入 + 两次脸部质量评估
- 产品上也没有强需求——"把我和朋友一起变成古风角色"的需求频率远低于单人变身

**对设计的影响**：
- 短期策略：多人场景自动降级为"只处理主脸"（画面中最大的那张脸）
- 或者：多人场景只做风格化（不做 ID 注入），接受"不像某个人"但"整体好看"
- 长期：等多 ID 注入技术成熟（Kontext 多图参考 + 区域分离）后再开放

### 判断 6："AI 生成"标识应该做成正面的社交标签，而不是负面的合规义务

**理由**：
- 法规要求 AI 生成内容必须标识——这是刚性约束
- 但标识的设计可以是"酷的"而不是"警告的"
- 如果把"AI 生成"标识做成像 Instagram 滤镜标签那样的社交标签，它就变成了社交货币
- "用 AI 创作"不应该是丢脸的事——应该是"我会用新工具"的能力展示

**对设计的影响**：
- "AI 生成"标识的视觉设计应该是好看的、酷的——不是灰色小字，而是带设计感的标签
- 可以做成"AI × [特效名]"的格式——同时标识 AI 生成 + 推广特效本身
- 标识位置：视频左下角或文案区，不遮挡主画面
- 待讨论：是否允许用户主动展示"我用了什么 AI 特效"作为互动元素

### 判断 7：人种/年龄/性别的保真度不均匀是隐性公平问题——需要专项测试

**理由**：
- 底层人脸识别模型（ArcFace/InsightFace）在不同人种上精度不一致——这是学术界已知问题
- 表现在产品上：某些用户用特效"很像本人"，另一些用户"完全不像"——但用户不知道原因
- 字节内部应该有自己的人脸模型，但也需要验证在中国用户（多民族、各年龄段）上的覆盖度
- 老年用户、儿童用户、化妆浓的用户——这些边缘群体的保真效果可能显著差于"标准自拍"

**对设计的影响**：
- 建立人脸保真度的"公平性测试集"——覆盖不同年龄/性别/肤色/光照条件
- 对保真度低于阈值的群体，考虑降级策略（降低 ID scale + 提高风格化程度，让"不像"不那么明显）
- 在特效设计阶段就用多样化人脸做测试，不只用"标准年轻女性正脸"

## 3. 产品设计建议

### 建议 1：人脸特效的两种模式 + 对应安全策略

| 模式 | 输入 | ID 注入强度 | 安全策略 | 风格空间 |
|------|------|------------|---------|---------|
| 实时拍摄 | 摄像头 | 0.6-0.7（可以更像） | 活体检测前置，安全宽松 | 广 |
| 上传照片 | 相册 | 0.5-0.6（偏风格） | 输出审核加严，明星识别 | 窄（避免太真实） |

### 建议 2：安全审核和生成并行的时序设计

```
时间轴：
0ms     ——→ 输入审核启动（活体/明星/年龄）
0-200ms ——→ 输入审核完成 → 进入生成
200ms   ——→ 生成启动（PuLID ID 注入 + 模板生成）
3-5s    ——→ 生成完成
3-5s    ——→ 输出审核 + 人脸修复 并行启动
3.3-5.3s——→ 全部完成 → 展示给用户

用户感知延迟 = 纯生成时间（3-5s），审核"隐形"。
```

### 建议 3：建立特效级别的"安全预审"机制

不是每张图都做全量审核——而是在特效模板设计阶段就做"安全预审"：
- 确认这个模板不可能产出 NSFW 内容（通过 prompt 设计 + negative prompt 保证）
- 通过预审的模板可以简化输出审核（只做轻量合规检测），降低延迟
- 未通过预审的模板走全量审核流程

### 建议 4：明星保护白名单——输入层就拦住

- 维护一个"明星人脸特征库"
- 用户上传照片时先过明星识别——如果匹配到明星，直接拒绝并友好提示
- 这比在输出层检测更安全（输出层可能漏检）也更省成本（不需要白跑一次生成）

## 4. 待入职验证

1. 当前特效使用的人脸 ID 注入方案是 PuLID 还是其他？scale 值预设在什么范围？
2. 安全审核的实际延迟数据——输入审核 + 输出审核分别多长时间？
3. 多人场景当前的处理策略是什么——拒绝/降级/还是尝试做？
4. 人脸保真度在不同人群（年龄/性别/肤色）上的表现数据——有没有公平性测试？
5. "AI 生成"标识当前的设计方案——是合规义务还是做了社交化设计？
