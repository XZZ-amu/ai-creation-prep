# 人脸与安全：各家解法对比

> 身份保真、安全审核、合规策略、产品约束——各家平台怎么选、怎么做、为什么这么做。

---

## 一句话总结

**没有一家平台同时解决了"人脸像 + 风格强 + 绝对安全 + 合规无死角"——所有人都在做取舍，差异在于取舍的方向由产品定位决定。** 中国平台的独特之处是合规约束更强、安全投入更前置，且"算力成本"作为贯穿性约束决定了方案上限。

## 维度一：身份保真技术——各家怎么让"是他"

### 技术方案谱系

| 方案 | 核心思路 | 优势 | 劣势 | 典型采用者 |
|---|---|---|---|---|
| **DreamBooth / LoRA 微调** | 为每个用户训一个小模型 | 保真度最高（★★★★★） | 每人训练 5-30 分钟、成本高、不可规模化 | 早期 Lensa、部分高端写真 App |
| **IP-Adapter FaceID 系列** | ArcFace embedding → cross-attention 注入 | 单图即用、推理快 | 脸型可能漂、风格被压 | 开源社区广泛使用、部分中小 App |
| **InstantID** | ArcFace + 5 点 ControlNet 双路 | 单图、结构稳、风格融合好 | 只支持单人、性别耦合 | 小红书（InstantX）、即梦（待验证） |
| **PhotoMaker** | CLIP 多图合并到 class token | 编辑性强、prompt 可控 | 需要 3-4 张图、单图不够像 | 腾讯系产品（待验证） |
| **PuLID** | 对比学习压"ID 污染" | 风格不被压、字节自研 | 推理稍慢 | 抖音/即梦（字节自家方案） |
| **ConsisID** | 频率分解注入 DiT | 视频帧间一致 | 算力高（44GB VRAM）、分辨率受限 | 视频特效场景 |
| **FLUX.1 Kontext** | 原生 image-to-image 编辑 | 无需额外 ID 模块、质量高 | 人脸 ID 保持非专门优化 | Black Forest Labs 生态 |

### 各平台的实际选择

**可灵（快手）**
- 公开信息显示使用自研人脸保持方案
- 产品上提供"相似度滑块"——本质是 ID 注入 scale 的封装
- 支持单人写真、不支持多人合影变身（截至公开信息）
- 推测底层：InstantID 类思路 + 自研人脸 backbone（快手有人脸识别技术积累）

**即梦（字节跳动）**
- 作为字节的专业创作工具，保真度要求比抖音特效高
- 提供"创意度"滑块（本质同上）
- 支持"AI 写真"功能：用户上传多张照片 → 训练个人模型 → 多风格生成
- 推测底层：PuLID（字节自研）+ 可能结合 DreamBooth 快训路线用于高端模式

**抖音特效（本体）**
- 定位"普通用户快速出特效" → 不可能让用户等训练 → 必须是单图即用方案
- 推测底层：PuLID 或 InstantID 类方案，scale 预设偏向风格（0.5-0.7）
- 不支持"相似度滑块"（产品形态是"一键出效果"，不暴露参数）
- 多人场景推测是拒绝/降级处理（待入职验证）

**Midjourney**
- v6 后人脸一致性显著提升（DiT 架构升级带来的底层提升）
- 不提供"人脸注入"功能——不接受用户上传自拍做身份迁移
- 策略是**完全规避人脸安全风险**：不做换脸/写真 → 不存在 deepfake 问题
- 如果用户想要"像某人"：只能通过 prompt 描述，精度有限

**DALL-E 3（OpenAI）**
- 同样不支持基于上传照片的人脸生成
- 通过 system prompt 拒绝生成"真实人物肖像"——但用户绕过方法多
- GPT-4o 图像生成（2025）开始支持图片编辑，人脸策略待观察
- ChatGPT 的图像生成近期火爆的"吉卜力风"等风格化中包含人脸，但不支持精确 ID 迁移

**美图秀秀**
- 核心场景就是人脸美化 → 积累最深的人脸处理技术栈
- AI 写真功能：上传照片 → 多风格证件照/艺术照
- 底层推测：自研人脸 backbone + DreamBooth 快训（美图有自己的人脸数据集和模型）
- 特点：**保真度调得很高**（美图用户要的是"精修过的我"不是"风格化的我"）——和抖音特效的产品定位相反

### 身份保真的产品哲学差异

| 平台 | 产品定位 | "像"的目标 | scale 偏向 |
|---|---|---|---|
| 即梦/美图 | 专业/精修 | "精致版的我" | 高保真（0.8+） |
| 抖音特效 | 社交娱乐 | "能认出是我的有趣效果" | 偏风格（0.5-0.7） |
| 可灵 | 创意工具 | 中间态，用户自己调 | 提供滑块 |
| Midjourney/DALL-E | 通用生成 | 不做人脸迁移 | N/A |

**对抖音的启发**：抖音特效不需要和即梦/美图比"像不像"——产品定位决定了"60-70% 像 + 风格惊喜感"是正确答案。这不是技术能力不足，是产品选择。

## 维度二：安全审核体系——各家怎么防"出事"

### 审核管线的共性架构

所有平台的安全审核管线基本是同一个结构（差异在每一层的深度和精度）：

```
输入层检测 → 生成过程干预 → 输出层检测 → 发布后巡检
```

### 各平台具体策略

**抖音/字节（推测 + 公开信息）**

| 层级 | 具体做法 | 来源 |
|---|---|---|
| 输入层 | 活体检测（前置摄像头）、明星识别、年龄估计、prompt 敏感词过滤 | 产品可观察到的行为 |
| 生成过程 | Negative prompt 注入（强制加安全相关 negative 词）、特效模板预审 | 推测（业界通用做法） |
| 输出层 | 多模态内容审核（自研）、NSFW/政治/暴恐多维度检测、二次明星识别、显式水印 | 公开报道 + 产品可观察 |
| 发布后 | 内容巡检、用户举报响应、热门内容人工复审 | 字节内容安全公开信息 |

字节的核心优势：**自建多模态审核栈**——不依赖 CLIP 系开源方案，自有训练数据覆盖中文语境（谐音、方言、网络黑话）。这是字节内容安全团队多年积累，是真实壁垒。

**可灵（快手）**

- 审核体系与快手主站共享基础设施
- 输入层：人脸上传时做身份验证（实名 + 活体，待验证）
- 输出层：快手自建审核系统（AiSee 等公开报道的内容安全品牌）
- 特点：快手在内容安全上的投入与字节在同一量级（两家都有数千人审核团队 + 自研模型）

**即梦（字节跳动）**

- 共享字节内容安全基础设施
- 但因为是"专业工具"定位，**审核阈值比抖音特效宽松一些**（允许更多成人向内容、艺术裸体等）——待验证
- 用户群体更小、生成量更低 → 可以承担更高的人工复审比例

**Midjourney**

| 层级 | 具体做法 |
|---|---|
| 输入层 | Prompt 过滤（banned words 列表 + AI 分类器）、不接受人脸上传 |
| 生成过程 | 模型层面做了 safety training（RLHF 对齐） |
| 输出层 | 自动检测 + 社区举报 |
| 发布后 | Discord 社区巡检、ban 违规用户 |

Midjourney 的核心策略：**通过"不做人脸迁移"从根源上规避最大风险**。这是产品选择——他们选择不进入这个雷区。

**DALL-E 3 / GPT-4o（OpenAI）**

| 层级 | 具体做法 |
|---|---|
| 输入层 | System prompt 拒绝真实人物生成请求、prompt 改写（自动添加安全约束） |
| 生成过程 | RLHF + Constitutional AI 训练时对齐 |
| 输出层 | Safety classifier + C2PA 元数据 |
| 发布后 | API 用量监控、违规封号 |

OpenAI 的特点：**重度依赖模型层对齐**——试图在模型内部解决安全问题，而不只是外挂过滤器。优势是更难绕过；劣势是 false refusal 率高（经常拒绝正常请求）。

**美图秀秀**

- 国内平台，遵守相同法规
- 输入层：人脸上传 + 实名验证
- 输出层：内容审核（与美图的 ToB 内容审核服务共用基础设施）
- 特点：美图的场景（美颜/写真）天然 NSFW 风险较低——用户目的是"变美"不是"造假"

### 安全审核的核心 tradeoff

| 维度 | 严 → | ← 松 |
|---|---|---|
| 安全性 | 高（漏报少） | 低（漏报多） |
| 用户体验 | 差（误报多、正常内容被拦） | 好（通过率高） |
| 延时 | 高（多层检测） | 低 |
| 成本 | 高（模型 + 人工） | 低 |

**抖音特效的独特约束**：8 亿 DAU + 免费 → 每一层检测的**单次成本必须极低**，但**总量极大** → 必须是全自动化审核，人工只用于高置信度的边缘 case。

## 维度三：合规策略——中国特色 vs 海外差异

### 中国平台的合规必修项

| 合规要求 | 抖音/即梦做法（推测） | 可灵做法（推测） | 海外平台做法 |
|---|---|---|---|
| 算法备案 | ✅ 已备案（上线前提） | ✅ 已备案 | 不适用 |
| 生成内容标识 | 显式水印"AI 生成" + 不可见水印 | 类似 | C2PA 元数据（可选） |
| 训练数据合法 | 自有数据为主 + 开源 backbone | 类似 | 争议中（Stability AI 被诉） |
| 用户实名 | 手机号实名（中国手机号即实名） | 同上 | 邮箱注册即可 |
| 未成年保护 | 青少年模式 + 年龄估计 + 部分特效限制 | 类似 | 13 岁以下禁止（COPPA） |
| 政治敏感 | 专项过滤（领导人、国旗等） | 同上 | 不适用 |

### 中国 vs 海外的策略差异深度对比

**标识策略**

- **中国**：**强制 + 显著位置 + 元数据** → 监管可查可验
  - 抖音：视频右下角白色文字"AI 生成"
  - 即梦：生成图带水印 + 元数据
- **海外**：**自愿为主 + 行业倡议**
  - OpenAI：C2PA 元数据 + 内容凭证（Content Credentials）
  - Google：SynthID 不可见水印
  - Meta：计划给 AI 生成内容打标签
  - EU AI Act 要求标识但执行细则仍在制定中

**人脸数据策略**

- **中国**：《个保法》将面部数据列为"敏感个人信息" → 收集需要**单独同意**（弹窗 + 勾选，不能默认同意）
  - 实践：抖音特效首次使用人脸功能时有授权弹窗
  - 存储：推测用完即删或短期缓存（法务合规要求）
- **海外**：GDPR 将生物特征列为"特殊类别数据" → 需要明确合法基础
  - 实践：Midjourney 直接不碰人脸上传 → 规避问题
  - OpenAI：ChatGPT 图片编辑的人脸策略仍在演进

**责任归属**

- **中国**：**平台主体责任**——用户违规，平台买单（罚款对象是平台）
  - 这意味着抖音必须**事前审核**，不能事后响应
- **海外**：更多是**通知-删除**模式（Section 230 保护平台）
  - 但 EU DSA 正在推动平台承担更多事前审核义务

### 对抖音的启发

1. **合规不是成本中心，是竞争壁垒**：字节在合规基础设施上的投入（审核模型、明星库、实名体系）形成了高门槛——小团队根本做不到这个覆盖度。
2. **海外平台"不做人脸"的策略在中国不适用**：抖音特效的核心场景就是"把我的脸变成 XX"——不能像 Midjourney 那样绕开。必须正面解决。
3. **合规要"提前量"**：中国监管节奏快，新规频出。产品设计要预留弹性——水印位置、标识方式、授权流程都要做成可配置的，方便监管要求变化时快速调整。

## 维度四：产品设计约束——各家怎么把技术限制变成产品选择

### 约束 1：用户上传路径

| 平台 | 上传方式 | 约束原因 |
|---|---|---|
| 抖音特效 | 前置摄像头实时拍 / 相册选取 | 活体检测 + 快速体验 |
| 即梦 | 相册选取多张 | 高保真需要多角度 |
| 可灵 | 相册选取 | 灵活性 |
| Midjourney | 不支持人脸上传 | 风险规避 |
| 美图 | 相册选取 + 拍照 | 美颜场景 |

**抖音特效的产品选择**：如果强制"只能前置摄像头实时拍" → 安全性最高（活体检测 + 一定程度防止用他人照片）但用户路径长、场景受限（如想用一张好看的旧照片做特效）。如果允许相册选取 → 便利但无法确认是本人。**折中方案**：首次使用做一次活体验证 + 之后允许相册选取。

### 约束 2：失败状态设计

当安全检测拒绝时，怎么告诉用户？

| 平台 | 拒绝方式 | 设计理念 |
|---|---|---|
| 抖音/即梦 | "该内容无法生成，请更换照片/描述"（模糊） | 不暴露检测逻辑（防绕过） |
| Midjourney | "This prompt has been blocked"（直接） | 社区自治 + 透明 |
| OpenAI | "I can't generate that image"（对话式） | 对齐叙事 |
| 美图 | 推测类似抖音（模糊提示） | 合规 |

**产品设计原则**：拒绝提示必须**模糊到不能被用来反向推断检测逻辑**，但**清晰到用户知道该怎么做**（换照片？换描述？还是放弃？）。这是安全产品设计的经典张力。

### 约束 3：多人脸场景的产品降级

| 平台 | 多人脸策略 | 产品表现 |
|---|---|---|
| 抖音特效 | 推测：只处理最大脸 / 提示"仅支持单人" | 场景受限 |
| 即梦 | 推测：分割 + 单独处理 | 算力贵、延时长 |
| 可灵 | 推测：类似抖音 | 场景受限 |
| Midjourney | 不适用 | — |

**产品选择**：与其让多人场景"勉强出但质量差"（脸串、风格不统一），不如**明确拒绝 + 给替代方案**（"检测到多人合影，建议裁剪为单人后重试，或使用 XX 功能"）。清晰的能力边界 > 模糊的勉强结果。

### 约束 4：水印 / 标识的设计平衡

| 维度 | 保守策略 | 激进策略 |
|---|---|---|
| 位置 | 右下角固定 | 覆盖全画面（半透明） |
| 大小 | 小字 | 大字 |
| 可去除性 | 容易裁掉 | 难以去除 |
| 用户体验 | 好（几乎不影响画面） | 差（影响画面美观） |
| 合规满足度 | 最低线 | 超额满足 |

**抖音的当前策略**：右下角小字"AI 生成"——满足合规最低线 + 用户体验优先。配合不可见水印做第二层保障。

**更好的可能方向**：把"AI 生成"标识**做成产品特色而不是合规累赘**——例如设计好看的"AI Created with [特效名]"标签，让用户**主动愿意保留**（就像 Instagram 的滤镜标签）。当标识本身有社交价值时，用户不会想去除。

## 对抖音场景的六条启发

### 1. PuLID 是当前最适合抖音的人脸方案——但不是终局

**理由**：字节自研 → 无许可证风险；对比学习压"ID 污染" → 风格不被压制（正中抖音"惊喜感 > 保真度"的定位）；单图即用 → 符合"快速出特效"的产品路径。

**但不是终局**：PuLID 还是 SDXL 时代的方案，DiT（FLUX 类）已经是下一代 backbone——字节如果切换 backbone，人脸方案也要跟着升级。且多人脸场景 PuLID 也没解决。

### 2. 安全审核管线的延时必须控制在 500ms 以内

**算账**：抖音特效的总生成时间预算（从点击到出图）大约是 3-8 秒。安全管线（前置 + 后置）如果占 2 秒 → 用户可感知的等待增加 25-60%。

**策略**：
- 前置检测（活体 + 明星 + 年龄）：~200ms（本地模型 + 边缘推理）
- 后置检测（NSFW + 二次明星）：~300ms（与生成结果同步/异步结合）
- 水印嵌入：~50ms（轻量操作）

如果后置检测不通过 → 不展示结果 + 重新生成 / 提示用户换输入。**用户不应感知到"先看到了然后又被撤回"**。

### 3. "不做人脸迁移"不是抖音的选项——必须正面解决

Midjourney / DALL-E 选择不碰人脸上传，规避了最大风险但放弃了最大场景。抖音特效的核心价值主张就是"把我变成 XX"——这是用户来的理由。

**意味着**：抖音必须在"人脸安全"上做到行业最好，而不只是"够用"。这是产品定位决定的安全投入等级。

### 4. 明星库 + 未成年识别是上线门槛，不是 V2 优化项

Lensa 的教训：上线时没做 → 翻车后被迫加 → 品牌已经崩了。

**抖音的壁垒**：字节有全中文互联网最大的公众人物库（抖音本身就是明星聚集地）+ 最大规模的人脸数据用于训练识别模型。这个壁垒小团队 / 新玩家无法复制。**产品设计要把"被拦截"的体验做好**——不是冷冰冰的"违规"，而是"检测到可能包含公众人物，为保护他人权益已停止生成"。

### 5. "AI 生成"标识应该做成社交货币而不是合规包袱

当前做法（右下角小字）是**合规最低线**。更好的做法：

- "AI 生成 · 古风武侠特效" → 带特效名 → 观众好奇 → 点击同款 → 形成传播链
- 设计视觉上好看的标签 → 用户主动保留
- 和抖音的"同款特效"功能打通 → 标识本身成为入口

这样用户不会想去除标识——因为标识本身有社交价值。**合规和产品增长可以是同一件事。**

### 6. 多人脸是下一个必须攻克的场景——但先用产品设计兜底

技术上：SAM2 分割 + 单独注入 + 拼接 → 算力 3-5x → 当前成本预算可能不允许。

产品兜底：
- 检测到多人时明确提示"该特效支持单人，请裁剪后使用"
- 推出专门的"多人特效"分类（基于非 ID 注入的方案，如统一风格化但不保 identity）
- 长期：等 ConsisID 类方案成本下降后再开放多人 ID 注入

**"明确能力边界"远好于"勉强出一个半成品"。**
