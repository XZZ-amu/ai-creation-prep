# 生成质量与兜底：抖音场景判断

---

## 1. 判断锚点

抖音特效的质量标准不是"精美"而是"能发"——用户拿到一个结果后 3 秒内判断"行不行"，行就发，不行就关。**产品要做到的是"一次出图就过关"，而不是"出 4 张让用户选"。** 对社交内容来说，"好玩 + 没有明显缺陷"比"精雕细琢"重要十倍。

---

## 2. 核心判断

### 判断 1："好玩"比"精美"重要——抖音特效的质量容忍度远高于商业用途

**理由**：
- 社交内容的消费场景是 feed 流快速浏览，用户看一条内容的平均时间不到 10 秒
- 轻微的手部畸变、背景瑕疵在风格化特效里几乎不被注意——风格越夸张，瑕疵越不明显
- 用户的判断标准是"发出去会不会被嘲笑"，而不是"像素级完美"
- 对比：商业广告/电商图需要 95 分品质，抖音特效 75 分就够用

**对设计的影响**：
- 质量兜底的投入应该集中在"致命缺陷"（脸崩、NSFW、严重变形），而不是"微小瑕疵"
- 不要把后处理管线做得太重——每加一个修复环节就多一份延迟和成本
- 可以容忍一定的"AI 味"——风格化内容里 AI 痕迹反而是一种审美标签

### 判断 2：后处理管线应该极简——只做"人脸修复"一项就够

**理由**：
- **人脸是抖音特效的核心素材**——脸崩/脸糊是唯一让用户立刻放弃的致命缺陷
- 手部修复：风格化场景下手经常被遮挡或不重要，且 HandRefiner 推理慢（增加延迟）
- 超分放大：抖音视频分辨率要求不高（720p 足矣），生成模型输出已够用
- 整体精修（SDXL Refiner）：风格化特效不需要写实质感，精修反而可能破坏风格

**对设计的影响**：
- 后处理管线 = 合规检查 + 人脸修复（CodeFormer/GFPGAN）。到此为止
- 修脸时必须注意 identity 漂移——修清楚但不像本人了等于白修
- 评估字节自研的轻量级修脸方案（如果有），替代开源 CodeFormer 的塑料感

### 判断 3："一次出图就过关"是产品体验的铁律——不应该让用户选图

**理由**：
- 抖音特效的用户预期是"拍/传 → 出结果 → 发"，中间不应该有"选图"环节
- "出 4 张选 1 张"是 Midjourney/即梦的交互模式——适合有审美判断力的创作者
- 普通用户面对 4 张图：要么选不出来（决策疲劳），要么随便选一张（那为什么要生成 4 张？浪费算力）
- 正确做法：系统内部生成 2-4 张候选 → 评委自动选最好的一张 → 呈现给用户

**对设计的影响**：
- 前端体验永远是"一个结果"，不暴露选择
- 后端可以跑 best-of-N（N=2-4），用 reward model 自动挑
- 权衡 N 的大小：N=2 成本 2x 但质量提升有限；N=4 成本 4x 但效果显著。建议 N=2 作为基线，高流量特效可以降到 N=1 + 更强的基座模型

### 判断 4：评委（reward model）应该重点看三个维度，其他不看

**理由**：基于抖音特效场景，评委打分的优先级应该是：
1. **合规性**（一票否决）：NSFW、政治、暴恐、明星肖像
2. **人脸质量**（核心）：脸是否清晰、五官是否正常、是否像本人
3. **整体美学**（加分项）：构图、色彩是否达到模板设计的预期

不需要看的：
- 手部质量（风格化场景不重要）
- prompt 遵循度（抖音特效的 prompt 是模板预设的，不存在用户 prompt 偏离问题）
- 背景细节准确性（社交内容不需要）

**对设计的影响**：
- 评委模型可以非常轻量——不需要 VBench 那种 16 维全面评估
- 训练数据：用抖音内真实特效图 + 用户互动数据（点赞/评论相关性）训一个抖音专属偏好模型
- 合规评委必须前置且快速（<100ms），美学评委可以后置

### 判断 5：兜底策略 = 悄悄重抽，永远不把失败暴露给用户

**理由**：
- 合规被拒 → 悄悄重抽（换 seed 重新生成），不告诉用户"内容违规"
- 质量不达标 → 悄悄重抽，不告诉用户"生成失败"
- 连续 3 次重抽都不过关 → 降级方案（用更保守的 prompt/更低的风格强度重试）
- 最终兜底 → 给一个预计算的"安全结果"（可能是端侧轻量模型生成的备选）

**对设计的影响**：
- 用户永远不应该看到"生成失败"的 toast/弹窗
- 需要一个快速降级路径：cloud inference timeout → 端侧 fallback
- 重抽次数有上限（成本约束），超过上限走降级而不是无限重试

### 判断 6：视频质量兜底比图片难 10 倍，应该分阶段投入

**理由**：
- 视频评估需要帧间一致性检测（闪烁、漂移）——比单图评估复杂得多
- 视频重抽成本是图片的 N 倍（N=帧数）
- 视频的"致命缺陷"不只是单帧问题——一帧脸崩用户可能没注意到，但连续闪烁用户一定注意到

**对设计的影响**：
- 短期：视频特效先做短片段（2-4 秒），减少帧间问题暴露的机会
- 视频质量评估用轻量方案：首尾帧美学分 + 帧间像素差检测闪烁
- 视频兜底不做重抽（太贵），做"逐帧修复"（闪烁帧做帧间插值平滑）

### 判断 7：利用抖音用户互动数据训"抖音审美"的 reward model 是独特优势

**理由**：
- 抖音有海量的"用户点赞/评论/完播率 vs 内容特征"的关联数据
- 这些数据可以训出一个"抖音用户偏好的 reward model"——比 LAION-Aesthetics 通用评委更贴合实际场景
- 字节的数据飞轮优势：用的人越多 → 偏好数据越多 → reward model 越准 → 出图质量越高 → 用的人更多

**对设计的影响**：
- 建议和推荐团队合作，利用已有的内容质量评估信号
- Reward model 应该定期更新——用户审美在变化，半年前高赞的特效半年后可能审美疲劳
- 待讨论：这个数据飞轮在内部是否已经在跑？

## 3. 产品设计建议

### 建议 1：后处理管线极简化——只做合规 + 修脸

```
生成结果 → 合规检查（<100ms）→ 通过？
  → 是：人脸质量检测 → 需要修？
    → 是：轻量修脸（CodeFormer, <200ms）→ 输出
    → 否：直接输出
  → 否：悄悄重抽（换 seed）→ 循环（最多 3 次）→ 降级方案
```

总延迟预算：后处理不超过 300ms。

### 建议 2：成本友好的 best-of-N 策略

- 热门特效（日调用 >100 万）：N=1（压成本），靠更强的基座模型保质量
- 中频特效（日调用 10-100 万）：N=2，评委快速二选一
- 低频/新上线特效（未验证质量）：N=4，确保首批用户体验

随着特效经过验证（合格率 >95%），逐步降低 N 值。

### 建议 3：建设端侧 fallback 能力

云端生成超时或连续失败时，端侧轻量模型给一个"虽然品质低但立刻有"的结果。用户可以选择等云端重试或用端侧版本。

- 端侧模型只做"风格滤镜"级别的变换（非生成式，是确定性变换）
- 成本 = 0（纯端侧计算）
- 品质 80 分 vs 云端 90 分——对社交内容来说够用

### 建议 4："好玩"优先的质量标准重新定义

把内部质量评审的标准从"精美度"改为"发布意愿度"：
- 不问"这张图好不好看"
- 问"看到这张图的用户会不会愿意发到抖音"
- 用 A/B 测试验证：哪些质量维度真正影响发布率，哪些不影响
