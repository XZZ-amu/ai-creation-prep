# 生成质量与兜底：各家解法对比

> 问题谁都有，差异在于怎么兜底。有些靠模型硬扛，有些靠后处理管线补救，有些靠产品设计绕开——三层防线的组合才是完整策略。

---

## 一句话总结

**质量兜底分三层：模型层（让基座本身少出错）、后处理层（出了问题自动修）、产品层（让用户感知不到问题或能自己解决）。各家差异主要在后两层——模型层趋同，产品策略才是真正的竞争壁垒。**

---

## 三层防线框架

```
┌─────────────────────────────────────────────┐
│          产品层（用户感知的最终体验）          │
│  重试机制 / 选图 / 引导 / 预期管理 / 降级     │
├─────────────────────────────────────────────┤
│          后处理层（出图后的修补管线）          │
│  修脸 / 修手 / 超分 / 帧间稳定 / 自动重抽     │
├─────────────────────────────────────────────┤
│          模型层（基座能力）                    │
│  更大模型 / 更好训练数据 / RLHF-DPO / DiT     │
└─────────────────────────────────────────────┘
```

**从下往上**：模型层是地基（决定下限），后处理层是自动补丁（不让用户看到裂缝），产品层是最后的安全网（就算漏了，用户体验也不崩）。

---

## 第 1 层：模型层解法

### 1a. 更大更好的基座模型

**核心逻辑**：模型参数量 ↑ + 训练数据质量 ↑ = 生成质量下限 ↑。

| 模型 | 架构 | 参数量 | 核心改进 |
|---|---|---|---|
| SD 1.5 | UNet | ~860M | 社区基准，问题最多 |
| SDXL | UNet (larger) | ~3.5B | 构图、文字渲染改善 |
| SD 3 | DiT (MMDiT) | ~2B-8B | DPO 对齐、文字能力 |
| FLUX | DiT | ~12B | 文字渲染跨代提升、手部大幅改善 |
| DALL-E 3 | 未公开 | 未公开 | RLHF + caption 改写 |

**关键趋势**：UNet → DiT 的架构转型让模型容量突破了瓶颈。DiT 的 scaling law 更好——参数翻倍，质量提升更线性。

**对抖音的启示**：字节有算力优势，能用大模型。但大模型推理慢——需要蒸馏/量化来压推理时间。模型层是"花钱能买到"的优势，不是差异化壁垒。

### 1b. RLHF / DPO 后训练对齐

**核心逻辑**：不改模型大小，用人类偏好数据把模型"调教"成"出来就好看"的状态。

**两条路线**：

- **RLHF（DALL-E 3 路线）**：训 reward model → PPO 优化画家 → 画家学会画"reward model 喜欢的图"
- **DPO（SD 3 路线）**：直接拿偏好对训画家 → 不用显式 reward model，隐含偏好在画家输出概率里

**效果**：显著提升"开箱质量"——经过 DPO 的模型出来的图第一眼就好看，不需要挑选。

**副作用**：多样性下降。模型被训成"一种偏好的画家"——所有输出风格趋同。SD 3 社区反馈"出来的图都长一个样"。来源：SD 3 发布后 Reddit 社区讨论（2024）。

**对抖音的启示**：
- 基座用 DPO 对齐是标配——提升下限
- 但不能只靠 DPO——特效需要风格多样性，过度对齐反而让特效"都长一个样"
- LoRA 微调 + 风格数据是保持差异化的关键

### 1c. 更好的训练数据

**核心逻辑**：模型质量上限 = 训练数据质量上限。垃圾进垃圾出。

**各家做法**：
- **DALL-E 3**：用 GPT-4V 重写所有训练图的 caption，使 prompt-图对齐度大幅提升（来源：DALL-E 3 technical report, 2023）
- **Midjourney**：自建高美学数据集（未公开细节），被认为是其"审美"优势的核心来源
- **FLUX**：高质量合成数据 + 筛选后的真实数据组合（来源：Black Forest Labs 官方博客）

**对抖音的启示**：抖音有海量用户生成内容 + 用户互动数据（点赞/评论/完播率）——这是天然的"偏好对"数据源。理论上可以训出"抖音用户偏好的 reward model"。待验证：字节内部是否已在利用这些信号。

## 第 2 层：后处理层解法

### 2a. 修补匠串行管线

**核心逻辑**：生成完后串一系列自动修复模块，每个模块专治一类问题。

**典型管线顺序**：

```
原始生成 → 合规检查 → 人脸修复 → 手部修复 → 整体精修 → 超分放大 → 最终输出
```

**各模块的工具选项**：

| 修复类型 | 工具 | 优点 | 缺点 |
|---|---|---|---|
| 人脸修复 | CodeFormer | 抗严重退化，稳定 | 偏塑料感，identity 漂移 |
| 人脸修复 | GFPGAN V1.4 | 自然度高 | 轻微 identity 漂移 |
| 人脸修复 | ADetailer（自动检测+重绘） | 无需手动标注，端到端 | 依赖检测精度 |
| 手部修复 | HandRefiner | 结构正确性强（mesh 约束） | 慢，极端姿态仍不稳 |
| 手部修复 | 手部 LoRA + inpainting | 更快，效果看 LoRA 质量 | 无物理约束 |
| 整体精修 | SDXL Refiner | 纹理质感提升 | 不救结构性错误 |
| 超分 | Real-ESRGAN | 忠实放大，稳定 | 可能过度平滑 |
| 超分 | SD Upscale（latent） | 能"补细节" | 可能幻觉 |
| 视频帧间稳定 | Temporal smoothing | 减少闪烁 | 可能模糊运动细节 |

**关键产品决策**：
- 管线里每多一步 = 延迟增加 50-200ms + 成本增加
- 必须区分"默认必跑"和"按需触发"
- 视频场景下管线成本按帧数倍增——必须极精简

### 2b. Best-of-N 自动选图

**核心逻辑**：生成 N 张 → 用 reward model 打分 → 选最高分交付。

**各家的 N 值和评分策略**（推测 + 公开信息）：

| 产品 | 推测 N 值 | 评分依据 | 呈现方式 |
|---|---|---|---|
| Midjourney | 4（用户可见） | 内部美学+偏好模型 | 4 宫格让用户挑 |
| DALL-E 3 | 未知（内部选） | RLHF reward model | 直接给最好的 1 张 |
| 即梦/可灵图片 | 4（用户可见） | 内部评分 | 多张供选择 |
| 抖音特效（推测） | 2-4（内部选） | 内部评分 | 直接给 1 张 |

**ImageReward 的 Best-of-N 数据**：
- N=4 时，77.1% 的人偏向 ImageReward 选的图 vs 随机选（来源：ImageReward 论文）
- 边际收益递减：N=4→N=8 的提升远小于 N=1→N=4
- 成本是线性的：N=4 就是 4 倍算力

### 2c. In-process Pruning（生成中剪枝）

**核心逻辑**：不等画完再评，在去噪中途就判断"这条路走偏了"，提前放弃并重新分叉。

**学术进展**：
- ImageReward ReFL：在去噪 t≥30 步时用 reward predictor 评估中间结果
- Reward-Guided Diffusion：把 reward gradient 直接加到去噪方向上
- DRAFT（2024）：训练专门的"中途质量预测器"，在第 10 步就能预测最终质量

**实际落地情况**：公开的生产环境部署案例极少。推测 Midjourney 和字节内部可能在用某种形式的 in-process 评估，但没有公开确认。待验证。

**对抖音的启示**：这是抖音特效"一次过关"的关键技术路径——用户不能等 4 张画完再选，但又需要质量保证。In-process pruning 是唯一能在"单张生成时间"内完成质量兜底的方案。

### 2d. 视频专项后处理

**视频比图片多的后处理需求**：

| 处理 | 作用 | 工具/方法 |
|---|---|---|
| 帧间平滑 | 去闪烁 | Temporal bilateral filter、光流引导融合 |
| 视频超分 | 提升分辨率 | Real-ESRGAN-Video、BasicVSR++ |
| 人脸一致性 | 跨帧保持同一张脸 | 逐帧 identity embedding 约束 |
| 运动插帧 | 补帧使运动平滑 | RIFE、AMT |

**关键矛盾**：视频后处理的每一项都是帧级别的成本——3 秒 24fps = 72 帧。对每帧跑修脸+超分 = 72 倍图片管线成本。实际生产中必须只对关键帧跑完整管线，中间帧用轻量插值。

## 第 3 层：产品层解法

### 3a. 重试机制设计

**各家做法对比**：

| 产品 | 用户触发 | 自动触发 | 重试体验 |
|---|---|---|---|
| Midjourney | "Re-roll"按钮 | 未知 | 快，用户习惯了 |
| DALL-E 3 | 重新生成按钮 | 合规拒绝自动重抽 | 较慢，有明确等待 |
| 即梦 | 重新生成 | 未知 | 等待时间较长 |
| 可灵 | 重新生成 | 未知 | 等待时间长 |
| 抖音特效（推测） | 无明确按钮/待验证 | 合规+质量阈值 | 必须极快 |

**产品设计关键**：
- 抖音场景不能有"重试等待"——用户在"拍摄流"里，2 秒的额外等待就是流失
- 自动重抽必须对用户透明——不能显示"质量不够正在重新生成"
- "重新生成"按钮应该是轻触可达但不是必须的——让满意的用户直接发布

### 3b. 选图交互设计

**各家的呈现方式**：

- **Midjourney**：4 宫格是核心交互——选中一张后可以 Upscale / Variation。用户"挑"的动作被设计成了核心乐趣
- **DALL-E 3（ChatGPT 里）**：默认出 1 张，用户不满意可以要求重新生成或微调 prompt
- **即梦**：出多张供选择，可以点击放大、选中继续编辑
- **可灵**：图片模式出多张，视频模式出 1 条
- **抖音特效**：必须出 1 张/1 条——普通用户不会挑选

**关键洞察**："选图"是创作者工具的交互，不是普通用户特效工具的交互。抖音特效的用户心智模型是"按一下 → 出结果 → 发布"，中间加任何决策节点都在增加流失。

### 3c. 引导与预期管理

**各家做法**：

- **Midjourney**：强引导 prompt 写法（/imagine 格式 + 参数）——用户学会写好 prompt 后质量提升。但学习门槛高
- **DALL-E 3**：GPT-4 自动改写 prompt（把用户的模糊描述扩展成详细描述）——降低了 prompt 工程门槛
- **即梦/可灵**：提供模板+示例引导，告诉用户"怎么描述能得到好结果"
- **抖音特效**：不需要用户写 prompt——特效已经内置了 prompt/LoRA/参数，用户只提供素材（自拍）

**对抖音的启示**：
- 抖音特效的"引导"不是教用户写 prompt，而是**引导用户上传合适的素材**（光照好的正脸、背景简洁等）
- 可以在拍摄引导阶段就提示"正面光更好哦"——把质量问题前移到输入端
- 特效的"适用范围"应该透明——比如"这个特效对侧脸效果不好"

### 3d. 优雅降级策略

**核心逻辑**：当质量实在兜不住时，怎么体面地失败。

**各家做法**：
- **DALL-E 3**：直接报错"无法生成此内容"（合规原因）——体验生硬
- **Midjourney**：几乎不报错，总能出东西——但可能质量差
- **即梦/可灵**：长时间生成后报错"生成失败请重试"——最差体验
- **抖音特效（应该的做法）**：永远出结果，只是在质量阈值之间选择不同策略

**优雅降级的层级设计**：

```
质量极好 → 直接输出
质量中等 → 跑后处理管线修补后输出
质量偏差但能看 → 输出 + 小概率弹"再试一次？"按钮
质量极差/合规拒绝 → 悄悄重抽 → 如果还不行 → "换个特效试试"
```

**关键原则**：用户永远不应该看到"生成失败"这四个字——那是产品的失败，不是用户的问题。

### 3e. 算力动态分配

**核心逻辑**：不是每张图都值得花同样的算力——根据场景/用户/流量动态调整质量策略。

**可能的策略**：
- 低峰期：Full pipeline（修脸+修手+超分+Best-of-4）
- 高峰期：精简 pipeline（跳过修手+降级超分+Best-of-2）
- 高互动用户（经常发布的活跃创作者）：给更多算力保证质量
- 低互动用户（可能只是试玩）：快速出结果优先

**对抖音的启示**：这是字节的强项——推荐系统已经证明了"千人千面"的算力分配能力。把同样的逻辑用到"生成质量的精细化管控"上是自然延伸。

## 各产品策略对比

### 可灵

| 层级 | 做法 |
|---|---|
| 模型层 | 自研视频大模型，持续迭代基座质量 |
| 后处理层 | 待验证（推测有修脸+超分管线） |
| 产品层 | 图片出多张供选择；视频出 1 条 + 重新生成按钮；有"高质量"选项（耗时更长） |
| 特点 | 偏专业创作者定位，允许等待换质量 |

### 即梦

| 层级 | 做法 |
|---|---|
| 模型层 | 字节自研模型，和可灵共享部分基础设施（待验证） |
| 后处理层 | 待验证（从产出质量看有修脸和超分） |
| 产品层 | 出多张供选择；有画质选项；支持局部重绘（让用户自己修） |
| 特点 | 定位在"创作者精细化编辑"，用户自己是质量把关者 |

### Midjourney

| 层级 | 做法 |
|---|---|
| 模型层 | 完全闭源自研模型，以"审美"闻名 |
| 后处理层 | 内部完全黑箱——推测有自动修复 + 超分 |
| 产品层 | **4 宫格是核心设计**——把"选图"变成了用户体验的一部分而非负担；Vary 功能让用户微调 |
| 特点 | 把 Best-of-N 的"选"变成了产品优势而非产品缺陷——让用户觉得"4 张都给我是服务好" |

**MJ 的产品智慧**：它没有试图隐藏"质量波动"，而是把波动变成了"给你更多选择"的正面叙事。但这个策略只对创作者用户有效，对普通用户（抖音场景）不适用。

### DALL-E 3

| 层级 | 做法 |
|---|---|
| 模型层 | RLHF 对齐 + GPT-4V caption 重写（改善训练数据） |
| 后处理层 | 未公开——推测有合规筛选 + 质量评分 |
| 产品层 | 默认出 1 张；GPT-4 自动改写 prompt（把质量问题前移到输入端）；合规拒绝直接报错 |
| 特点 | **把 prompt 改写作为质量保障手段**——用户写"一只猫"，系统扩展成详细描述，从源头减少歧义 |

**DALL-E 3 的核心逻辑**：与其出了问题再补，不如从 prompt 就消除歧义。GPT-4 的 caption rewriting 本质是"把用户模糊意图翻译成模型能精准执行的指令"——这是产品层和模型层的融合创新。

### Runway Gen-3

| 层级 | 做法 |
|---|---|
| 模型层 | 自研视频模型，Gen-3 Alpha 在运动质量上领先 |
| 后处理层 | 未公开 |
| 产品层 | 出 1 条视频 + 重新生成；支持 motion brush（引导运动方向减少随机性）；有预览帧 |
| 特点 | **Motion Brush 把"运动质量"问题交给用户控制**——不是让模型猜怎么动，而是用户画怎么动 |

## 对抖音特效场景的启发

### 1. 抖音特效必须走"后端全自动"路线

可灵/即梦/MJ 都把"质量把关"的一部分交给了用户——让用户选图、让用户手动重试、让用户局部编辑。**这在抖音特效里完全不可行**：
- 用户在拍摄流里，每多一步操作就多一次流失
- 普通用户没有"评判生成质量"的能力和意愿
- 特效的体验是"按一下 → 出来 → 发"，不是"按一下 → 选 → 调 → 发"

**结论**：所有质量评判、修复、重抽必须在后端自动完成，对用户表现为"直接出好结果"。

### 2. DALL-E 3 的"输入端优化"思路值得借鉴

DALL-E 3 用 GPT-4 改写 prompt 的逻辑是：**与其出了问题再补，不如从输入就消除歧义**。在抖音特效里的对应是：
- 引导用户上传高质量素材（光照好、正脸、背景简洁）
- 特效设计时内置"最优 prompt + 参数"——用户不需要调
- 根据用户素材质量自动调整生成参数（素材质量差 → 加强后处理管线）

### 3. Midjourney 的"把缺点变优点"思路启发产品叙事

MJ 把"生成不稳定要选图"变成了"给你 4 张选择是服务好"——**同样的技术限制，不同的产品叙事**。在抖音特效里：
- "AI 每次结果不同"可以被叙事为"每次都是独一无二的创作"
- "生成需要时间"可以被叙事为"AI 正在精心为你定制"
- 关键是：叙事必须配合实际体验——如果等了 10 秒出来的东西很丑，再好的叙事也没用

### 4. 算力动态分配是字节的独有优势

其他产品（MJ、DALL-E）面对的是固定定价的 API 场景——每个用户花同样的算力。抖音作为平台方，可以做精细化的算力分配：
- 根据特效热度分配：爆款特效给更多后处理预算
- 根据用户价值分配：高发布率用户给更好的质量体验
- 根据时段分配：低峰期跑完整管线，高峰期精简管线
- 这种"千人千面的质量策略"是推荐系统思维在生成领域的延伸

### 5. 视频质量兜底比图片难一个数量级

图片可以花 2-3 秒跑修脸+超分完整管线；视频的每一帧都要处理，且帧间必须一致。**视频的质量兜底策略必须比图片激进得多地简化**：
- 关键帧精修 + 中间帧插值，而非逐帧跑完整管线
- 视频评估只看首尾帧+抽样帧（2-3 帧），不逐帧评分
- 宁可接受"轻微模糊"也不接受"帧间不一致"——不一致比模糊更让人难受
- 视频后处理的优先级：帧间稳定性 > 单帧清晰度 > 单帧美学

### 6. "永不报错"是产品底线

从各家对比看，"生成失败请重试"是最差的用户体验。抖音特效应该：
- **永远给出结果**——哪怕质量不是最优的
- **用"换个特效试试"代替"生成失败"**——把问题归因到"匹配"而非"失败"
- **把重抽做成后台操作**——用户看到的是"正在生成"的统一等待，不是"失败了正在重试"
- **实在不行就降级**——用预生成的通用模板兜底，保证"总有东西出来"
