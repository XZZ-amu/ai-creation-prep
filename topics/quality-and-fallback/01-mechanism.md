# 生成质量评估 + 失败兜底：一张图怎么算"过关"，不过关怎么救

> 写作说明：本文是 v2 范式的延伸——沿用 [intent-understanding/01-mechanism-v2.md](../intent-understanding/01-mechanism-v2.md) 的"画室五人组"比喻体系，并新增一个角色（**评委**）。如果还没读过 v2，建议先读那篇——本文里"翻译官 / 画家 / 草稿世界 / 骨架图 / 风格手册"五个角色不再重新解释。

---

## 一句话总结

**"出图质量"不是模型一锤定音，是画室门口有一个评委把关——画家画完一摞草稿，评委挑出最像样的那张交给用户。**
评委本身有几种打分体系（美学分、人类偏好分、合规分、视频还要看时序）。挑完之后，如果还差点意思，画室还配了几个"修补匠"专门处理：脸糊了找修脸匠，手画歪了找修手匠，整张糊了找放大匠。
**最关键的产品决策不是"用哪个评委"，而是"评委放在哪个环节"——出图后挑选 vs 出图中剪枝**。前者是"出 4 张让用户挑"的产品兜底，后者是"默认一次过关"的技术兜底。**抖音特效场景必须走后者**——普通用户不会看 4 张选 1 张。

---

## 贯穿案例

整篇文档围绕这一个抖音特效场景走完全程：

> 用户在抖音 AI 特效里上传自拍，选"魔法少女变身"特效，1.5 秒后出一张/一段视频。**用户期望**：一次出图就好看、像自己、能直接发出去。**系统要做的**：背后可能算了 4-8 张候选，用一套评委体系挑了最好的一张，路上经过修脸/修手/超分一系列修补，最后呈现给用户的是"过关版"。

每一种机制讲完，都会回答一个问题：**这一关在魔法少女这个场景里是怎么生效的？哪种失败它救得了，哪种救不了？**

## 画室六人组（在 v2 五人组基础上 + 一个评委）

| 角色 | 对应概念 | 它在干什么 |
|---|---|---|
| 翻译官 | Text encoder | 把人话翻成画家能读的语义矩阵 |
| 画家 | UNet / DiT | 真正动笔，从噪声去噪到结果 |
| 草稿世界 | Latent space | 画家工作的压缩空间 |
| 骨架图 | ControlNet | 硬约束：形状必须按骨架来 |
| 风格手册 | LoRA / IP-Adapter | 软约束：按这个味道画 |
| **评委（新）** | Reward model / Aesthetic predictor / VBench | 画完之后给画打分，不及格的退回去重画 |

外加几个**修补匠**（不是常驻角色，按需召唤）：
- **修脸匠**（CodeFormer / GFPGAN）：脸糊了、五官歪了找他
- **修手匠**（HandRefiner / hand LoRA）：手指数错了找他
- **放大匠**（Real-ESRGAN / SD Upscale）：分辨率不够、要发到大屏找他
- **二阶画家**（SDXL Refiner）：第一遍画粗了，让他用更细的笔补一遍

## 第 1 关：评委是谁——四种打分体系

评委不是一个人，是一个委员会，每个委员看的东西不一样。**抖音特效背后大概率几个委员同时在打分，最后加权出个总分。**

### 1a. 美学评委（Aesthetic Predictor）

**干什么**：单看"好不好看"。不管 prompt、不管像不像目标，纯粹从构图、配色、清晰度打分，输出 0-10 的连续值。

**机制**：拿大量"打过分的图"训一个简单 MLP，接在 CLIP 图像编码器后面。LAION-Aesthetics 数据集是几十万张图 + 人工评分，训出来的预测器是 VBench 里"美学质量"这一维度的实现。

**比喻不准在哪**：说"评委"高估它——**它学的是统计偏好**："训练数据里被打高分的图大多是高对比、构图饱满、风格鲜明"。对训练数据外的"好看"（极简风、低饱和高级感）会打低分。**美学分高 ≠ 真好看，美学分高 = 像评委训练时见过的好看**。

**回到魔法少女**：
- ✅ 高饱和闪光特效华丽服装——美学预测器一般高分；
- ⚠️ 它对"像不像用户"完全无感——单靠它选不出"自拍变身"该选哪张，所以不能独裁。

### 1b. 人类偏好评委（HPS v2 / ImageReward / PickScore）

**干什么**：综合"好不好看 + 是不是按 prompt 来 + 整体合不合人意"，输出标量分数。给 best-of-N 选图用，也给训练时 RLHF/DPO 当反馈源。

**机制（以 ImageReward 为例）**：收 8878 个 prompt × 4-9 张图，让标注员排序得到 13.6 万对"A 比 B 好"的偏好对；用 BLIP 做 backbone + MLP 头输出标量；pairwise loss 训练 `score(A) > score(B)`。HPS v2 是同思路 + 更大数据集（79.8 万对），用 CLIP 做 backbone。

**比喻不准在哪**：说"人类偏好"过度神圣化。**它学的是标注员群体的统计偏好**——HPS v2 对"动漫女孩 + 闪亮特效"打分天然偏高（训练数据里这类被标得多），对"性冷淡极简风"偏低。**不是普世审美，是某个群体的审美统计画像**。

**ImageReward 的关键数字**（论文 best-of-N）：4 张候选选 1 张，preference accuracy 65.14%（CLIP score 54.82%、aesthetic 57.35%）；vs 随机选，77.1% 的人偏向 ImageReward 选的那张。

**回到魔法少女**：
- ✅ 知道"魔法少女 + 闪光"应该是华丽的，能拒绝姿势歪、构图垮的失败图；
- ⚠️ 依然不看 identity——face embedding 相似度是另一套评委；
- ⚠️ 候选都"差不多"时判别力不强，分数都在 0.6-0.7 之间。

### 1c. 合规评委（NSFW filter / safety classifier）

**干什么**：一张图不管多好看，如果触发了 NSFW、暴力、未成年人风险，**直接打回 0 分，进入"重抽"分支**。

**机制**：通常是一个轻量级 CLIP 分类器，在 NSFW 数据上微调，输出"safe / unsafe"概率。Stable Diffusion 默认管线带一个 safety_checker（已被很多复刻版关掉了）；商业产品（MJ、DALL-E 3、即梦）都有更复杂的多分类器组合。

**和美学/偏好评委的关系**：是**前置一票否决**。先过合规，再看美学。合规没过，根本不进入候选池。

**回到魔法少女**：
- 用户上传的自拍是合规的，但生成结果里如果衣服画得过短、姿态过暴露，会被合规评委拒掉，触发 auto-retry。
- ⚠️ 这里有个产品坑：**抖音特效不能直接报错"内容违规"给用户**——用户根本不知道自己做错了什么（他只是上传了张自拍）。所以兜底必须是悄悄重抽，而不是把锅甩给用户。

### 1d. 视频专属评委（VBench）

**干什么**：图片只看一张，视频要看一段。VBench 把视频质量拆成 16 个维度——空间 7 维（美学、清晰度、对象正确等），时间维度上有主体一致性、背景一致性、时序闪烁、运动平滑度、动态程度等。

**关键维度怎么算**（VBench 论文具体方法）：
- **主体一致性**：DINO 抽每帧特征，算相邻帧 + 和首帧的余弦相似度；
- **背景一致性**：换 CLIP 特征做同样的事；
- **时序闪烁**：静态场景 prompt（光流过滤过的）+ 相邻帧像素 MAE；
- **运动平滑度**：丢奇数帧，用插帧模型 AMT 重建，比对差异；
- **动态程度**：RAFT 光流，识别是否有视频"作弊"靠不动骗过一致性分。

**比喻不准在哪**：VBench 不是单个评委，是**16 个独立指标的诊断书**，不是一个总分。**它做模型评测，不做实时 best-of-N 选片——16 维全跑要几十秒，太贵**。

**生产环境视频评委更轻量**：通常用"首尾帧美学分 + 帧间像素差检测闪烁"这种近似指标。

**回到魔法少女（视频版）**：
- ✅ "魔法少女转身一周"——VBench 能查出"第 1.2-1.4 秒主体一致性掉到 0.6"，对应用户感知的"突然换了一张脸"；
- ⚠️ 实际产品要选哪几个维度卡阈值、哪些只看不卡，是产品决策。

### 用户视角的失败现象（评委层）

- **"出来的图很丑但模型说它选过了"**——评委训练分布偏离了你的审美（美学评委只见过华丽风，给极简风打低分）；
- **"出来的图好看但不像我"**——评委没看 identity，face similarity 是另一套体系；
- **"出来内容被悄悄换了"**——合规评委拒了第一张，触发 auto-retry，给你的是第二张；
- **"视频前 2 秒帅后 1 秒崩"**——视频评委只看首帧或抽样帧，没盯到具体哪一秒掉链子。

## 第 2 关：评委放哪——出图后挑 vs 出图中剪枝

这是本话题里**最关键的产品策略决策**，比"用哪个评委"重要得多。

### 2a. 出图后挑选（Best-of-N）

**做法**：画家画 N 张（典型 N=4 或 8），全部画完再让评委打分，挑最高分一张交付。

**优点**：
- 实现简单——画完跑一次评委；
- 上限高——4 张里大概率有一张是好的；
- 有"展示候选让用户挑"的产品延伸（MJ 的 4 宫格就是这个范式的极致版）。

**缺点**：
- 算力是 N 倍——视频场景下 N=4 直接把成本拉到 4 倍；
- 用户感知到延迟变长——必须等所有候选画完；
- 浪费严重——3 张被丢进垃圾桶。

**回到魔法少女**：
- 抖音特效如果走 N=4：1.5 秒变 6 秒，用户大概率划走；
- 即梦 / 可灵这类专业工具走 N=4 用户能接受——他们就是要挑；
- 抖音特效不能这么走。

### 2b. 出图中剪枝（in-process pruning）

**做法**：在生成的某个中间步骤就让评委介入打分，把"明显走偏"的路径提前终止，把算力留给"看起来有戏"的路径。

**机制（学术界叫 Reward-Guided Search）**：
- 在 latent diffusion 中段（比如 50 步去噪走到第 30 步），用一个轻量级 reward predictor 对当前的中间 latent 打分；
- 分数低的样本直接丢弃，重新从前几步分叉重抽——把算力集中在还有救的路径上。

**ImageReward 的 ReFL 思想已经在做这件事**——它发现"在去噪 t≥30 步时，预测出的 x_0' 已经有质量区分度了"，所以可以用 reward model 在中段做反馈。

**优点**：
- 单次出图成本接近 N=1，但效果接近 N=4；
- 用户感知延迟稳定。

**缺点**：
- 工程复杂度高——要改去噪管线，要训配套的中段 reward predictor；
- reward predictor 在中段的判断不一定准（图还没画完，看不全）；
- 调度策略很玄学（什么时候开始打分、低于多少分要重抽）。

**回到魔法少女**：
- ✅ 这是抖音特效的正解方向——保住"用户感知一次过关"，但背后的成本是单张生成；
- ⚠️ 但工程门槛高，产品上线初期更可能用退化版本（生成 2 张、快速评分挑 1 张），后续再演进到真正的 in-process。

### 2c. RLHF / DPO（把评委的偏好注入画家本身）

**做法**：不是事后挑选，而是**用评委的分数当训练信号，反向调画家的脑子**——让画家从一开始就更倾向于画评委喜欢的图。

**两条主流路线**：

**路线 A：RLHF + reward model（DALL-E 3、ImageReward 的 ReFL）**
- 训一个 reward model（如 ImageReward）；
- 用 PPO / ReFL 类算法，让画家最大化 reward model 给的分；
- 画家学会"画那种 reward model 给高分的图"。

**路线 B：DPO（SD 3 用的方法）**
- 不训单独的 reward model，直接拿"人类标注的偏好对"（A 比 B 好）当训练信号；
- 用 DPO 算法直接调画家：让画家给 A 的概率增加、给 B 的概率减少；
- 隐含 reward 在画家自己的输出概率里，**省掉一个模型**。

SD 3 论文明确说自己用 DPO，没用显式 reward model。DALL-E 3 公开信息少，普遍认为它走 RLHF 路线（OpenAI 系一贯做法）。

**比喻不准在哪**：说"把评委的偏好注入画家"听起来像画家变聪明了，**但实际上画家只是被调成了一种特定偏向的画家**——它现在画啥都偏华丽、偏高对比、偏中心构图。这就是为什么 RLHF 调过的模型常被批"画得都长一个样"——评委的偏好被刻进画家的肌肉记忆，画家失去了多样性。

**回到魔法少女**：
- ✅ DALL-E 3 / SD 3 出来的图普遍"第一眼漂亮"，不需要挑——这是 RLHF/DPO 的红利；
- ⚠️ 但同一个 prompt 出来 4 张很相似——多样性被牺牲；
- ⚠️ "想要一种小众风格"用 RLHF 模型很难——评委没见过的风格全被打低分。

### 三种放置策略对照

| 策略 | 在哪用评委 | 算力成本 | 上限 | 多样性 | 用户延迟 |
|---|---|---|---|---|---|
| **Best-of-N** | 出图后 | N 倍 | 高 | 高（N 张候选） | 高（等 N 张） |
| **In-process pruning** | 出图中 | 接近 1 倍 | 中高 | 中 | 低（=单张） |
| **RLHF / DPO** | 训练时 | 训练贵，推理 1 倍 | 中（评委偏好被固化） | 低（趋同） | 低（=单张） |

**抖音特效的现实选择**：底层模型用 RLHF/DPO（继承"一次过关"红利）+ 推理时叠 in-process pruning（兜底走偏的样本）+ 极少数情况下 fallback 到 best-of-2（用户体感不强烈但提升上限）。

## 第 3 关：修补匠——评委不及格之后怎么救

评委判了"不够好"——不是所有失败都要重抽。很多时候只有一小块崩了（脸、手、清晰度），找对应的修补匠局部救一下即可。

### 3a. 修脸匠（CodeFormer / GFPGAN）

**干什么**：图整体没问题，但脸糊了、五官歪了——专门把脸抠出来重画。

**两种思路对比**：

**CodeFormer（codebook 路线）**：先训面部 VQGAN，把高质量人脸压缩成"码本"（几千上万个合格人脸特征片段）；推理时让 transformer 去码本里查"哪些片段拼起来最像这张脸"。重建结果一定在"合格人脸流形"上，特别抗严重退化。关键参数 `w`（0-1）：w 小靠码本（更"假"更清晰），w 大靠输入（更"像原脸"更糊）。

**GFPGAN（GAN prior 路线）**：用预训练 StyleGAN2 当"人脸先验"，训练时模拟各种退化让网络学反向恢复。输出更锐利，但 V1.3 模型卡坦承"slight change on identity"（轻微 identity 漂移）。

**真实表现**：CodeFormer 在严重退化上更稳但偏塑料感；GFPGAN 在轻度退化上更自然但 identity 不太保。产品里常见"两个都接，按图质量自动选"。

**ADetailer：自动化版本**——SD WebUI 扩展，**不是新的修脸技术，是把"检测→mask→重画"自动化的工程胶水**。跑 YOLOv8 检测脸、生成 mask、走 inpainting 重画。抖音特效后端大概率有类似管线——用户根本不知道有"修脸"这一步。

**回到魔法少女**：
- ✅ 自拍变身的原始脸只有 200×200 像素细节，整张图 1024 看着很糊——CodeFormer 抠出来重画到 512 再贴回去；
- ⚠️ 修脸不保 identity——魔法少女 vs 用户原脸的相似度要靠 IPAdapter-Face / InstantID / PhotoMaker 等专模；
- ⚠️ 视频修脸还要保跨帧一致——逐帧独立修会闪。

### 3b. 修手匠（HandRefiner / Hand LoRA）

**干什么**：手指数错了（六指、四指、扭曲），单独把手抠出来按解剖结构重画。

**HandRefiner 的做法（论文核心）**：检测手部区域 → 用**手部 mesh 重建模型**（一定遵守 5 指、关节位置正确的物理结构）抠出对应姿势的标准 mesh → 渲染成深度图 → 用 ControlNet（depth）+ inpainting 重画手部——ControlNet 强制按 mesh 形状走。

**关键思路**：**"画家画不好手"不是教画家学解剖，是给画家递一张正确的手骨架图**——回到 v2 里 ControlNet 的角色。手在训练数据里姿势千变万化 + 互相遮挡 + 手指相似度高，画家学不好"5 指必须 5 指"这条硬约束。FLUX、SD3 的手比 SD 1.5 好一个量级，但极端姿势（弹钢琴、握物件）还是会崩。

**回到魔法少女**：
- ✅ 举杖姿势的手指穿模能用 HandRefiner 兜底；
- ⚠️ 视频里逐帧修手太贵——"手部一致性"基本靠模型本身硬扛；
- ⚠️ 修手在产品端常是开关——"高质量模式"开、"快速模式"关。

### 3c. 二阶画家（SDXL Refiner）

**干什么**：第一遍粗画 + 第二遍精修。SDXL 推荐"base 画前 80%、refiner 画后 20%"——两个模型各管一段噪声水平。

**机制**：主流用法是 **Ensemble of Expert Denoisers**（源自 NVIDIA eDiff-I）——同一个去噪过程拆两段：base 跑高噪声段（构图、布局），refiner 跑低噪声段（细节、纹理），中途切换模型。refiner 专精低噪声段，比 base 在细节上更准。

**比喻不准在哪**：把 refiner 说成"修补匠"不准——**它不是"哪里坏了修哪里"，是"整张图统一过一遍精修"**。强项在细节质感、整体清晰度，对局部明显崩了（脸糊、手歪）的修复力有限——那种还得靠 ADetailer/HandRefiner。

**回到魔法少女**：
- ✅ refiner 让整张图质感更好——皮肤纹理、衣服褶皱、闪光特效更细腻；
- ⚠️ 救不了构图崩、姿势错——构图在 base 阶段就定了；
- ⚠️ FLUX、SD 3 这一代不再用 base+refiner 双模型——单模型直接到位。

### 3d. 放大匠（Real-ESRGAN / SD Upscale）

**干什么**：图本身没问题，但分辨率不够——512×512 在 1080p 视频里看着糊。

**Real-ESRGAN 的做法**：用纯合成数据训练——人为给清晰图加各种退化（模糊、噪声、JPEG 压缩、缩小），让网络学反向恢复。推理给一张糊图，输出 4× 分辨率清晰版。有动漫专版、视频专版、通用专版，选错版本会有伪影。

**Real-ESRGAN vs SD Upscale**：Real-ESRGAN 更"忠实修复"；SD Upscale 在 latent 空间放大再解码，会"创造"细节（凭模型记忆补充），更花哨但偶尔幻觉。

**回到魔法少女**：
- ✅ 模型 1024 出图 → Real-ESRGAN 4× → 下采到 1080p 视频看着锐——抖音特效几乎必跑的最后一步；
- ⚠️ 视频超分算力是图片的 N 倍（N = 帧数）——实时不可能逐帧 Real-ESRGAN，常用视频超分专模 + 时序约束。

## 第 4 关：自动重抽（Auto-retry）

最后一道兜底——如果所有评委都不及格，且修补匠也救不回来，怎么办？

**做法**：换 seed 重抽，从头再来一次。一般有几种触发条件：
1. **合规拒绝**——NSFW 评委判定 unsafe，自动换 seed；
2. **质量阈值**——HPS v2/ImageReward 综合分低于阈值（比如 0.3），重抽；
3. **特定失败模式**——人脸检测器没检测到脸（生成失败到没出脸）、object detector 没找到目标物体；
4. **用户主动**——用户点"重新生成"按钮（这是最可靠的兜底，也是最贵的）。

**比喻不准在哪**：把 auto-retry 说成"兜底"听起来很安全，**但每次重抽都是 N 倍延迟和 N 倍成本**。1 次成功率 70% 的模型，最多重抽 3 次后总成功率约 97%——但平均算力变成 1.4 倍（70% 一次过、21% 两次过、6% 三次过）。

**抖音场景的硬约束**：
- 用户不会等"重抽进行中..."的转圈圈超过 3 秒；
- 大流量下 auto-retry 是真金白银的成本——同接入量下，重抽率从 5% 涨到 20% 直接让 GPU 成本翻几倍。

**所以 auto-retry 的产品策略**：
- 合规拒绝必须重抽（不能让用户看到不当内容）；
- 质量阈值要谨慎设——设太高每张都重抽，设太低形同虚设；
- 抖音侧比即梦更倾向**"差也发"**——评分稍低也送出去，因为重抽延迟更不可接受；
- 极端失败（脸完全没生成、严重崩坏）才走 auto-retry，常规失败让产品其他层兜底。

## 把六类机制摆在一起

| 机制 | 解决什么 | 抖音特效里用吗 | 算力成本 | 质量上限 |
|---|---|---|---|---|
| **美学评委** | 整体好不好看 | 用（in-process pruning 的打分组件） | 极低（一个 MLP） | 中（统计偏好） |
| **人类偏好评委（HPS/ImageReward）** | 综合质量 | 用（best-of-N、训练 reward） | 低（一次推理） | 高（人类对齐） |
| **合规评委（NSFW filter）** | 内容安全 | **必用** | 低 | 高（一票否决） |
| **VBench（视频）** | 视频专项打分 | 评测期用，生产期用退化版本 | 高（16 维全跑很贵） | 高（最全面） |
| **修脸匠（CodeFormer/GFPGAN/ADetailer）** | 局部脸坏 | 用 | 中（一次 inpainting） | 高（专业） |
| **修手匠（HandRefiner）** | 局部手坏 | 选用（看场景） | 中 | 中（极端姿势仍崩） |
| **二阶画家（SDXL Refiner）** | 整体精修 | SDXL 时代用，FLUX/SD3 时代不用 | 中（多 20% 步数） | 中 |
| **放大匠（Real-ESRGAN）** | 分辨率不够 | **必用** | 中（一次推理） | 高（成熟） |
| **Auto-retry** | 极端失败兜底 | 用（合规必用，质量选用） | N 倍 | 看运气 |
| **RLHF/DPO** | 把评委注入画家 | 模型层面用 | 训练一次性贵 | 高（提整体下限） |

## 失败模式清单（用户视角的统一总结）

- **"评分高但不像我"**——评委没看 identity，face similarity 是另一套体系（IPAdapter-Face、InstantID）。
- **"评分高但很丑"**——评委的训练数据偏离了你的审美（HPS v2 学的是标注员群体偏好）。
- **"修脸把我修成另一个人"**——CodeFormer w 太低、GFPGAN identity 漂移；这是 generative prior 修复的通病。
- **"出图很久"**——多张候选 + 评分 + 修脸 + 修手 + 超分串起来，一张图可能跑了 5-8 个模型。
- **"明明是我自己的特效，悄悄给我换了一张"**——auto-retry 触发了，前一张被合规评委拒了。
- **"视频前两秒帅后一秒崩"**——视频评委只盯首帧或抽样帧，时序问题被漏掉。
- **"风格越来越同质化"**——RLHF/DPO 的副作用，评委偏好被画家固化，多样性下降。
- **"魔法少女的杖手指穿模"**——HandRefiner 没接入或没识别到这个具体姿态。
- **"生成的图分辨率达标了但纹理糊"**——Real-ESRGAN 的过度平滑（去噪强度太高）。

## 产品设计师的落地洞察

1. **抖音特效的核心命题是"一次过关 vs 多张挑选"——这是产品策略，不是技术选择**。即梦/MJ 走"出 4 张让用户挑"是合理的——他们的用户是创作者，愿意挑。抖音特效里普通用户**根本不会看 4 张挑 1 张**——划走的概率比挑选的概率高得多。所以抖音必须把评委放在"出图前"或"出图中"，不是"出图后让用户挑"。**这条决定了整个质量评估架构的方向**。

2. **评委不能只设一个，必须"合规一票否决 + 美学/人类偏好综合 + identity 单独保"**。单一评委必然偏科——HPS v2 不看像不像你、面部相似度模型不看构图美不美、合规模型只看安全。**真实生产里的"质量分数"是几个独立模型加权出来的**，权重就是产品策略：抖音特效里 identity 权重应该比即梦更高（用户上传自拍，最在意像不像自己）。

3. **修脸/修手/超分应该默认开，不应该是用户开关**。用户根本不知道"AI 修脸"是什么、为什么要开。把这些做成"高质量模式开关"是把产品决策推给用户——不专业。**正解是产品后端默认全开，对用户表现为"出图就是好的"**；只在算力极紧张时（峰值流量）才悄悄关掉某些可选项（比如关 HandRefiner）。

4. **Auto-retry 的产品体验比技术更重要**。重抽进行中的等待时间不能让用户感知到"系统在挽救失败"——抖音用户对失败信号的容忍度极低。**正解是"乐观显示"**——先显示"AI 正在为你生成..."的稳定文案，重抽在后台悄悄发生；只有重抽 3 次还失败时才告诉用户"这张图我们没生成好，换个特效试试"。**用户感知到的不是"AI 错了"，是"AI 没找到好的素材"**——把锅甩给运气而不是产品。

5. **视频评估比图片评估贵 1-2 个数量级，要选"够用"的指标，不要追求 VBench 全跑**。生产环境视频质量评估应该用**最少够用的轻量指标组合**（首帧美学分 + 帧间像素差检测闪烁 + 主体一致性的简化版）；VBench 只在模型评测、版本对比、年度复盘时跑一次。**模型评测和实时质量把控是两件事，不能混为一谈**。

6. **RLHF/DPO 是"基础质量保障"层面的事，不能解决所有问题**。SD 3 / DALL-E 3 之所以"开箱就好看"是因为评委的偏好已经被注入画家。但**这种"开箱好看"是趋同的好看**——抖音特效要有"特效感"、"差异化风格"，过度依赖 RLHF 反而让所有特效画风变同质。**真正的差异化在 LoRA、训练数据、prompt 工程上做出来，而不是堆 reward model**。

## 还没搞清楚的部分（诚实标记）

- **抖音特效产品后端实际用的是哪些评委、什么权重**：这是入职后才能知道的内部信息。从公开信息推测大概率有"NSFW filter + face identity + 美学/HPS 综合"三档，但具体阈值和权重未知。
- **DALL-E 3 的具体 RLHF 配方和 reward model**：OpenAI 没发详细论文，"用了 RLHF"是从 system card 推测的。具体 reward model 的训练数据、PPO 超参、迭代次数都没披露。
- **MJ V6 的内部评分体系**：MJ 完全闭源，几乎没有任何技术披露。从用户使用观察，MJ 的"4 宫格"是 best-of-N 的最终态，但不知道背后还有没有 in-process pruning。
- **视频生产环境的轻量评估指标具体长什么样**：VBench 是学术评测工具，生产环境用什么"够用"的指标、怎么平衡延迟和准度，公开材料几乎为零。需要进入团队后看具体方案。
- **In-process pruning 的工程实现细节**：reward-guided search 是学术界正在研究的方向，没看到主流模型有公开的生产部署案例。Stability、Midjourney、字节内部是否在用、用到什么程度，都是黑盒。
- **Auto-retry 在不同流量水位下的策略变化**：高峰期重抽率被限制吗？怎么限制？是产品策略问题，公开案例未知。
