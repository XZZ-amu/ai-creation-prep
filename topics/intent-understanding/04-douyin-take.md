# 意图理解：抖音场景判断

> 基于 product-foundation 的产品定位和成本约束，对"意图理解"在抖音 AI 特效场景下的产品判断。每条判断都标注理由和对设计的具体影响。

---

## 1. 判断锚点

**产品定位**：抖音加号入口的 AI 特效 = 普通用户快速产出社交内容。不是创作工具，是表达放大器。

**成本约束**：平台承担所有生成成本。8 亿 DAU，1% 渗透 = 每天 800 万次生成。意图理解环节（LLM rewrite + text encoding）的成本虽然远低于生成本身，但乘以调用量后仍然是显著数字。

**核心张力**：低门槛 vs 有自己特色。意图理解环节的设计直接决定了用户表达自由度的上限和下限。

---

## 2. 核心判断

### 判断 1：抖音用户不写 prompt——意图理解的主输入应该是"选模板 + 传素材"

**判断**：抖音 AI 特效的意图理解不应依赖用户输入自然语言 prompt。主路径应该是"用户选择一个特效模板 + 上传自己的素材（人脸/照片/视频片段）"，系统自动完成意图拼装。

**理由**：
- 抖音用户的心智模型是"刷到一个好玩的特效 → 我也拍一个"，不是"我有一个创意想法 → 描述给 AI 听"。这和 Midjourney 用户从第一步就不同——Midjourney 用户的动机是实现脑中画面，抖音用户的动机是参与一个已被验证的玩法。
- 从 02-problems.md 的分析看，prompt 理解的 5 层信息损耗对专业用户都是挑战，普通用户根本不可能应对。
- 模板化路径把意图理解问题从"开放域理解"降维为"填槽"——系统只需要理解用户素材的关键属性（人脸位置、背景内容、主体轮廓），不需要理解一段自然语言的完整语义。

**对设计的影响**：
- 主交互流程里不应有文字输入框作为必填项
- 模板内的可调参数应该是选择题（风格 A/B/C）而非填空题
- 如果保留文字输入，定位是"锦上添花"的可选项，不是核心路径

---

### 判断 2：Prompt rewrite 应完全隐藏在后台，用户不应感知其存在

**判断**：如果系统内部使用 LLM 对意图做扩写/补全，这个过程不应暴露给用户。不显示"优化后的 prompt"，不提供"编辑优化结果"的入口。

**理由**：
- 可灵和即梦暴露 prompt rewrite 结果，是因为它们服务的是有创作意识的用户——这些用户关心"模型到底看到了什么"。抖音特效用户不关心，暴露反而增加认知负担。
- 从 product-foundation 的核心张力看，暴露 rewrite = 把调试复杂度推给用户 = 门槛上升 = 违背产品定位。
- 但这不代表 rewrite 本身不重要——它在后台的质量直接影响生成结果。应该投入做好，但投入方向是"让 rewrite 更准"而非"让用户能编辑 rewrite"。

**对设计的影响**：
- 不需要设计 prompt 优化的对比 UI
- 用户反馈"不满意"时，系统应自动调整策略（换种 rewrite 方式 / 换参数），而非让用户去修改 prompt
- 团队的 prompt rewrite 模型质量标准要更高——因为没有用户编辑兜底

---

### 判断 3：中文 prompt 理解是刚需，但优先级取决于主路径设计

**判断**：如果主路径是模板+素材（判断 1），中文 prompt 理解的优先级是中等而非最高。如果存在自由生成入口，中文理解能力则是 P0。

**理由**：
- 从 01-mechanism.md 的分析看，CLIP 对中文几乎无感，T5 系需要多语言版本（如 Wan 的 umT5-xxl），LLM 编码器天然支持中文。字节内部大概率有中文 LLM 优势可以直接利用。
- 但如果主路径是模板+素材，用户连 prompt 都不写，中文理解能力的优先级就被模板质量和素材理解能力压下去了。
- 真正需要中文理解的场景是：用户在模板基础上做微调时用中文描述（"把背景换成海边"、"让我看起来更年轻"）。这是中文理解能力的核心战场。

**对设计的影响**：
- 自由文字输入的场景里，禁止"先翻译成英文再理解"的临时方案——用户能感知翻译导致的语义漂移
- 模板内的微调描述应支持中文口语化表达（"酷一点"、"更梦幻"）
- 待验证：字节内部的文本编码器对中文口语化表达的理解深度

---

### 判断 4：多模态输入的优先级——图 > 视频 > 文字

**判断**：在意图表达方式上，图片输入应作为最高优先级的补充通道，视频片段次之，文字 prompt 最低。

**理由**：
- 抖音用户手机里最不缺的就是照片和视频。"用我的这张照片"是最自然、最零门槛的意图表达方式——比写任何文字都精确。
- 从 03-solutions.md 的分析看，参考图能携带的信息量远超文字：风格、色调、主体外观、构图偏好——一张图全包。
- IP-Adapter / CLIP-Vision 编码一张图的成本远低于 LLM 理解一段复杂文字的成本。
- 视频片段作为输入更适合"视频特效"场景：用户传一段自拍 → 系统理解动作/表情/场景 → 生成 AI 加工后的版本。这和端侧特效的使用流程完全一致。

**对设计的影响**：
- 素材上传入口应该比文字输入框更显眼、更容易触达
- 系统对用户上传图片/视频的理解能力（人脸检测、场景分类、主体分割）决定了特效匹配的精准度
- 参考图和模板之间的关系要想清楚：参考图是"我要这个风格"还是"把我放进去"——两种意图背后的技术路径完全不同（IP-Adapter vs img2img/inpainting）

---

### 判断 5：失败时不让用户改 prompt，而是产品自动兜底

**判断**：当生成结果不符合预期时，用户的出路不应该是"请修改你的描述"，而应该是"再来一次 / 换个方向 / 选局部修改"。

**理由**：
- 从 02-problems.md 的严重度排序看，"微调不收敛"是致命级体验问题。普通用户不具备通过改 prompt 来修正结果的能力——这是 prompt engineer 的技能，不是普通用户的技能。
- "再来一次"= 换 seed 重新生成，成本低、操作零门槛。
- "换个方向"= 给用户 3-4 个变体选择（类似 Midjourney 4 图选 1），把意图澄清从"描述能力"转化为"识别能力"。
- "选局部修改"= 用户指出"这里不对"，系统做局部 inpainting/重新生成。

**对设计的影响**：
- 生成结果页不应有"编辑 prompt"按钮作为主要操作
- 应提供"不满意？试试这些"的多变体推荐
- 应提供轻量级的"这里不对"圈选工具（长按 → 局部修改）
- 成本考量：多变体意味着多次生成。但可以用"一次生成多个低分辨率预览 → 用户选择后生成高分辨率版本"来控制成本（待验证可行性）

---

### 判断 6：意图理解的成本在总成本中占比小，但值得投入

**判断**：意图理解环节（LLM rewrite + text/image encoding）的算力成本约占单次生成总成本的 5-15%，不是成本瓶颈。但它对结果质量的影响是杠杆性的——理解对了一次出好结果 vs 理解错了用户反复重试，后者的成本是前者的 3-5 倍。

**理由**：
- 生成主体成本 = diffusion model 的去噪过程（20-50 步 UNet/DiT forward pass），这是大头。
- 意图理解成本 = 一次 LLM 调用（prompt rewrite）+ 一次 text encoding（T5/CLIP）+ 可能的一次 image encoding。这些都是单次 forward pass，计算量远小于去噪过程。
- 但意图理解的准确率直接影响用户的重试次数。如果首次生成命中率从 40% 提升到 70%，平均每个用户的生成次数从 2.5 次降到 1.4 次——等于在生成环节省了 44% 的成本。
- 因此，在意图理解环节多花 10% 的成本（用更强的 LLM、更好的 encoding），如果能减少一次无效生成，ROI 是正的。

**对设计的影响**：
- 不应为了省意图理解的成本而用弱模型——这是"花小钱省大钱"的环节
- 可以考虑分级策略：简单模板用轻量 encoding，复杂自由生成用重量级 LLM 理解
- 待验证：字节内部是否有针对特效场景蒸馏过的轻量意图理解模型

---

### 判断 7：模板的"意图理解"本质是结构化预设——绕过而非解决 prompt 问题

**判断**：模板化路径是对意图理解问题最高效的"产品侧解法"，但它本质是规避问题而非解决问题。需要清醒认识其天花板。

**理由**：
- 模板 = 预设好的 prompt + ControlNet 结构 + 风格 LoRA + 参数配置。用户选了模板，意图理解的难度从"理解用户想法"变成"把用户素材适配进预设框架"。这条路成本可控、体验可控。
- 但天花板在于：模板无法覆盖用户的长尾创意。当某个模板火了，所有人出来的内容千篇一律——这和 product-foundation 里"有自己特色"的目标冲突。
- 解法方向：模板 + 可变量。模板锁住主框架（风格、运镜、节奏），但用户的素材本身提供差异化（你的脸、你的环境、你的表情）。这种"受控变量"式的设计，让意图理解只需要处理"素材→模板填充"的映射问题，复杂度可控。

**对设计的影响**：
- 模板设计时要明确"哪些是固定的、哪些随用户变"——这决定了意图理解模块需要处理什么
- 应该有从"纯模板"到"半自由"到"全自由"的能力梯度，服务不同阶段的用户
- 模板的更新节奏要跟上热点——这是运营问题而非技术问题

---

### 判断 8：AI 在抖音特效中的角色是"执行层"，未来应向"品味翻译"层演进

**判断**：AI 的能力分六层光谱：执行（承担劳动）→ 品味翻译（越用越懂你）→ 标准提炼（群体选择变标准）→ 趋势显性化（信号翻译）→ 连接（推给对的人）→ 方向（帮你找到下一步）。抖音 AI 特效当前在"执行"层，下一步应该进入"品味翻译"层。

**理由**：
- "执行"层 = 用户选模板、AI 生成结果——这是当前所有 AI 创作产品的位置
- "品味翻译"层 = AI 从用户历史行为（选过什么风格、发过什么内容、点赞过什么）学习审美偏好，自动调整生成参数——越用越省力
- 抖音有天然优势做这一步：用户的消费数据（刷视频偏好）+ 创作数据（用过什么特效）= 全平台最丰富的"品味信号"
- 这是 Topic 4"个性化与偏好积累"的具体落点

**对设计的影响**：
- 短期：特效选择列表按用户历史偏好排序（简单个性化）
- 中期：生成参数（风格强度、色调偏好）从用户历史中学习，默认值因人而异
- 长期：AI 理解"你是什么风格的创作者"并主动推荐"你可能想试的下一个特效"

**状态**：待讨论——这是否在 AI 特效团队的 scope 内，还是推荐系统团队的事？

## 3. 与现有端侧特效的协同

**现状**：
- 端侧特效 = 人脸检测 + 人体分割 + 表情追踪 + 手势识别 + 实时渲染。成本为零，实时响应。
- AI 特效 = 云端 diffusion 模型生成新画面。每次花钱，有延迟。

**协同判断**：

### 端侧做"粗理解"，云端做"精生成"——这是最优成本路径

**具体逻辑**：
1. 用户打开 AI 特效时，端侧模型已经在实时工作——人脸位置、人体轮廓、背景分割、表情状态，这些信息不需要额外成本就能拿到。
2. 这些端侧信息直接作为意图理解的"输入补充"送给云端——云端不需要再花算力做人脸检测、场景理解等基础工作。
3. 云端模型只需要做"基于端侧理解结果 + 模板预设 → 生成新画面"这一步。

**等价于**：端侧负责 ControlNet 需要的结构信息提取（骨架图、深度图、分割图），云端直接拿现成结构做生成。省掉了"上传原图 → 云端做前处理提取结构"这一步的延迟和成本。

**进一步可能性**（待讨论）：
- 端侧能否做轻量 prompt rewrite？比如用户选了模板后，端侧小模型根据检测到的场景自动补全 prompt（"检测到室内暖光 → prompt 加 warm indoor lighting"），减少云端 LLM 的调用需求。
- 端侧追踪的人脸/身体关键点能否直接作为 ControlNet 的 conditioning input？如果能，意味着用户对着摄像头做动作就能实时引导 AI 生成——接近"AI 镜子"的体验。

**风险点**：
- 端侧模型和云端模型的"理解口径"要对齐——端侧识别的人脸区域和云端期望的人脸区域如果坐标系不一致，会导致错位。
- 端侧模型更新频率和云端模型更新频率不同步时，可能出现兼容性问题。

## 4. 待入职验证的问题

以下判断基于公开信息和产品逻辑推导，入职后需要用内部数据和系统实际情况来验证或修正：

### 关于现有系统
1. 当前抖音 AI 特效的意图理解链路具体是什么？用的是什么 text encoder？是否有 prompt rewrite？
2. 用户上传素材后，系统对素材做了哪些预处理？是否已经在用端侧检测结果？
3. 当前单次生成的成本拆分：意图理解环节占多少、生成环节占多少、后处理占多少？
4. 用户的平均生成次数是多少？首次命中率（用户对第一次结果满意的比例）是多少？

### 关于模型能力
5. 字节内部的 text encoder 对中文口语化表达的理解边界在哪？（"酷一点"、"仙气飘飘"、"暗黑风"这类表达能理解到什么程度？）
6. 是否有针对特效场景蒸馏过的轻量意图理解模型？和通用 LLM 比性能差多少？
7. 多模态理解（图+文混合输入）的实际支持程度？是已上线还是 roadmap？

### 关于成本和规模
8. 当前 AI 特效的日均生成量是多少？生成成功率（不报错/不超时）是多少？
9. 如果每次生成增加一次轻量 LLM 调用（做 prompt rewrite），总成本增加的百分比是多少？
10. "生成多个低分辨率预览 → 用户选后出高分辨率"这条路径在视频场景下的延迟和成本是否可接受？

### 关于用户行为
11. 用户对"不满意"的主要反应是什么？重试？退出？还是尝试修改？
12. 使用自由文字输入的用户占比多少？他们写的文字通常多长、什么内容？
13. 模板的使用数据：头部模板集中度多高？用户尝试新模板的意愿如何？
