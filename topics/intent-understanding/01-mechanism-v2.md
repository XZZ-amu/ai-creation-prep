# 意图理解：用户输入是怎么变成生成结果的（v2）

> v2 改写说明：v1 是技术密度档（337 行，事实扎实），但读完容易记不住。v2 用一个贯穿案例 + 一套画室比喻 + 用户视角的失败现象，把同样的事实重新组织一遍。事实层和 v1 一致，事实出处见 v1 末尾的来源列表。

---

## 一句话总结

**用户输入变成生成结果，不是一步翻译，是一个画室的协作流水线。**
里面有翻译官、画家、草稿世界、骨架图、风格手册——五个角色，五道关。
**每一道关都会丢信息**，所以同样一句 prompt，不同模型、不同环节出问题，最终结果会差很多。

---

## 贯穿案例

整篇文档围绕这一个 prompt 走完全程：

> **"一只戴红色围巾的橘猫坐在咖啡馆窗边看雪"**

每一个机制讲完，都会回到这只猫——这一步对它做了什么、可能在它身上出什么错。

---

## 画室五人组（核心比喻体系）

把整条生成链路想象成一个画室，里面有五个角色协作完成一张画：

| 角色 | 对应概念 | 它在干什么 |
|---|---|---|
| **翻译官** | Text encoder（CLIP / T5 / MLLM） | 把人类语言翻译成画家能懂的"语义坐标" |
| **画家** | UNet / DiT 主网络 | 真正动笔的人，从一团乱涂开始反复修改 |
| **草稿世界** | Latent space（潜空间） | 画家工作的地方——一个被压缩过的草稿空间，不是真实画布 |
| **骨架图** | ControlNet | 一张递给画家的硬性约束图——形状必须按这个来 |
| **风格手册** | LoRA / IP-Adapter | 一本翻给画家看的参考册——按这个味道画 |

**每个比喻都不完全准**。所以下面每个角色单独讲时，会附一行"这个比喻不准在哪"。

---

## 第 1 关：翻译官（Text Encoder）

### 干什么

用户写的是 "一只戴红色围巾的橘猫坐在咖啡馆窗边看雪"——这是人话，画家听不懂。
翻译官把这句话切成一个个 token，每个 token 翻成一串数字（向量），整句话变成一个矩阵。
这个矩阵才是画家能读的"参考资料"。

### 比喻不准在哪

**翻译官还会顺手做摘要**。Prompt 越长，越次要的细节会被淡化、被吞掉。SD 1.5 的翻译官（CLIP）一次只能装 77 个 token，超过的部分**直接被剪掉**——你写的"咖啡馆窗边的雪景，远处有路灯，桌上有一杯拿铁"，可能从"拿铁"开始就被剪了，画家根本没收到。

### 不同模型用的翻译官不一样

| 模型 | 翻译官 | 特点 |
|---|---|---|
| SD 1.5 | CLIP（一个） | 77 token 上限，超长 prompt 会被截断 |
| SDXL | CLIP × 2 | 一个抓概念，一个抓细节 |
| SD 3 / FLUX | CLIP + T5 | T5 让模型能正确写出图里的英文字 |
| HunyuanVideo（腾讯） | 多模态 LLM（MLLM） | 用大模型当翻译官，复杂指令理解更强 |
| Wan 2.1（阿里） | umT5（多语言） | 中文 prompt 优势 |

**为什么字节、腾讯、阿里都在换 LLM 当翻译官？** 因为传统 CLIP/T5 是几年前的模型，对复杂指令、多步推理力不从心。LLM 翻译能力强，但要补一些组件来弥补它的"单向注意力"短板。

### 回到我们的橘猫

- "橘猫"和"红围巾"翻译官能听懂——它见过大量类似图文对。
- "戴红围巾的"这种**修饰关系**——CLIP 经常分不清"红围巾戴在猫身上"还是"红围巾在猫旁边"。
- "在咖啡馆窗边"——CLIP 对**空间关系**（在...旁边、在...里）几乎无感。换 T5/MLLM 会好很多。
- "看雪"——这是一个动作 + 视线方向，CLIP 能记住"雪"，但"看雪的姿态"很难被这一关传下去。

### 用户视角的失败现象

**"我说红围巾它给我红猫"**——翻译官没把"红"和"围巾"绑住，"红"这个属性飘到了主体上。
**"我写了一长串细节，它只画了前半句"**——长 prompt 后半段被翻译官截断或淡化。
**"我说在左边，它给我在右边"**——翻译官（CLIP）对空间关系不敏感。

## 第 2 关：画家（UNet / DiT）和它的工作方式（Cross-Attention）

### 干什么

画家拿到翻译官给的矩阵之后开始动笔。但它不是一笔一笔画——它是**从一张全是噪点的纸开始，反复"去噪"**，每一轮把纸变得更清晰一点，最后变成成品。

每一轮去噪时，画家做一件事：把当前的"半成品图"切成很多小块，**每一块都去问翻译官的矩阵**："我这块该听哪个词的？"

这个"小块去查询哪个词最相关"的机制，就是 **cross-attention**。

### 比喻不准在哪

**这不是有意识的判断，是机械的对齐打分**。每个小块和每个 token 之间算一个相关度分数，分数高的 token 影响这一块的画法。所以"橘猫和黑猫"在画面里容易串色——它们的语义太近，同一块小区域被两个 token 同时强烈影响。

### Cross-attention 是 prompt 影响图片的唯一通道

记住这一句。**如果某个词在 cross-attention 里没被任何区域关注到，那它在最终图里就没声音。** "我说红苹果它没给我红"——大概率是"红"这个 token 没被苹果区域强烈关注到。

### 回到我们的橘猫

画到中期时，cross-attention 大概是这样工作的：
- 图片中央的小块**强烈关注 "橘猫"**——往猫的特征靠；
- 猫脖子区域的小块**强烈关注 "红围巾"**——但要靠"戴在...上"这个修饰关系正确建立；
- 背景区域**强烈关注 "咖啡馆"和"雪"**——往室内+窗外雪景靠；
- "看雪"这个动作要让猫头朝向窗户的方向——这要求"看"和"雪"两个词同时影响猫头部区域。

任何一处没对上，都会出错。

### 用户视角的失败现象

**"两个东西串色了"**——cross-attention 把相似 token 的影响混到了同一块区域。比如"红色围巾和蓝色帽子"，画出来变成红色帽子或蓝色围巾。
**"加一个新词，原来没问题的地方崩了"**——多了一个 token 抢注意力，原来的稳态被打破。
**"动作没出来，主体只是站着"**——动作词（"看雪"、"跳"）需要被空间区域稳定关注，往往拿不到。

## 第 3 关：草稿世界（Latent Space）

### 干什么

画家工作的地方不是真实画布。**它在一个被压缩过的草稿空间里画**——512×512 的图在草稿空间里只有 64×64。画完了再用一个解码器（VAE）还原成真实像素。

这是为什么 Stable Diffusion 能跑在普通显卡上的原因——它不是在 512×512 的画布上反复涂改，是在 64×64 的草稿上。

### 比喻不准在哪

**这不是真的"小图"——是一个数学空间，每个坐标点对应一张可能的图**。
更准的说法：草稿世界是一张**地图**，每个坐标代表一张可能的画。生成不是"动笔画图"，是"在地图上找路"——从一个随机起点（噪声）一步步走到"戴红围巾的橘猫坐在咖啡馆窗边看雪"那个坐标。

这个比喻有红利：
- **为什么换种子（seed）出来的图差别大？** 起点不一样，走出来的路就不一样；
- **为什么 prompt 越具体收敛越快？** 目标坐标更明确，路径更直接；
- **为什么有时候 prompt 没变图也变了？** 调度器（scheduler）和步数变了，等于换了走路方式。

### 回到我们的橘猫

我们写的 prompt 在草稿世界里对应一片"区域"——里面装着所有可能的"戴红围巾的橘猫坐在咖啡馆窗边看雪"。
不同 seed = 不同起点 = 走到这片区域里的不同坐标 = 不同的橘猫。
你抽中一只可爱的、抽中一只丑的、抽中一只多了一条腿的——都在这片区域里。

### 用户视角的失败现象

**"换了 seed 完全不一样的图"**——起点变了，路径变了。
**"prompt 没动但出来不一样"**——调度器、步数、CFG 任何一个变了都会改路径。
**"抽到歪了重抽一遍正常了"**——同一个目标区域里，有些坐标是好图，有些是坏图，靠运气。

## 第 4 关：骨架图（ControlNet）—— 硬约束

### 干什么

如果只靠 prompt，画家有很大自由度（"咖啡馆窗边"具体什么构图，画家自己定）。
但有时候你**需要精确的构图**——比如猫必须坐在画面右下角、头朝左 45 度、尾巴绕到身体前面。

ControlNet 就是给画家**额外递一张骨架图**：可以是线稿（canny）、可以是姿态骨架（pose）、可以是深度图（depth）、可以是语义分割图。画家被强制按这个骨架来画。

### 机制（一句话版）

ControlNet 是一个**额外训练的小网络**，它读骨架图、处理后把"形状信号"加进画家的工作中。画家的本职（听 prompt、动笔画）不变，但每一笔都被骨架图约束住。

### 比喻不准在哪

这个比喻基本是准的——ControlNet 真的就是给画家一张额外的形状参考图。

唯一要补的是：**强度可调**。`controlnet_conditioning_scale` 参数——0.5 是软约束（画家可以稍微违背骨架），1.0 是强约束，1.0+ 会把图绑死出现畸变。

### 回到我们的橘猫

- 用户给一张猫坐姿的姿态骨架图 + prompt "戴红围巾的橘猫坐在咖啡馆窗边看雪"；
- ControlNet 强制猫的姿态按骨架来，prompt 决定外观、围巾、背景。
- 强度 1.0 时姿态严格按骨架；强度 0.5 时画家可以把姿态调得更自然。

### 用户视角的失败现象

**"骨架太强，整张图被绑死，看起来很僵硬"**——`scale` 调高了。
**"骨架不够强，构图没听话"**——`scale` 调低了。
**"骨架对了，但风格和参考图差很远"**——ControlNet 只管形状，不管风格，需要再加一个 IP-Adapter。

---

## 第 5 关：风格手册（LoRA / IP-Adapter）—— 软约束

风格手册是个统称——里面其实有两种不同的"参考资料"，机制不一样。

### 5a. IP-Adapter：递一本图册给画家看

**干什么**：用户给一张参考图（"我要这个画风/这个氛围/这个人脸"），IP-Adapter 把这张图压缩成图像向量，画家在画的时候**除了听翻译官（文字），还会再听一路图像参考**。

**和 ControlNet 的核心差别**：
- ControlNet 控制**结构**（形状、轮廓、姿态）；
- IP-Adapter 控制**风格/特征**（画风、材质、人脸特征、整体氛围）；
- 两个可以同时用：ControlNet 锁结构，IP-Adapter 锁风格。

**回到橘猫**：用户给一张"宫崎骏风格的水彩咖啡馆"参考图 + prompt + ControlNet 姿态骨架。
- ControlNet 锁猫的姿态；
- IP-Adapter 锁咖啡馆的水彩画风；
- prompt 提供具体内容。

### 5b. LoRA：教画家一项新本事

**干什么**：LoRA 不是一张参考图，是一个**几 MB 的小补丁**——专门让画家"额外学会一个东西"。
比如训练一个"我家橘猫"的 LoRA，触发词是 "myorangecat"。推理时 prompt 里带上 "myorangecat"，画家就知道画的是这只特定的猫，而不是泛泛的橘猫。

### 比喻不准在哪

**"风格手册"对动漫/水墨风格 LoRA 是准的。但 LoRA 也能训特定人物、特定物体（不只是风格）**——这时候它更像"教画家记住一个新东西"。

更精确的比喻：**LoRA 是给画家发一本随身手册**——里面可能装着一种风格、一个人物、一种特殊物体。挂上手册就记得，摘下就忘。

### 三种"风格/参考资料"机制对照

| 机制 | 改的是 | 输入 | 解决什么 |
|---|---|---|---|
| **LoRA** | 画家的本事（小补丁直接装在画家脑子里） | 仅文字 prompt（带触发词） | 教画家不认识的东西 |
| **ControlNet** | 给画家加一张骨架图 | 文字 + 结构图 | 锁住构图/形状 |
| **IP-Adapter** | 给画家加一本图册 | 文字 + 参考图 | 锁住风格/主体外观 |

三个可以叠加。"角色一致性 + 特定姿势 + 特定风格"经常一次性用全。

## 5 个信息丢失点（用户视角版）

整条链路有 5 道关，**每一关都可能丢信息**。下面把每一关常见的失败现象列出来——以后看到生成失败的案例，能立刻定位是哪一关出问题：

### 丢失点 ① 翻译关：用户写的，没全部翻过去

**现象**：
- 写了一长串，只画了前半部分（长 prompt 被截断）
- 中文 prompt 比英文 prompt 出图差很多（翻译官训练数据偏英文）
- "在...左边"、"在...后面"这种空间关系没被理解（CLIP 对空间不敏感）

**根因**：翻译官能装的 token 有限、训练数据有偏、对某些语义类型先天弱。

### 丢失点 ② 画家关：词被翻过去了，但没落到对的地方

**现象**：
- 红色围巾画成红色帽子（属性飘到了别的物体上）
- 两只猫一橘一黑结果都画成橘的（相似 token 串色）
- 加一个新词，原来对的地方就崩了（注意力被重新分配）

**根因**：cross-attention 是机械打分，相似词会抢同一片区域的注意力。

### 丢失点 ③ 草稿关：路走偏了

**现象**：
- 同一个 prompt + 同一个 seed 应该出同一张图，但换 seed 完全变样
- prompt 没改但 sampler/steps/CFG 改了，图也变了
- 抽 10 次，9 次正常，1 次出怪图

**根因**：草稿世界是一张地图，从随机起点找路到目标，路径有随机性。

### 丢失点 ④ CFG 关：信号被错误放大或抑制

**现象**：
- CFG 太高：画风过饱和、人物畸变（特征被过度放大）
- CFG 太低：prompt 几乎没作用，模型自由发挥（信号被抑制）

**根因**：CFG 是一个调节"画家多听 prompt 还是多自由发挥"的旋钮，调过头两边都坏。

### 丢失点 ⑤ 视频专属关：prompt rewrite 改坏了

**现象**：
- 用户精确的描述被默默扩写成另一个意思
- "猫慢慢转头" 被 rewrite 成 "猫快速转头看向镜头"
- 用户觉得"我明明写得很清楚"，但模型看到的根本不是这个

**根因**：Sora、可灵、HunyuanVideo 都会用 LLM 把短 prompt 扩写成详细描述。**改写后的 prompt 才是模型真正看到的**——这是一道隐形黑盒。

## 视频生成的特殊性（极简版）

视频生成本质和图片一样——**画家从噪声去噪到结果**。但多了"时间"这个维度，所以有几个升级：

1. **画家切的小块从 patch 变成 spacetime patch**——一个小块同时覆盖一小块画面 × 一小段时间。"猫**跳起来**"的"跳"必须被时间维度上多个连续小块共同关注，否则猫只是站着。

2. **训练时和推理时的 prompt 要一致**——训练时模型见到的是详细的 200 字描述（用 video-to-text 模型给训练视频写 caption），推理时用户给的是 5 字短句。所以 Sora、可灵、HunyuanVideo 都会**默默把用户的短 prompt 扩写**（用 LLM）。这是"prompt rewrite"，是黑盒。

3. **时序一致性主要是从训练数据里学的，不是从文字里拿的**——文字只能给"语义提示"（"一个人在跑步"），具体跑步的物理、布料、运镜，模型必须**在大量真实视频里见过类似场景**才能复现。OpenAI 自己承认 Sora 在"复杂物理"、"因果关系"、"分清左右"上有明显短板——这都是时间维度上的意图理解还没解决的问题。

## 产品设计师的几个落地洞察

把上面的内容压缩成几条可以拿去做产品决策的判断：

1. **"模型理解"是分层的，前端可干预的层比想象多**。翻译官只是其中一层；prompt rewrite、负面 prompt、CFG、ControlNet 强度、IP-Adapter scale 都是产品可以选择"暴露给用户 / 自动调"的层级。每暴露一层，自由度上升、心智负担也上升。

2. **"意图理解"和"创作能力"是两件事**。意图理解强（"它听懂了我说啥"）≠ 创作能力强（"它画得好看"）。Midjourney 的画质美学是通过特定训练数据 + RLHF 调出来的，它的意图理解未必比开源模型强。

3. **prompt rewrite 是个隐藏的产品决策**。改写后的 prompt 才是模型真正"理解"的东西。要不要让用户看见、要不要让用户编辑改写后的版本——是真实的设计岔路口。HunyuanVideo 直接开源了 PromptRewrite 模型，等于把这件事透明化。

4. **多输入混合时，"听谁的"有优先级**。骨架图（ControlNet）优先级最高（结构是硬约束），风格手册（IP-Adapter）次之（风格是软约束），翻译官给的（prompt）最弱（语义经常被结构和风格压过去）。设计混合输入界面时，要让用户能感知和调整这个优先级。

5. **失败诊断要分层问**。不是"为什么不行"，而是"哪一关不行"——是翻译官没听懂？还是画家没把词路由到对的地方？还是骨架图绑太死？还是 prompt rewrite 改坏了？产品如果能给出这种分层提示，能极大降低用户的挫败感。

6. **视频意图理解的下一步是 LLM 化**。HunyuanVideo 已经在做了：用 MLLM 当翻译官、用专门的 LLM 做 prompt rewrite。这个方向意味着未来用户可以用更"对话式"的方式描述视频（"先慢慢推近，然后切到反应镜头"），而不是堆视觉关键词。

    │  ① 翻译官 (CLIP/T5)  │  ← 长 prompt 截断、空间关系丢失
    │  把人话翻成数字矩阵     │
    └──────────┬───────────┘
              │ 文本向量矩阵
              ▼
    ┌──────────────────────┐    ┌─────────────────┐
    │  ② 画家 (UNet/DiT)    │←───┤  ④ 骨架图 (CtrlNet) │
    │  在草稿世界③里反复去噪  │←───┤  ⑤ 风格手册 (LoRA  │
    │  通过 cross-attention   │    │     /IP-Adapter)    │
    │  让每个小块"听"prompt  │    └─────────────────┘

              │ 草稿世界里的最终坐标

        最终图像/视频

## 还没搞清楚的部分（诚实标记）

- **可灵（KLING）的具体架构和文本编码器**：官网和论文公开信息都很少，无法确认它用的是 T5 系还是 LLM 系编码器。如果做相关产品决策需要更准确的对比信息，要从快手的工程师博客或访谈里找。
- **字节内部模型（豆包视觉、即梦）的意图理解链路**：这是入职后才能知道的。但从公开论文（如 PixArt-α、Seedream 系列）看，字节方向偏向于用更大的 LLM 当文本编码器 + 大量 in-house 数据 recaptioning，整体路线和 HunyuanVideo 接近。
- **Cross-attention 在 DiT 架构（替换 UNet）下的具体差异**：DiT 把 attention 模式从"局部 + cross"改成"全局 self-attention 跨所有 token"，这对意图理解的影响（比如长 prompt 是否更稳定）目前公开材料给的是定性结论，缺少对比实验。
- **多模态输入的"优先级"在不同模型里的具体表现**：本文给的是经验直觉，没有量化对比。做产品决策时如果要明确"在 A 模型里 ControlNet 多强能压过 prompt"，需要自己跑实验。
