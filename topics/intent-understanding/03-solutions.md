# 意图理解：各家解法对比

> 各家产品在"让模型更准确地理解用户意图"这件事上走了非常不同的路——有的在模型底层死磕，有的在交互层绕过，有的让 LLM 当翻译。没有银弹，每条路都有明确的 tradeoff。

---

## 一句话总结

意图理解的解法可以分为四层：**prompt 层**（帮用户把话说对）、**模型层**（让模型听力更强）、**交互层**（绕过自然语言的模糊性）、**多模态层**（让用户不只用文字表达）。各家的差异不在于用了哪一层，而在于**哪层做主、哪层做辅、用户感知到的是哪层**。

---

## 一、Prompt 增强类解法

核心逻辑：用户说不清楚，那就帮用户说清楚。

### 1.1 LLM Prompt Rewrite（自动扩写）

**谁在用**：几乎所有主流产品（2024 年后的标配）
- Sora：GPT-4V 自动扩写
- 可灵：内部 LLM 自动优化（具体模型未公开）
- 即梦：豆包大模型扩写
- HunyuanVideo：专门训练的 PromptRewrite 模型
- Wan 2.1：Qwen 扩写
- Midjourney：内部 LLM 处理（V6 起）
- Pika：内部模型扩写

**底层逻辑**：训练数据的 caption 是精细的长描述（200-500 字），但用户输入通常是短句（5-20 字）。两者之间的分布差距（distribution gap）会让模型困惑。Prompt rewrite 用一个 LLM 把用户短句"翻译"成模型训练时见过的那种长描述，缩小 gap。

**优点**：
- 不改模型，纯前端方案，部署灵活
- 能补充用户没写但"应该有"的细节（光照、视角、氛围）
- 让低质量 prompt 也能出合格结果
- 可以把中文 prompt 翻译成英文再扩写（解决中文弱势问题）

**缺点**：
- 信息黑盒：用户不知道模型看到的和自己写的不一样
- 过度发挥：LLM 加的细节可能偏离用户意图
- 微调困难：用户想改一处时，整个扩写可能重来
- 丢约束：用户的否定/风格约束可能在扩写中被吞掉
- 额外推理成本：多一次 LLM 调用

**各家差异**：

| 产品 | Rewrite 策略 | 用户可见性 | 用户可编辑 |
|---|---|---|---|
| 可灵 | 默认开启，有"prompt 优化"开关 | 可以查看优化后结果 | 可手动编辑 |
| 即梦 | 默认开启 | 展示优化后版本 | 可编辑 |
| Midjourney | 默默执行，无开关 | 不可见 | 不可编辑 |
| Sora | 默认开启 | 部分可见 | 有限编辑 |
| Pika | 默认开启 | 不透明 | 不可编辑 |

### 1.2 Negative Prompt（反向约束）

**谁在用**：SD 系生态（Civitai、ComfyUI）、可灵、即梦、通义万相

**底层逻辑**：既然模型不理解"不要 X"，那就用 CFG 的数学机制做排斥——把"不想要的东西"写在另一组 prompt 里，在 guidance 计算时从生成方向中减去这些特征的影响。

**优点**：
- 技术上优雅——不是 hack，是 CFG 机制的自然延伸
- 对排除特定视觉元素非常有效（排除模糊、变形、水印等）
- 用户可以精细控制"不要什么"

**缺点**：
- 增加心智负担——用户需要学习"negative prompt 应该写什么"
- 不是所有产品都暴露这个能力（Midjourney 不支持传统 negative prompt）
- 过度使用会让生成结果变得僵硬/不自然
- 新架构（FLUX）的 guidance distillation 让传统 negative prompt 的接口消失

**2025-2026 趋势**：随着 FLUX 类模型的普及，传统 negative prompt 正在被弱化。取而代之的是模型本身更强的否定理解力 + 产品层面的"风格排除"选项。

---

### 1.3 Structured Prompt / 分区 Prompt

**谁在用**：ComfyUI 社区、Midjourney（--style/--sref 参数）、部分 API 接口

**底层逻辑**：把一个 prompt 拆成多个语义区块（主体描述 / 场景描述 / 风格描述 / 技术参数），每个区块独立编码后以不同权重注入。避免所有信息在一个 token 序列里互相干扰。

**优点**：
- 减少 attention 争夺——关键词不会被场景描述稀释
- 让用户建立"分区控制"的心智模型
- 对属性绑定问题有一定缓解

**缺点**：
- 增加用户操作复杂度
- 需要模型/pipeline 层支持分区编码
- 不是所有模型都兼容

---

## 二、模型架构改进类解法

核心逻辑：不靠前端补丁，从根上让模型"听力"更强。

### 2.1 更强的文本编码器

**谁在用**：
- FLUX / SD3：CLIP-L + T5-XXL（5B 参数）
- HunyuanVideo：多模态 LLM（MLLM）
- Wan 2.1：umT5-xxl（多语言）
- DALL-E 3：T5 + GPT-4 caption 训练
- 字节 Seedream：推测为大参数 LLM 编码器（待验证）

**底层逻辑**：CLIP 是 2021 年的模型，对复杂语义理解有先天限制。换用更大、更新、训练数据更多的语言模型（T5-XXL、MLLM）来做编码器，直接提升"翻译官"的能力上限。

**优点**：
- 根本性解决——不是绕过问题，是解决问题
- 空间关系、数量、属性绑定、长 prompt 都有系统性改善
- 一次升级，所有下游问题同步缓解

**缺点**：
- 推理成本上升（T5-XXL 本身就有 5B 参数，跑一次不便宜）
- 模型文件变大，端侧部署困难
- 换编码器意味着要重新训练或微调主网络，工程成本高
- MLLM 的推理速度慢于 CLIP/T5

**成本参考**：T5-XXL 的推理成本约是 CLIP 的 3-5 倍（待验证：取决于具体实现和 batch size）。对抖音这种日活 8 亿的平台，编码器成本乘以调用量是一个显著数字。

---

### 2.2 DiT 架构替换 UNet

**谁在用**：FLUX、SD3、Sora、可灵 1.5+、HunyuanVideo、Wan 2.1

**底层逻辑**：传统 UNet 的 cross-attention 是"文本查询图像"的单向交互。DiT（Diffusion Transformer）把文本 token 和图像 token 放在同一个序列里做全局 self-attention——所有 token 都能互相看到、互相影响。

**对意图理解的具体改善**：
- 长 prompt 更稳定：不再有 77 token 截断，T5+DiT 支持 512 token 输入
- 属性绑定更准：全局 attention 让"红色→围巾"的绑定关系更容易建立
- 多实体分离更好：不同区域的图像 token 能通过全局 attention "知道"彼此的存在，减少概念混合
- 空间关系稍有改善：但不如 ControlNet 显式控制可靠

**优点**：
- 架构级改善，一揽子解决多个问题
- 和更强的编码器配合效果翻倍
- Scaling law 更友好（堆 transformer 层数就能提升性能）

**缺点**：
- 推理更慢（全局 attention 的计算量是 O(n^2)，token 数多时成本高）
- 需要大量训练数据和算力才能训出一个好的 DiT 模型
- 生态兼容性差——之前围绕 UNet 建的所有工具（ControlNet、LoRA）需要重新适配

---

### 2.3 Training Caption 优化（Recaptioning）

**谁在用**：DALL-E 3（GPT-4V recaption）、Sora、HunyuanVideo、Wan、Stable Diffusion 3

**底层逻辑**：模型的意图理解上限由训练数据的 caption 质量决定。互联网图片的 alt text 和视频标题都是垃圾。用一个强大的 vision-language 模型给所有训练数据重写高质量 caption，让模型在训练时就见到"精确描述→对应画面"的样本。

**优点**：
- 从数据源头解决问题——模型见过什么样的描述方式，就能理解什么样的描述
- 和所有其他改进正交，可以叠加
- 一次 recaption 永久受益

**缺点**：
- 成本高昂：给几亿张图/几千万条视频做 recaption 的 LLM 调用成本极高
- 质量依赖 recaption 模型本身的能力——如果 recaption 模型理解错了，错误就被永久烙进训练数据
- 只能提升"模型见过的概念"的理解力，对全新概念无效

---

### 2.4 Attention 机制修复

**谁在用**：学术界 → 逐步进入产品

- **Attend-and-Excite**（2023）：强制让每个关键名词在 attention map 上至少有一个高激活区域，防止"某个词被忽视"
- **Structured Diffusion**（2023）：把 prompt 做句法解析，基于句法树分配 attention
- **SynGen**（2024）：用 NLP 依存句法来引导属性绑定
- **Regional Prompting**：不同画面区域绑定不同 prompt 片段

**底层逻辑**：不改编码器和主网络架构，只在推理时对 attention 分布施加额外约束——确保"每个词都被听到"、"属性绑定到正确的名词"。

**优点**：
- 不需要重新训练模型
- 针对特定问题（属性绑定、实体缺失）效果明显
- 可以作为推理时的可选插件

**缺点**：
- 增加推理时间（每步都要额外计算 attention 修正）
- 只是补丁，不是根治——新的 DiT 架构本身就做了很多同类改善
- 和不同模型的兼容性参差不齐

## 三、UI/交互层解法

核心逻辑：如果自然语言天然模糊，那就用交互设计减少模糊性。

### 3.1 参数面板 + 预设选项

**谁在用**：Midjourney（--ar, --style, --chaos 等参数）、通义万相（风格/比例选择器）、即梦（风格预设+参数滑块）、可灵（风格选择 + 画面比例）

**底层逻辑**：把"常见但难以用自然语言精确表达的意图"抽象为选项/滑块。用户不需要写"cinematic lighting, 16:9 aspect ratio, photorealistic"——只需要点一个"电影感"按钮。

**优点**：
- 大幅降低 prompt engineering 门槛
- 消除歧义——选项是确定性的，不像自然语言有多种理解
- 让产品能控制生成质量（预设选项背后可以绑定优化过的 prompt 模板）

**缺点**：
- 自由度受限——用户被圈定在预设选项内
- 选项越多越复杂，又回到"高门槛"
- 需要持续运营——新风格、新模型都要更新预设

**各家风格**：

| 产品 | 交互形式 | 暴露参数程度 |
|---|---|---|
| Midjourney | 命令行参数（--ar, --style, --s, --c） | 中高（面向有学习意愿的用户） |
| 通义万相 | 图形面板 + 滑块 | 中（可选展开"高级设置"） |
| 即梦 | 风格卡片 + 参考图上传 | 中 |
| 可灵 | 简洁面板 + 风格选择 | 中低 |
| Pika | 极简输入框 + 少量选项 | 低 |

---

### 3.2 意图澄清对话（Conversational Clarification）

**谁在用**：DALL-E 3（ChatGPT 集成）、Adobe Firefly（部分场景）、通义万相（2024-2025 版本）

**底层逻辑**：不让用户一次性把所有意图塞进一句话，而是通过多轮对话逐步明确。LLM 先理解用户的大方向，然后主动追问不明确的部分（"你想要什么风格？"、"人物是正面还是侧面？"），最后再拼装成精确 prompt 给模型。

**优点**：
- 最符合"自然交互"直觉——像和一个设计师沟通
- 能处理极短的初始输入（"帮我画个头像"）
- 歧义消解在生成前完成，避免无效生成浪费算力

**缺点**：
- 慢——多轮对话增加用户从输入到看到结果的时间
- LLM 对话本身有成本
- 不适合"快速迭代"场景——用户想快速试很多变体时，对话模式太重
- 追问太多会让用户烦（"我就想随便看看，别问我那么多"）

**关键设计取舍**：什么时候追问 vs 什么时候直接生成？追问太少→意图不准→浪费生成算力；追问太多→用户流失→互动过重。

---

### 3.3 生成后选择（Generate-then-Choose）

**谁在用**：Midjourney（4 图选 1）、DALL-E（多图选择）、即梦（多结果展示）、可灵（多结果+变体）

**底层逻辑**：承认模型对意图的理解有不确定性，不试图一次给出"正确答案"，而是同时生成多个变体，让用户从中选择。用户的选择行为本身就是意图的澄清——"我选了 C 说明 C 最接近我想要的"。

**优点**：
- 把"意图理解"的负担从模型转移到用户的"识别能力"
- 用户选择比用户描述容易得多
- 每次选择都能收集 preference 数据用于模型改进
- 减少单次生成的压力——不需要一次"完美"

**缺点**：
- 算力成本乘 N（生成 4 张的成本是 1 张的 4 倍）
- 用户可能"都不满意"，需要有明确的迭代路径
- 选择界面设计影响体验——太多选项让用户决策困难

**Midjourney 的精妙设计**：4 张不是随机 4 张——它们在多样性上做了控制，覆盖不同的构图/色调/细节方向，让用户的选择能提供最大信息量。（待验证：具体算法未公开）

---

### 3.4 局部编辑 + Inpainting（事后修正）

**谁在用**：即梦、通义万相、Adobe Firefly、Midjourney（Vary Region）、DALL-E（编辑模式）

**底层逻辑**：生成后对不满意的局部进行针对性修改——用户框选区域+新描述，只重新生成那个区域。这不是"提升意图理解"，而是"允许意图理解不完美"——给用户一个低成本的修正路径。

**优点**：
- 把"一次性理解完美"的压力转变为"逐步修正"
- 用户心智模型简单：哪里不对改哪里
- 保持了整体生成的一致性（只改局部）

**缺点**：
- 局部和整体的衔接可能不自然（边缘伪影、光照不一致）
- 多次 inpainting 后画面质量可能累积退化
- 对视频场景更难实现（时序一致性问题）

---

## 四、多模态输入类解法

核心逻辑：文字天然不精确，那就让用户用更直观的方式表达意图。

### 4.1 参考图/风格图（Image-to-Image / Style Reference）

**谁在用**：所有主流产品
- Midjourney：--sref（风格参考）、--cref（角色参考）
- 可灵：图生图、风格参考
- 即梦：参考图+prompt
- 通义万相：图文混合输入

**底层逻辑**：一张参考图携带的信息量远大于一句话。"我想要这个感觉"配一张图，比用 50 个词描述风格精确得多。IP-Adapter / CLIP-Vision 把参考图编码为向量，和文本向量一起注入生成过程。

**优点**：
- 用户不需要懂术语——"就像这个"是最自然的表达
- 风格/氛围等抽象概念难以用文字精确表达，图片是更好的载体
- 减少 prompt engineering 依赖

**缺点**：
- 用户可能没有合适的参考图
- "参考到什么程度"（风格 vs 构图 vs 内容）难以精细控制
- 参考图和文字 prompt 冲突时优先级不透明

**Midjourney 的创新**：`--sref` 和 `--cref` 是两个独立参数——前者只提取风格，后者只提取角色特征。这种"参考维度的拆分"让用户能更精确地表达"我要的是这张图的哪个方面"。

---

### 4.2 结构控制图（ControlNet 类）

**谁在用**：ComfyUI 生态、即梦（线稿/深度图）、通义万相（结构引导）、可灵（姿态/构图控制）

**底层逻辑**：文字描述不了精确构图（"主体在画面偏左上 1/3 处，视线方向朝右下"），但一张线稿或深度图可以。ControlNet 把结构信息作为硬约束注入生成过程。

**优点**：
- 解决了文字无法表达的空间精确性问题
- 对有绘画基础的用户非常友好——画个草稿就能精确控制构图
- 和 prompt 正交互补：结构归结构，内容归内容

**缺点**：
- 门槛高——普通用户不会画线稿、不懂深度图
- 需要额外的前处理（把照片转成线稿/骨架图）
- 过强的结构约束会限制模型的创造力

**产品层的解法**：让用户拍照/上传照片，产品自动提取结构信息（边缘检测、人体骨架识别、深度估计），而不是让用户手动画。即梦和通义万相走的是这条路。

---

### 4.3 多模态 LLM 作为意图理解前端

**谁在用**：DALL-E 3（GPT-4V）、Gemini 系列（Imagen 3）、通义万相（Qwen-VL）

**底层逻辑**：不再让用户直接和生成模型对话，而是在前面放一个多模态 LLM。这个 LLM 能同时理解文字、图片、甚至语音输入，把用户的多模态意图统一翻译成生成模型能理解的精确描述。

**优点**：
- 用户可以混合使用文字+图片+指示来表达意图
- LLM 能做推理——"帮我做一个类似这个但更暗一些"这种相对描述可以被理解
- 为"对话式创作"打下基础

**缺点**：
- 多一层 LLM = 多一层延迟 + 多一层成本
- LLM 的理解也可能出错，且错误更隐蔽（它很自信地理解错了）
- 用户对"LLM 真正理解了什么"没有透明度

---

## 对比表：各家核心策略差异

| 维度 | Midjourney | 可灵 | 即梦 | 通义万相 | Runway | Pika | DALL-E 3 |
|---|---|---|---|---|---|---|---|
| **主要依赖** | 模型审美 + 参数体系 | Prompt rewrite + 模型能力 | 多模态输入 + rewrite | LLM 前端 + 参数面板 | 模型能力 + 运镜术语 | 极简输入 + 自动优化 | GPT-4V 对话 + recaption 训练 |
| **Prompt rewrite** | 隐式执行 | 显式可见 | 显式可编辑 | 显式可见 | 隐式 | 隐式 | GPT-4V 承担 |
| **用户门槛** | 中（需要学参数） | 低-中 | 中（功能多） | 低-中 | 中（专业术语优势） | 极低 | 低（自然对话） |
| **意图精确度** | 中高（靠参数微调） | 中（靠 rewrite） | 中高（多模态补充） | 中高（LLM 澄清） | 中（运镜强，语义中等） | 中低（自动处理） | 高（多轮澄清） |
| **多模态支持** | sref/cref/图生图 | 图+结构+风格 | 图+线稿+深度+风格 | 图+文+可能语音 | 图+视频参考 | 图参考 | 图+文+对话 |
| **失败时的用户出路** | 换 seed/调参数/变体 | 编辑 prompt/重新生成 | 局部重绘/调参 | 继续对话/局部编辑 | 调参/re-roll | 重新生成 | 继续对话要求修改 |
| **成本倾向** | 中（4 图并行） | 中高（rewrite+生成） | 高（多模态处理链长） | 中高（LLM 前端） | 高（视频生成） | 低（短视频） | 高（GPT-4V+DALL-E） |

## 值得关注的新兴方向（2025-2026）

### 方向 A：意图理解和生成的一体化

传统做法是"编码器理解意图 → 生成器执行"两步分离。新趋势是让整个模型端到端地从输入到输出，不再有显式的"编码→生成"分界线。

**代表**：
- Transfusion（Meta，2024）：在一个 transformer 里同时做语言理解和图像生成
- Show-o（2024）：统一的自回归+离散 diffusion 模型
- 字节的探索方向（Seedream 系列推测）

**对意图理解的意义**：如果理解和生成是同一个模型在做，"翻译损耗"理论上降到最低。

---

### 方向 B：用户 Preference 学习

**代表**：Midjourney 的隐式偏好系统、DALL-E 3 的 RLHF

**逻辑**：不依赖用户一次性把意图说清楚，而是通过多次选择/评分学习用户的偏好模式。"这个用户总是喜欢暖色调、极简构图"——那么当用户输入模糊时，就往这个方向倾斜。

---

### 方向 C：实时预览 + 交互式引导

**代表**：Krea.ai（实时生成预览）、部分 ComfyUI 工作流

**逻辑**：用户边打字/边调整，实时看到预览图变化。这把意图表达从"一次性描述"变成"渐进式引导"——用户不需要一开始就想清楚，可以"看着调"。

**挑战**：需要极快的推理速度（SDXL Turbo / LCM-LoRA 1-4 步出图），牺牲一定质量换速度。

---

### 方向 D：视频意图的结构化表达

**代表**：Runway（运镜控制）、Pika（动作强度滑块）、可灵（运镜+主体运动分离）

**逻辑**：视频意图比图片复杂太多——不只是"画面里有什么"，还有"怎么动"、"镜头怎么走"、"节奏怎么变"。纯文字很难表达这些，需要结构化输入（时间线、关键帧标注、运镜参数）。

## 对抖音场景的启发（不做判断，只列思考方向）

以下方向值得在 04-douyin-take.md 中展开讨论：

1. **Prompt rewrite 的透明度取舍**：抖音用户群偏向"随手拍"心态，不太愿意学 prompt engineering。默默优化可能是更好的选择——但怎么处理"优化后偏离意图"的投诉？

2. **"选"比"说"更适合普通用户**：Midjourney 的 4 图选 1 模式让用户用识别力替代表达力。在抖音特效场景下，生成 4 个视频变体让用户选，可能比让用户写精确 prompt 更有效——但视频的算力成本是图片的 10-100 倍。

3. **参考图在短视频场景下的天然优势**：用户手机里有大量照片/视频，"用我的照片+这个特效"天然解决了意图表达问题。参考图可以替代 80% 的 prompt 功能。

4. **多模态 LLM 做意图前端的成本问题**：DALL-E 3 的 GPT-4V 方案效果最好，但成本也最高。在抖音的调用量级下，每次生成都走一次 GPT-4V 级 LLM 是否可承受？有没有更轻的替代方案（比如专门蒸馏的小模型）？

5. **"模板+填充" vs "自由生成"的路线选择**：大多数抖音用户的意图是"我想用那个很火的特效"——这时意图理解的问题被简化为"识别用户选了哪个模板"+"把用户的素材塞进模板"。这条路完全绕过了 prompt 理解问题，但牺牲了创作自由度。

6. **失败时的降级体验**：当意图理解失败时，不是显示"生成失败"，而是给出"也许你想要的是这个？"的推测。用"猜测+确认"替代"理解+执行"。

7. **实时预览在移动端的可行性**：如果能做到"用户滑动滤镜那样滑动 AI 效果"，意图理解的准确性压力会小很多——用户通过实时反馈"自己找到想要的"，而不是事先描述清楚。受限于移动端算力和延迟。
