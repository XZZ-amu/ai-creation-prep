# 意图理解：常见问题与失败模式

> 用户说的话和模型画出来的东西之间，至少隔了 5 层信息损耗。每一层都可能出错，而且错误会叠加。理解"哪一关出了什么问题"，是设计诊断反馈和降低用户挫败感的前提。

---

## 一句话总结

意图理解的失败本质上是**信号衰减问题**——用户的意图是一个高维复杂信号，经过分词、编码、attention 分配、去噪采样等环节后，每一步都在丢信息或引入歧义，最终用户看到的结果是所有损耗叠加后的产物。

---

## 第一关：翻译官失败（文本编码器层）

文本编码器是用户意图进入模型的第一道关卡。它的能力边界直接决定了"哪些意图根本不可能被理解"。

### 问题 1.1：空间关系丢失

**现象**：用户写"猫在桌子左边，花瓶在桌子右边"，结果猫和花瓶的位置随机，或者干脆反了。

**背后原因**：CLIP 是通过图文匹配训练的，它学到的是"猫+桌子+花瓶共现"这种关联，而不是"左/右/上方/后面"这种精确空间关系。CLIP 的训练目标是把整句话和整张图做相似度匹配，不是理解句法结构。

**典型触发条件**：
- 涉及"左/右/前/后/上方/下方"的精确描述
- 多个物体需要在画面中被精确定位
- 用介词短语描述位置关系（"A next to B, behind C"）

**2025-2026 新模型缓解情况**：FLUX（T5-XXL）和 SD3（T5+CLIP）对空间理解有明显提升，因为 T5 有更强的句法理解能力。DiT 架构的全局 attention 也让模型更容易建立不同区域之间的空间关系。但仍不稳定——复杂三物体以上的空间排列依然是挑战。

---

### 问题 1.2：数量失控

**现象**：用户写"三只鸟停在电线上"，结果可能出现 2 只、4 只、5 只——很少精确是 3 只。

**背后原因**：数量概念对 CLIP 来说极其抽象。"three birds" 在 CLIP 的向量空间里和 "birds" 的差异很小——CLIP 更关心"是鸟"这个视觉概念，而不是"几只"这个精确数量。

**典型触发条件**：
- 数量 > 2 时错误率急剧上升
- 物体越小、越密集，数量越不可控
- 数量和其他约束叠加时（"三只红色的小鸟"）更容易出错

**2025-2026 新模型缓解情况**：FLUX 和 SD3 通过 T5 编码器对数量准确性有明显提升（尤其是 2-5 个物体的场景）。但超过 5 个以上的精确数量依然不稳定。DALL-E 3 通过 GPT-4 rewrite + 专门的 counting loss 做到了相对最好的数量控制。

---

### 问题 1.3：属性绑定错误（attribute binding）

**现象**：用户写"一个戴红色围巾的白猫和一个穿蓝色外套的男人"，结果围巾变蓝、外套变红，或者猫变成红色。

**背后原因**：这是 CLIP 编码器的经典短板——**属性和名词的绑定关系**在向量空间里是模糊的。"red scarf white cat blue coat man" 被编码后，"red"到底修饰谁、"blue"到底修饰谁，在 cross-attention 分配时经常出现"串色"。

**技术上叫**：attribute leakage / attribute binding failure。

**典型触发条件**：
- 多个物体 + 多个颜色（最经典的失败场景）
- 多个物体 + 不同材质
- 修饰词和被修饰词在句子中距离较远

**2025-2026 新模型缓解情况**：这是被研究最多的问题之一。Attend-and-Excite（2023）、Structured Diffusion（2023）从 attention 层做修复；FLUX/SD3 靠更强的 T5 编码器 + DiT 全局 attention 大幅缓解。DALL-E 3 通过把每个物体拆成独立子描述（GPT-4 做拆分）基本解决。但在开源模型中，2 个以上物体各自有独立属性时仍有 20-30% 的失败率（待验证：具体数字来自社区测试，非论文）。

---

### 问题 1.4：否定词被忽略

**现象**：用户写"no sunglasses"或"a room without windows"，结果偏偏出现了太阳镜和窗户。

**背后原因**：CLIP 对否定语义（negation）几乎没有学习能力。"no sunglasses" 在向量空间里反而和 "sunglasses" 很接近——因为 CLIP 训练数据中很少有"图里没有 X"这种标注。模型看到 "sunglasses" 这个 token 就兴奋，不管前面有没有 "no"。

**典型触发条件**：
- 使用 "no"、"without"、"not"、"除了" 等否定词
- 想排除某个元素但又提到了它

**2025-2026 新模型缓解情况**：T5 编码器对否定语义的理解明显强于 CLIP。FLUX 在 "without X" 类 prompt 上的成功率比 SD 1.5 高很多。但这个问题没有被彻底解决——negative prompt（把不想要的东西写在另一个框里，用 CFG 的方式做排斥）仍然是更可靠的做法。

---

### 问题 1.5：中文 prompt 的先天劣势

**现象**：同样的语义，用英文 prompt 出图质量和准确率明显高于中文。

**背后原因**：
- CLIP 训练数据以英文图文对为主，中文覆盖极少
- 中文分词比英文更复杂——"赛博朋克"可能被切成多个 token，每个 token 单独的语义都偏了
- 很多中文文化概念（汉服、国风山水、工笔画）在 CLIP 的训练分布里是边缘区域

**2025-2026 新模型缓解情况**：阿里 Wan 2.1（umT5-xxl 多语言编码器）和通义万相显著改善了中文理解。字节 Seedream 系列也针对中文做了优化。FLUX 因为底层还是英文 T5，中文支持需要配合翻译或中文微调版本。

---

### 问题 1.6：Token 截断——长 prompt 后半段"消失"

**现象**：用户写了很长很详细的 prompt，但出来的图只反映了前半部分内容，后面的描述完全没体现。

**背后原因**：SD 1.5 的 CLIP 只有 77 个 token 的窗口。超出部分被物理截断，模型压根没看到。即使 SDXL 有"分段编码"的 hack，后段的注意力权重也会被稀释。

**典型触发条件**：
- Prompt 超过 77 token（约 30-50 个英文单词）
- 重要描述放在 prompt 末尾
- 堆叠大量修饰词和场景描述

**2025-2026 新模型缓解情况**：FLUX 支持更长的 token 序列（T5-XXL 支持到 512 token）。DiT 架构天然对长序列更友好。这个问题在新模型上基本解决——但"注意力被稀释"（虽然模型看到了，但太多信息让每个词的影响力都变弱）仍然存在。

## 第二关：画家失败（生成网络层）

即使翻译官完美理解了意图，画家（UNet/DiT）在执行时也会出错。

### 问题 2.1：Attention 争夺——加一个词，另一个词就弱了

**现象**：用户给 prompt 加了一些细节描述后，原来画得好的部分反而变差了。比如原来"一只猫"画得很好，加了"在阳光下的花园里"后猫的细节反而模糊了。

**背后原因**：Cross-attention 是一个零和博弈。所有 token 在竞争每个图像 patch 的注意力。新加的 token 必然稀释已有 token 的影响力。尤其当新加的词是视觉上"重"的词（大场景、强氛围词），它们会抢走很多注意力预算。

**典型触发条件**：
- 在已经稳定的 prompt 上追加描述
- 加入场景/环境词（这些词视觉信息量大，吃注意力多）
- Prompt 总 token 数接近上限时

**2025-2026 新模型缓解情况**：DiT 的全局 self-attention 让所有 token（图像+文本）在同一个 attention 池里交互，理论上信息流动更充分。但 attention 争夺的本质没变——只是从"局部争"变成了"全局争"。

---

### 问题 2.2：概念混合体（concept blending）

**现象**：用户写"一只猫旁边有一只狗"，结果出来一只"猫狗混合体"——有猫的脸但狗的身体，或两者特征混在一起。

**背后原因**：当两个语义相近的概念（都是动物、都是四足）出现在同一个 prompt 里时，cross-attention 可能把两者的特征"混"到同一个空间区域。模型没有学到"这两个必须是分开的实体"——它只知道这两个概念的向量都要在图中体现。

**典型触发条件**：
- 两个同类物体需要共存（两个人、两只动物、两个杯子）
- 物体语义相近且没有明确的空间分隔描述
- 没有使用 "and" 或明确的空间词做区隔

**2025-2026 新模型缓解情况**：FLUX 和 SD3 在多实体分离上有改善（DiT 全局 attention 帮助模型更好地区分不同实体的空间位置）。但复杂多实体场景（3 个以上同类物体各自独立）依然是挑战。

---

### 问题 2.3：CFG 失控——要么太听话要么太疯狂

**现象**：
- CFG 低（<5）：图好看但和 prompt 关系不大，模型在"自由发挥"
- CFG 高（>15）：图严格按 prompt 来但画面过饱和、出现伪影、色彩失真
- 没有甜点区：有些 prompt 在 CFG 7 时最好，有些要 12，没有通用值

**背后原因**：CFG（Classifier-Free Guidance）本质是"把有条件生成和无条件生成的差异放大 N 倍"。放大倍数（CFG scale）大了，prompt 的信号被过度增强，小到一个不重要的词也可能被放大成主导特征；放大倍数小了，prompt 被噪声淹没。

**典型触发条件**：
- 复杂 prompt + 高 CFG = 各种特征互相拉扯导致伪影
- 简短 prompt + 低 CFG = 完全不听话
- 用户无法直觉判断该用什么 CFG 值

**2025-2026 新模型缓解情况**：FLUX 使用了 guidance distillation，用一步 guidance 替代传统 CFG 的两步计算，减少了 CFG 过高时的伪影问题。SD3 也对 CFG 做了改进。整体趋势是模型本身变得更"听话"，不再需要极端 CFG 值。

---

### 问题 2.4：手/脚/文字——结构性生成盲区

**现象**：生成的人物手指数量不对（6 指、3 指）、文字写错或变形、精细结构（自行车辐条、楼梯栏杆）混乱。

**背后原因**：
- **手指问题**：训练数据中手的姿势极其多样，而手的结构约束（必须 5 根手指、关节角度有限）没有被编码进模型。模型只是在"统计上最可能"的特征中采样。
- **文字问题**：CLIP 不理解字母的视觉形状，它只把文字当语义 token 处理。模型要"画出文字"需要在训练数据里见过大量含文字的图片。
- **精细结构问题**：潜空间的压缩率（8x）意味着 512px 的图在潜空间只有 64px，精细结构在这个分辨率下被损失。

**2025-2026 新模型缓解情况**：
- 手指：FLUX/SD3 明显改善（更大的训练集 + DiT 更强的全局一致性），但未完全解决
- 文字：SD3 加了 T5 编码器后能写简短英文；FLUX 的文字生成在英文场景下已经相当可靠；中文文字仍然困难
- 精细结构：高分辨率模型（FLUX 原生支持 1024px+）缓解了潜空间压缩损失

---

## 第三关：Prompt Rewrite 失败（LLM 扩写层）

这是 2024-2025 年出现的**新问题类型**——随着 Sora、可灵、即梦等产品普遍引入 LLM 对用户 prompt 做自动扩写，一层新的信息黑盒被加入了链路。

### 问题 3.1：过度发挥——加了用户没要的东西

**现象**：用户写"一只猫在窗台上"，被扩写成"一只优雅的缅因猫慵懒地躺在阳光洒落的白色大理石窗台上，旁边有一盆薰衣草……"——结果出来的图里全是用户没要求的细节。

**背后原因**：扩写用的 LLM（GPT-4V、Qwen 等）被训练成"写出丰富详细的描述"。它的优化目标是生成流畅详细的文本，不是保真用户意图。当用户 prompt 很短时，LLM 不得不"猜"很多细节来填充，这些猜测可能完全偏离用户想象。

**典型触发条件**：
- 用户 prompt 很短（<10 个词）
- 用户有明确但隐含的意图（"我就想要简单的"）
- 用户第二次修改时，上一次的扩写结果不透明，无法针对性调整

---

### 问题 3.2：丢失关键约束——用户说了但被改没了

**现象**：用户写"黑白照片风格，一个老人在雨中"，扩写后变成了彩色描述——"黑白"这个关键约束在扩写过程中被稀释或丢失。

**背后原因**：LLM 扩写时会基于自己的"世界知识"补充内容，有时会覆盖用户的明确约束。尤其是当用户的约束和 LLM 认为"好看"的默认审美冲突时（比如 LLM 倾向于描述彩色、明亮的场景）。

**典型触发条件**：
- 用户的风格约束是"非主流"的（黑白、低饱和、故障风、粗糙质感）
- 用户的否定约束（"不要 X"）在扩写时被忽略
- 用户使用了行业术语但 LLM 翻译成了不精确的自然语言

---

### 问题 3.3：一致性中断——改一个词，整个扩写都变了

**现象**：用户在第一轮生成后只想改"把背景从白天换成夜晚"，但因为 prompt rewrite 的随机性，新的扩写结果整体都变了——人物服装变了、构图变了、风格也偏了。

**背后原因**：LLM 的生成是自回归的，改一个词可能触发完全不同的生成路径。如果扩写没有"锁定不变部分"的机制，每次修改都相当于重新写一篇描述。

**典型触发条件**：
- 用户做微调而非重写
- Rewrite 模型没有 diff 感知能力（不知道用户只改了哪里）
- 没有 seed/template 机制锁定扩写中的不变部分

---

## 第四关：多输入冲突（混合控制层）

当用户同时提供文字 prompt + 参考图 + ControlNet 结构图 + LoRA 时，这些输入之间可能打架。

### 问题 4.1：Prompt 和参考图矛盾

**现象**：用户上传一张红色连衣裙的参考图，prompt 写"蓝色礼服"，结果出来一个紫色的——模型试图"折中"两个矛盾的信号。

**背后原因**：IP-Adapter 的图像向量和 prompt 的文本向量在 cross-attention 里各自有一组投影矩阵。当两者传递的信号冲突时，最终结果取决于 `ip_adapter_scale` 的权重设置。但用户通常不知道这个参数的存在——他们只是"传了一张图+写了一句话"，期望模型自动判断该听谁。

**典型触发条件**：
- 参考图和 prompt 在颜色、风格、主体上有冲突
- 用户意图是"借参考图的形式但换内容"，但没有明确分离
- `ip_adapter_scale` 设置在 0.4-0.6 的模糊区间

---

### 问题 4.2：ControlNet 太强压死 prompt

**现象**：用户给了一个人物的 pose 骨架图 + prompt 描述"一只猫"，结果出来一只"人形猫"——猫被强制拉成了人的姿态，看起来非常怪异。

**背后原因**：ControlNet 的结构约束在生成过程中是"硬约束"——它在每一步去噪时都在强制注入结构信息。当结构约束和内容语义严重冲突时，模型会试图"在不可能的结构里塞不可能的内容"，结果就是畸形。

**典型触发条件**：
- 结构图的主体类型和 prompt 描述的主体类型不匹配
- `controlnet_conditioning_scale` 设置过高（>0.8）
- 结构约束过于精细，留给 prompt 的自由度不够

---

### 问题 4.3：多个 LoRA 互相干扰

**现象**：用户同时加载了"水彩风格 LoRA"和"赛博朋克风格 LoRA"，结果不是两种风格的和谐融合，而是画面割裂或出现伪影。

**背后原因**：每个 LoRA 都在修改 cross-attention 层的权重。多个 LoRA 修改同一组权重时，它们的改动可能互相冲突——就像两个人同时在一张画上画不同的东西。

**典型触发条件**：
- 两个 LoRA 都修改了同一层的权重
- 两个 LoRA 训练时的基底模型版本不同
- 两个 LoRA 的风格方向正交或对立

---

## 第五关：视频特有的意图理解失败

视频生成在图片的所有问题之上，还多了时间维度的挑战。

### 问题 5.1：动作语义无法传达

**现象**：用户写"一个人慢慢转身"，结果人物只是站着不动，或者突然瞬移到转身后的姿态，没有"慢慢"的过程感。

**背后原因**：动作的时序信息（速度、加速度、节奏）很难从文字中精确提取。"慢慢"在向量空间里的表征非常弱——模型知道"转身"是什么样子（起始帧和结束帧），但对"慢慢"的理解取决于训练数据中类似动作的速度分布。

**典型触发条件**：
- 涉及速度/节奏描述（"慢慢"、"突然"、"渐渐"）
- 复合动作序列（"先走三步，然后转身"）
- 动作的幅度描述（"微微偏头" vs "大幅度转身"）

---

### 问题 5.2：物理规则违反

**现象**：杯子悬浮、水往上流、物体穿模、人物走路时脚不着地。

**背后原因**：模型没有内建物理引擎——它的"物理理解"完全来自训练视频中的统计规律。当场景组合在训练数据中罕见时（比如"宇航员在水下弹吉他"），模型没有足够的先验来推导正确的物理行为。

**典型触发条件**：
- 非常规场景组合
- 涉及液体、布料、碰撞等复杂物理
- 长时间（>5 秒）的连续物理过程

**2025-2026 新模型缓解情况**：Sora、可灵 1.6 在物理一致性上有明显进步（更多真实世界视频训练+更大模型容量），但仍远未解决。这被普遍认为是视频生成的"下一个天花板"。

---

### 问题 5.3：时序一致性崩塌——前后帧不连贯

**现象**：人物的衣服颜色在不同帧之间闪烁变化、背景物体突然消失又出现、人脸在不同角度下长相不一致。

**背后原因**：视频生成的 attention 需要在时间维度上保持一致——同一个物体在不同帧的表征应该连贯。但当视频变长时，temporal attention 的有效范围有限，远距离帧之间的一致性约束减弱。

**典型触发条件**：
- 视频时长 > 4-5 秒
- 画面中有缓慢的镜头平移（同一物体从不同角度被拍到）
- 物体/人物有大幅度运动

---

## 交叉问题：用户认知与系统行为的错位

以上是技术层的问题。还有一类问题来自用户对系统行为的错误预期。

### 问题 6.1："我说得越详细应该越准确"——但实际是注意力被稀释

**用户预期**：写得越详细，出图越精确。
**实际行为**：过长的 prompt 导致每个词的 attention 权重被稀释，关键信息反而被淹没。

### 问题 6.2："我改了一个词应该只变一处"——但实际是全局重新采样

**用户预期**：微调 prompt 只改变对应的画面局部。
**实际行为**：prompt 的任何改动都会影响整个 attention 分布，加上 seed 可能变化，结果面目全非。

### 问题 6.3："参考图就是我要的，文字是补充"——但实际优先级不确定

**用户预期**：传了参考图，模型应该以图为主，文字只是微调。
**实际行为**：取决于 ip_adapter_scale 和具体实现，有时候 prompt 会反过来压过参考图。

## 问题严重度排序（对产品体验的影响）

从"最让普通用户挫败"到"可以忍受"排序：

| 严重度 | 问题 | 原因 |
|---|---|---|
| **致命** | Prompt rewrite 改变意图（3.1, 3.2） | 用户完全不知道自己的话被改了，无法诊断 |
| **致命** | 微调不收敛（6.2, 3.3） | 用户失去对创作过程的控制感，只能"赌" |
| **高** | 属性绑定错误（1.3） | 出现频率极高，且用户难以通过改 prompt 修复 |
| **高** | 多输入冲突（4.1, 4.2） | 用户不知道各输入的优先级，无法预期结果 |
| **中** | 空间关系/数量（1.1, 1.2） | 有明确的 workaround（用 ControlNet 或分步生成） |
| **中** | 否定词失败（1.4） | 有 negative prompt 作为替代方案 |
| **低** | 手/文字问题（2.4） | 用户已经普遍认知"AI 画不好手"，容忍度高 |

## 给产品设计的启示

1. **最需要解决的不是"模型更强"，而是"失败可诊断"**。用户能接受 AI 出错，但不能接受"不知道为什么错、不知道怎么改"。

2. **Prompt rewrite 是最大的黑盒**。如果产品不暴露改写结果，用户永远在和一个看不见的中间人做斗争。至少应该让用户看到"模型实际理解的版本"。

3. **优先级透明化**。多输入场景下，让用户知道"现在谁的声音最大"——是参考图主导还是文字主导——比让用户猜 ip_adapter_scale 应该调多少有效得多。

4. **微调应该是局部的，不是全局的**。用户的心智模型是"改一处变一处"，但模型的行为是"改一处变全局"。产品需要在中间做翻译——要么锁定不变区域，要么让用户明确"哪些部分可以变"。
