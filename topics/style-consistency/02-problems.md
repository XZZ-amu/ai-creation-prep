# 风格统一：常见问题与失败模式

> 基于 [01-mechanism.md](./01-mechanism.md) 的六类机制，梳理用户在"批量生成保持一致"时会遇到的典型失败。每个问题都标注了现象、根因、触发条件，以及新一代方案是否缓解。

---

## 一句话总结

**批量一致性的失败不是"模型不够好"这一个原因——而是六种不同机制各有各的盲区，用户踩的坑取决于产品给他走了哪条路。** 产品设计的核心任务是：让用户走对路，或者在走错路时尽早告诉他"这条路到不了你想去的地方"。

---

## 问题分类总览

| # | 问题类型 | 一句话概括 | 影响面 |
|---|---|---|---|
| 1 | 风格跑偏 | 参考图的画风/色调/氛围没保住 | 所有方案都可能出现 |
| 2 | 角色走样 | 同一人物换场景后认不出 | IP-Adapter、Reference-only 高发 |
| 3 | 多 LoRA 叠加串味 | 叠两个角色/风格 LoRA 出怪物 | LoRA 用户群体 |
| 4 | 训练过拟合 | 训出来只有一个姿势/一个角度 | DreamBooth-LoRA 用户 |
| 5 | 视频帧间身份漂移/闪烁 | 前一秒是这张脸，下一秒变了 | 视频生成场景 |
| 6 | 非人物对象一致性差 | 宠物/物品/虚拟角色保不住 | 全场景（人脸之外） |

---

## 1. 风格跑偏：参考图的画风/色调没保住

### 现象

- 给一张水彩风参考图，批量出的图有的是水彩、有的变成了写实风
- 参考图是暖色调，输出图里冷暖混杂
- 第 1-3 张像参考图，第 4 张开始画风飘了

### 根因

| 方案 | 为什么会风格跑偏 |
|---|---|
| IP-Adapter | `ip_adapter_scale` 偏低时，文字 prompt 中的风格关键词（或模型默认风格）会盖过参考图信号。CLIP 编码更侧重语义而非风格细节 |
| Reference-only | self-attention K/V 共享只传递"大致氛围"，具体的笔触纹理、线条特征传递力度弱 |
| DreamBooth-LoRA | 如果训练数据的风格不统一（比如 5 张训练图有 3 张暖色 2 张冷色），LoRA 学到的风格本身就不一致 |
| Kontext | 单张参考图条件下，DiT 的 self-attention 会在风格和内容之间做权衡——内容复杂时风格让步 |
| StoryDiffusion | 批内一致但"一致到哪种风格"是模型自己决定的，和用户预期可能不匹配 |

### 触发条件

- prompt 里有明确的风格词且与参考图风格冲突（如参考图是 anime，prompt 写 photorealistic）
- 使用 IP-Adapter scale < 0.5
- 参考图本身画风不明显/混合风格
- 批量生成时不同 prompt 的内容复杂度差异大（简单场景风格保住了，复杂场景风格丢了）

### 新一代方案是否缓解

- **FLUX Kontext 多图模式**：可以用一张图做角色参考 + 另一张图做风格参考，分离控制，显著减少风格跑偏。**部分缓解**。
- **StoryDiffusion**：批内风格统一性好，但必须接受模型选择的风格。**换了一种问题**（从"跑偏"变成"不可控"）。

## 2. 角色走样：同一人物换场景后认不出

### 现象

- 给一张断胡须橘猫参考图，生成 10 张——毛色对了，但胡须、耳朵缺角、斑纹等个体特征都不一样
- 人脸参考图做 AI 写真，正脸很像，一到侧脸或仰头角度就崩
- 10 张图里的"同一个人"，单独看每张都不错，放一起看明显不是同一个人

### 根因

**本质问题**：文字-图像模型存的是"类别"而非"个体"。不同机制的保真精度不同：

| 精度档 | 对应机制 | 能保住什么 | 保不住什么 |
|---|---|---|---|
| 类别级 | IP-Adapter base、Reference-only | 是橘猫、是亚洲女性 | 哪只橘猫、哪个人 |
| 风格级 | IP-Adapter Plus | 画风+大致外形 | 个体细节 |
| 身份级（人脸） | InstantID / PuLID / FaceID | 五官、面部比例 | 极端角度/遮挡下 |
| 身份级（通用） | DreamBooth-LoRA、Kontext | 像素级个体特征 | LoRA 受训练数据限制；Kontext 有微漂移 |

### 触发条件

- 只传一张参考图 + 用 IP-Adapter base（CLIP 全局编码，信息损失大）
- 参考图主体小/不清晰（编码器提取的特征本身就模糊）
- 目标场景和参考图差异极大（如参考图室内近景，生成图要全身远景）
- 人脸方案遇到大角度侧脸、遮挡、或非正光照

### 新一代方案是否缓解

- **FLUX Kontext**：DiT 的 self-attention 跨 token 能力让参考图的细节传递比 CLIP 编码强得多。个体级特征保留度显著提升。**大幅缓解**，但仍有微漂移（论文坦承 minimal drift ≠ zero drift）。
- **PuLID 的对比对齐损失**：解决了"身份注入挤压画风"的问题，让身份和画风能同时保持。**对人脸大幅缓解**。
- 对非人脸主体，目前没有等价突破。

## 3. 多 LoRA 叠加串味

### 现象

- 启用角色 A 的 LoRA + 角色 B 的 LoRA，生成图出现"A 的脸 + B 的发型 + 不明来源的体型"
- 角色 LoRA + 风格 LoRA 叠加，角色面部特征被风格 LoRA 扭曲
- 三个 LoRA 同时启用时输出完全不可控——既不像 A 也不像 B

### 根因

LoRA 的本质是在注意力层插入低秩矩阵。多个 LoRA 同时启用时，**它们的矩阵直接相加**（或按权重加权相加）。这个加法没有"语义边界"——A 的 LoRA 学到的权重和 B 的 LoRA 学到的权重在同一个注意力层上叠加，相当于两个"指令"同时给画家，画家只能折中。

具体表现：
- 两个角色 LoRA 在面部相关的 attention head 上打架 → 五官混合
- 角色 LoRA 在"发型/体型"维度和风格 LoRA 的"线条/色彩"维度重叠 → 互相干扰
- 权重分配（如 A=0.7, B=0.3）只能粗略控制强度，不能控制"影响范围"

### 触发条件

- 同时启用 2+ 角色 LoRA
- 角色 LoRA 和风格 LoRA 叠加，且风格 LoRA 的 rank 较高（影响范围大）
- 两个 LoRA 训练时用了相似的基座模型和相似的训练配方 → 更容易在同一个 attention head 上打架
- LoRA 权重都设到 0.8 以上

### 产品影响

这是"AI 穿搭"类产品的天花板问题：用户想"用我的脸 LoRA + 某个画风 LoRA + 某套服装 LoRA"做组合——听起来合理但技术上很难干净实现。

### 新一代方案是否缓解

- **FLUX Kontext 多图参考**：不用 LoRA 叠加，用多张参考图分别指定角色和风格，走统一的 self-attention，避免了矩阵打架。**从根本上换了路径**。
- **LoRA 合并技术（如 LoRA Merge、DARE）**：在合并前做冲突检测和分辨率对齐，减少串味。**有改善但不根治**。
- **区域分 LoRA（如 Regional Prompter + 不同 LoRA 控制不同区域）**：工程上可行但用户门槛极高。**不适合 C 端产品**。

## 4. 训练过拟合：只有一个姿势/一个角度

### 现象

- 训了个橘猫 LoRA，生成 10 张——每张猫都是 45 度侧身看镜头（训练数据里最多的姿势）
- 训了个人物 LoRA，换 prompt 写"跑步""跳舞"——输出还是站姿
- 背景和参考图高度相似，即使 prompt 写了不同背景

### 根因

DreamBooth / LoRA 训练的本质是**让模型记住"[V] 这个 token 对应什么"**。如果训练数据只有 3 张同角度照片，模型学到的不只是"这只猫的外观"，还会把"这个角度""这个光照""这个背景"一起绑进去。

Prior preservation loss（DreamBooth 的核心技巧）本来是防这个的——让模型同时记住"普通橘猫长什么样"来对冲。但如果训练数据量太小（1-2 张）或姿势太单一，prior preservation 也兜不住。

### 触发条件

- 训练图 < 3 张
- 训练图姿势/角度/光照高度一致
- 训练步数过多（步数多 + 数据少 = 过拟合加速）
- 没开 prior preservation loss（或 class prompt 选得不对）
- Rank 选太高（rank=64/128 时 LoRA 容量大，更容易记住无关信息）

### 用户痛点

这是"一键训 LoRA"类产品的核心 UX 问题：
- 用户只有 1-2 张照片（手机里随手拍的）→ 训练数据先天不足
- 用户不懂什么是"姿势多样性"→ 5 张照片角度都差不多
- 训完后发现"只有一个姿势"→ 觉得产品不行

### 新一代方案是否缓解

- **FLUX Kontext / in-context reference**：零训练方案天然不存在这个问题——参考图只在推理时起作用，不会"记死"姿势。**完全绕过**。
- **数据增强自动化**：产品侧可以在训练前自动对用户上传的图做翻转、裁切、轻微姿态变换，降低过拟合风险。**有效但增加训练时间**。
- **InstantID / PuLID**：零训练人脸注入，完全不需要多姿势数据。**对人脸完全绕过**。

## 5. 视频帧间身份漂移与闪烁

### 现象

- 生成 5 秒人脸视频，前 2 秒脸型正常，第 3 秒眼睛变大了，第 4 秒又回来——"闪烁"
- 角色转头时鼻子形状突变
- 长视频（>3 秒）里角色"越来越不像参考图"——累积漂移
- 头发颜色/长度在不同帧之间跳变

### 根因

视频生成比图片多了时间维度。帧间一致性的难度来源于：

1. **时间注意力的局限性**：DiT 视频模型（如 CogVideoX、Wan 2.1）用 temporal attention 连接帧间信息，但注意力窗口有限——距离远的帧之间信息传递衰减
2. **身份信号逐帧衰减**：参考图的 ID embedding 在第 1 帧最强，随着生成帧数增加，模型越来越"忘记"参考信息
3. **运动和身份的耦合**：大幅度运动（转头、弯腰）会扰动身份特征——模型很难在"保持运动自然"和"保持身份不变"之间平衡
4. **VAE 解码抖动**：即使 latent space 里帧间一致，VAE 解码时的精度限制也会引入帧间微小差异，表现为"闪烁"

### 触发条件

- 视频长度 > 2-3 秒（帧数多，漂移累积）
- 大幅度运动或角度变化
- 参考图与目标动作差异大（参考图正脸，视频要转 90 度侧面）
- 低分辨率生成后放大（放大加剧帧间不一致的可见度）
- 多人同框视频（模型对不同人的 ID 管理更难）

### 新一代方案是否缓解

- **ConsisID 的频率分解**：低频（整体身份）注入浅层持续存在，高频（细节）注入每帧的 attention——保证"大体一直是这个人"。**对人脸显著缓解**，但极端角度仍有挑战。
- **首帧条件生成（Wan 2.1 I2V / FLF2V）**：用户指定首帧（甚至首末帧），帧间一致性由模型的时间建模能力兜底。**帧间漂移减少**，但模型仍可能在中间帧"松弛"。
- **非人脸主体**（宠物、物品）的帧间一致性：目前没有等价的成熟方案。**未缓解**。

## 6. 非人物对象一致性差

### 现象

- 用户上传自家柯基照片做参考，生成 10 张——毛色差不多但体型、斑纹、耳朵形状每张都不同
- 参考一个红色杯子，生成 10 张杯子——颜色对了但形状/图案/手柄各不相同
- 虚拟角色（2D 动漫、游戏立绘）做参考，生成结果的五官比例波动大

### 根因

**人脸有 ArcFace 等成熟的 ID 编码器——猫、狗、杯子、动漫角色没有。**

机制层面：
- PhotoMaker / InstantID / PuLID 都是围绕人脸识别模型训练的，对非人脸主体不适用
- IP-Adapter 用 CLIP 编码，CLIP 对物体的编码是类别级（"这是一只柯基"而不是"这是左耳有黑斑的那只柯基"）
- DreamBooth-LoRA 对非人脸也有效，但用户需要训练——对 C 端产品来说门槛太高
- FLUX Kontext 的 in-context reference 对非人脸好于 IP-Adapter，但保真度仍低于人脸

为什么人脸能做到而其他不能：
- 人脸有统一的拓扑结构（两眼一鼻一嘴）→ 识别模型可以学"身份不变量"
- 猫/狗的品种差异 > 个体差异 → 通用视觉编码器很难抓住"个体 vs 品种"的区别
- 物品形状高度多样 → 没有统一的"物品 ID 表示"

### 触发条件

- 参考主体不是人脸（宠物、物品、虚拟角色、建筑）
- 使用零训练方案（IP-Adapter / Kontext）而非 DreamBooth-LoRA
- 参考主体的区分特征很细微（两只柯基区别只在斑纹分布）
- 目标场景要求主体大幅度变形或旋转

### 新一代方案是否缓解

- **FLUX Kontext**：DiT 的 in-context reference 对物体细节的保留好于 CLIP 编码的 IP-Adapter，但仍达不到人脸方案对人脸的保真度。**有改善，不根治**。
- **专域模型（如宠物 ID 编码器）**：学术界有零星工作（如 AnimalID），但没有达到产品化可用的程度。**待业界成熟**。
- **DreamBooth-LoRA 自动化**：产品可以把"训 LoRA"这步自动化（用户传 5 张图 → 后台自动训 → 几分钟后可用）——即梦、美图秀秀在做这件事。**可行路径但有等待成本**。

## 跨问题的关联关系

```
风格跑偏 ←→ 角色走样
  ↑ 两者都是"参考信号传递不足"的不同表现——
    前者是美学维度丢了，后者是身份维度丢了

多 LoRA 串味 ←→ 训练过拟合
  ↑ 都是训练派方案的固有代价——
    前者是多个 LoRA 之间的矛盾，后者是单个 LoRA 内部的矛盾

视频帧间漂移 = 角色走样在时间维度的放大版

非人物一致性差 = 角色走样的极端形式（连人脸 ID 编码器都没有了）
```

## 从产品设计角度看：哪些问题可以靠产品解，哪些只能等技术

| 问题 | 产品能做的 | 必须等技术的 |
|---|---|---|
| 风格跑偏 | 提供风格预设 + 锁定参数；给用户预览首张结果后再批量 | 更强的风格编码器 |
| 角色走样 | 明确告知"传一张图只能保风格不保身份"；分流到训练路径 | 通用物体 ID 编码器 |
| 多 LoRA 串味 | 限制同时启用数量；改用 Kontext 多图参考替代 LoRA 叠加 | LoRA 正交化技术 |
| 训练过拟合 | 引导用户上传多角度照片；自动数据增强；降低训练步数 | 小样本学习突破 |
| 视频帧间漂移 | 缩短单段长度；提供"重新生成某帧"的修补功能 | 更强的时间一致性模型 |
| 非人物一致性差 | 对宠物/物品引导用户走训练路径；降低用户对一致性的预期 | 通用 ID 编码器 |

## 一张图：问题与方案的对应关系

```
用户想要的                    会踩的坑                      新一代缓解方案
─────────────               ─────────────                ─────────────
"批量出图画风统一"   →   风格跑偏（scale/编码器）    →   Kontext 多图分离参考
"同一个人 10 个造型"  →   角色走样（CLIP 丢身份）    →   PuLID / Kontext
"两个角色同框"        →   多 LoRA 串味              →   Kontext 多图参考
"训我家猫的 LoRA"     →   过拟合只一个姿势          →   零训练 Kontext 绕过
"生成人脸视频"        →   帧间闪烁/漂移            →   ConsisID 频率分解
"我家狗/这个杯子"     →   非人物保不住              →   待突破（目前最好是 Kontext + LoRA 兜底）
```
