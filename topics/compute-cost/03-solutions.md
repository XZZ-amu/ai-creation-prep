# 算力成本：各家降本路径对比

> 承接 [02-problems.md](./02-problems.md) 的七个矛盾。本文梳理业界降本的四层路径，对比各家策略，找出对抖音特效场景的启发。

---

## 一句话总结

**降本路径分四层：模型层（让模型本身更轻）、推理层（让跑模型更快）、架构层（让硬件更省）、产品层（让需求更少）。前三层是工程能力，第四层是产品能力——产品设计师最能发力的恰恰是第四层，但必须理解前三层才能知道第四层的边界在哪。**

---

## 降本路径全景

```
┌──────────────────────────────────────────────────────────────────┐
│ 模型层（让模型本身更轻）                                          │
│   蒸馏 · 量化 · MoE · 剪枝 · 架构搜索                           │
├──────────────────────────────────────────────────────────────────┤
│ 推理层（让跑模型更快）                                            │
│   缓存 · Batching · 投机解码 · 编译优化 · 调度                   │
├──────────────────────────────────────────────────────────────────┤
│ 架构层（让硬件更省）                                              │
│   端侧推理 · 混合部署 · 专用芯片 · 多产品共享                   │
├──────────────────────────────────────────────────────────────────┤
│ 产品层（让需求更少）                                              │
│   降频 · 降质 · 异步 · 预生成 · 缓存复用 · 用户引导             │
└──────────────────────────────────────────────────────────────────┘
```

## 第一层：模型层降本

### 1a. 蒸馏（Distillation）

**原理**：用大模型（teacher）的输出训练小模型（student），使小模型用更少的步数/参数达到接近的质量。

| 方法 | 提出方 | 降本效果 | 质量损失 | 适用范围 |
|---|---|---|---|---|
| LCM | 清华, 2023 | 步数 50→4-8（6-12×） | 微损，细节↓ | SD 1.5 / SDXL / 各种基底 |
| SDXL Lightning | **字节**, 2024 | 步数 50→1-8（6-50×） | 4步几乎无损，1步有损 | SDXL |
| LADD（FLUX schnell） | Black Forest Labs, 2024 | 步数 30→1-4（8-30×） | 4步优秀，1步可用 | FLUX |
| AnimateLCM | 上交, 2024 | 视频步数 50→4-8 | 中等损失，时序抖动 | AnimateDiff 类 |
| CausVid | MIT/Meta(?), 2025 | 视频步数压缩 | 待验证 | 视频模型 |
| Hyper-SD | 字节, 2024 | 1步极限蒸馏 | 偏高（模糊/伪影） | SDXL/SD 1.5 |

**字节的位置**：SDXL Lightning 和 Hyper-SD 都是字节发的论文——**字节在蒸馏领域是全球第一梯队**。这意味着抖音特效用蒸馏模型不需要等外部，内部可以持续迭代。

**天花板**：步数已经压到 4 步了，继续压到 1 步质量断崖。下一步增量只能从其他维度找。

### 1b. 量化（Quantization）

**原理**：降低权重/激活的数值精度，用更少的位数表示相同的参数。

| 精度 | 相对速度（vs FP32） | 质量损失 | 硬件要求 |
|---|---|---|---|
| FP32 | 1×（基线） | 0 | 任何 GPU |
| FP16 | 2×（工业默认） | 极微 | Volta+ |
| BF16 | 2× | 极微 | Ampere+ |
| **FP8** | **4×** | 微损，需校准 | **H100/H200**（原生支持） |
| INT8（W8A8） | 4× | 小损，需 PTQ/QAT | Ampere+ |
| INT4（仅权重） | 理论 8× | 明显损失 | 需特殊 kernel |

**关键洞察**：
- **FP8 是当前甜点**——H100 原生支持，无需特殊 kernel，速度直接翻倍（vs FP16）
- INT4 仅权重量化（GPTQ/AWQ 思路）在 LLM 上成熟，但在视觉扩散模型上**还不够稳定**——attention 层的 softmax 对精度敏感
- **混合精度是生产常态**：敏感层（attention Q/K、timestep embedding）保持 FP16，其他层走 FP8

**字节的优势**：大量 H100 集群 → FP8 红利最大化。但芯片管制风险意味着未来可能被迫用 H800（FP8 性能打折）或国产替代。

### 1c. MoE（Mixture-of-Experts）

**原理**：模型总参数量大（知识丰富），但每次推理只激活一部分专家（成本低）。

| 特征 | Dense 模型 | MoE 模型 |
|---|---|---|
| 总参数量 | 12B = 推理 12B | 总 48B，激活 12B |
| 单步 FLOPs | 与参数量成正比 | 约为总参数的 1/4-1/8 |
| 质量 | 受参数量限制 | 总参数大 → 知识更丰富 |
| 显存 | 只需装一个模型 | 需装全部专家（显存更大） |
| 调度复杂度 | 低 | 路由器设计关键 |

**行业案例**：
- RAPHAEL（商汤/港大, 2023）：space-MoE + time-MoE，视觉生成首个 MoE 成功案例
- 谷歌 Imagen 3：据传走 MoE 路线（未公开确认）
- 字节豆包系列：大概率走 MoE（字节 LLM 侧已确认 MoE），视觉侧待验证
- Mixtral 8×7B：LLM 侧 MoE 标杆，证明路线可行

**对抖音特效的意义**：MoE 让"总模型能力大"和"单次推理便宜"可以兼得——但需要工程能力把路由做好、显存管理做好。**字节在 LLM MoE 上的经验可以迁移到视觉模型**。

### 1d. 剪枝与架构搜索

**原理**：去掉模型中不重要的连接/层/头，或自动搜索更高效的架构。

| 方法 | 降本效果 | 质量损失 | 成熟度 |
|---|---|---|---|
| 结构化剪枝 | 20-40% | 小-中 | 中（研究阶段多） |
| 注意力头剪枝 | 10-30% | 小 | 中 |
| NAS（神经架构搜索） | 一次性找到更优结构 | 可能更好 | 搜索本身贵 |
| Token pruning/merging（ToMe） | 20-40%（减少 token 数） | 小-中 | 已有开源实现 |

**ToMe（Token Merging）**特别值得关注：在 DiT 里 attention 成本 ∝ token²，如果能安全合并 30% 的 token → attention 成本降约 50%。这是 DiT 时代的"新缓存"。

## 第二层：推理层降本

### 2a. 缓存复用

| 方法 | 适用架构 | 降本效果 | 原理 |
|---|---|---|---|
| DeepCache | UNet | 30-50% | 浅层激活跨步复用 |
| 帧间 KV Cache | 视频 DiT | 40-60%（视频场景） | 相邻帧 K/V 大量重叠 |
| Δ-DiT / Token Cache | DiT | 20-40%（待验证） | 只更新变化大的 token |
| Prompt Cache | 文本编码器 | 5-10% | 相同 prompt 不重复编码 |
| VAE Cache | 解码器 | 5-10% | 相同 latent 结构不重复解码 |

**DiT 时代的关键挑战**：DeepCache 在 SDXL 上很成熟，但 DiT 的全局 attention 让"哪些激活不变"这个假设更难成立。**DiT 缓存是 2025 年的活跃研究方向**。

### 2b. 批处理与调度

| 策略 | 效果 | 前提条件 |
|---|---|---|
| Static batching | batch=8 → GPU 利用率 ×3-4 | 同模型、同分辨率、同步数 |
| Continuous batching | 动态插入新请求 → 利用率 +20-50% | 需要推理框架支持 |
| 优先级调度 | 高优先级请求优先 | 业务分级（VIP/普通） |
| 预测式扩缩容 | 根据历史流量预分配 | 流量模型准确 |
| 跨产品调度 | 豆包低谷时 GPU 给特效 | 集群统一管理 |

**字节特有优势**：火山引擎的统一调度能力 + 多产品线 GPU 共享 → 利用率比单产品公司高 20-40%。

### 2c. 编译优化

| 工具 | 效果 | 备注 |
|---|---|---|
| TensorRT | 推理速度 +30-100% | NVIDIA 专用，需要编译 |
| torch.compile (Inductor) | +20-50% | PyTorch 原生，适配快 |
| ONNX Runtime | +10-30% | 跨平台 |
| FlashAttention 2/3 | attention 速度 ×2-3 | 显存也省 |
| xFormers | attention 优化 | 被 FlashAttention 逐步替代 |

**FlashAttention 对 DiT 特别重要**——DiT 的 attention 是计算瓶颈，FlashAttention 3（H100 定制优化）直接影响 DiT 模型的实际推理速度。

### 2d. 投机解码 / 推测推理

**原理**：用小模型快速预测多步，大模型验证——LLM 侧的投机解码（speculative decoding）在视觉领域的迁移。

- **图片生成**：用蒸馏的小模型跑前几步草稿，大模型只精修最后 1-2 步
- **视频生成**：低分辨率快速出全帧 → 关键帧用大模型精修 → 中间帧插帧

目前还在研究阶段，但方向明确。字节在 LLM 投机解码上有积累（豆包），迁移到视觉模型有基础。

## 第三层：架构层降本

### 3a. 端侧推理

| 方案 | 当前能力 | 适用场景 | 时间线 |
|---|---|---|---|
| 高通骁龙 NPU（8 Gen 3+） | SD 1.5 可跑，SDXL 勉强 | 简单风格滤镜 | 现在 |
| Apple ANE（M4） | SD 1.5 流畅，SDXL 较慢 | 端侧预览 | 现在 |
| 联发科天玑 AI 引擎 | SD 1.5 可跑 | 低端安卓 | 现在 |
| 未来旗舰 NPU（2026+） | 预期 SDXL 流畅，FLUX 勉强 | 复杂特效端侧化 | 2-3 年 |

**端侧推理的产品意义**：
- **成本 = 0**（用户设备算力免费）
- **延迟极低**（无网络传输）
- **隐私友好**（图片不出设备）

**端侧推理的限制**：
- 只能跑小模型（SD 1.5/SDXL 量化版），画质上限低
- 发热/耗电——手机跑 10 秒高强度推理，用户手感到烫
- 安卓碎片化——适配 1000 种芯片是噩梦
- **短期内不能替代云端，但可以分流简单需求**

### 3b. 混合部署（端云协同）

```
用户请求 → 路由判断
    │
    ├─ 简单特效（风格滤镜、颜色调整）→ 端侧 NPU（成本=0）
    ├─ 中等特效（AI 美颜、简单风格迁移）→ 边缘节点（靠近用户的轻量GPU）
    └─ 复杂特效（文生图/视频生成）→ 云端 GPU 集群
```

**这个路由判断本身就是产品决策**——哪些特效走端侧、哪些走云端，决定了成本结构。

### 3c. 专用芯片与国产替代

| 芯片 | 性能（vs H100） | 可用性 | 对字节的意义 |
|---|---|---|---|
| NVIDIA H100 | 1×（基线） | 受管制限制 | 当前主力 |
| NVIDIA H800（中国特供） | ~70-80% | 可购买 | 备选主力 |
| 华为昇腾 910B | ~50-60%（推理） | 国产替代 | 风险对冲 |
| 字节自研（传闻） | 未知 | 未落地（待验证） | 长期战略 |
| Google TPU v5 | ~90-100% | 仅限 GCP | 字节无法用 |

**芯片管制的产品影响**：如果 H100 供应进一步收紧，字节可能被迫：
- 更激进地用蒸馏/量化（用软件弥补硬件）
- 更依赖端侧分流（把简单任务推到手机）
- 更严格的算力配额管理（产品层降频/限量）

## 第四层：产品层降本

**这是产品设计师最能直接影响的层级。**

### 4a. 降频（减少生成次数）

| 策略 | 降本效果 | 用户感知 | 适用场景 |
|---|---|---|---|
| 每日免费次数限制 | 可控（硬上限） | 负面（被限制） | 不推荐用于主功能 |
| 引导一次成功（提升首次成功率） | 间接降 30-50% | 正面（更快达到目的） | 所有场景 |
| 预设模板（减少自由生成） | 大幅减少无效生成 | 中性（限制自由度） | 一键特效 |
| 渐进式预览（低质量预览→确认后高质量生成） | 只有确认的才花大成本 | 正面（可预知结果） | 复杂生成 |

**"引导一次成功"是最优雅的降本**——用户达到目标的路径更短，平台花的钱更少，双赢。

### 4b. 降质（接受更低的生成质量）

| 策略 | 降本效果 | 质量损失 | 对用户透明度 |
|---|---|---|---|
| 默认用轻量模型 | 3-5× | 有，但用户未必感知 | 透明 |
| 降低输出分辨率 → 超分 | 2-3× | 超分伪影 | 半透明 |
| 减少后处理环节 | 1.3-1.5× | 细节瑕疵 | 透明 |
| 高峰期自动降档 | 2-3× | 时好时坏 | 不透明（有风险） |

**关键原则**：降质可以做，但**不能让用户感知到"时好时坏"**——稳定的"还行"比波动的"有时很好有时很差"体验好得多。

### 4c. 异步（不实时出结果）

| 策略 | 降本效果 | 适用场景 | 用户接受度 |
|---|---|---|---|
| 排队机制 | 削峰填谷 | 复杂/视频生成 | 需要良好等待体验 |
| 推送通知（"你的视频做好了"） | 允许低优先级调度 | 视频特效 | 中等 |
| 预生成推荐（凌晨提前算好） | 把成本移到低谷 | 热门模板 | 正面（秒出） |
| 批量任务合并 | 提升 batch 利用率 | 后台任务 | 透明 |

### 4d. 缓存复用（产品层）

| 策略 | 降本效果 | 原理 |
|---|---|---|
| 相同特效+相似输入 → 返回缓存结果 | 极高（命中时成本=0） | 用户上传相似图+同模板 → 结果差异小 |
| 热门模板预渲染 | 首次用户秒出 | 提前生成 N 种变体 |
| 素材库复用 | 减少从头生成 | 预生成的半成品素材拼装 |

### 4e. 用户引导（减少无效生成）

| 策略 | 降本效果 | 产品形态 |
|---|---|---|
| 更好的 prompt 引导 | 减少"试错"次数 | 模板选择、参数滑块 |
| 预览缩略图（低成本小图预览） | 只生成用户确认的 | "先看小图再出大图" |
| 限制不合理输入 | 避免必定失败的生成 | 输入检测+提示 |
| 教育用户预期 | 减少因不满意重试 | 展示"AI 能做到什么" |

## 各家策略对比

### 字节跳动（抖音特效 + 即梦）

| 维度 | 策略 | 依据 |
|---|---|---|
| 蒸馏 | SDXL Lightning / Hyper-SD（自研） | 字节自己的论文 |
| 量化 | FP8（H100 集群） | 硬件优势 |
| MoE | 大概率采用（LLM 侧已确认） | 豆包 MoE 经验 |
| 调度 | 跨产品线 GPU 共享 | 火山引擎能力 |
| 产品策略 | 模型分档（特效用轻档/即梦用重档） | 产品定位差异 |
| 端侧 | 短期不走端侧（云端优先） | 安卓碎片化+画质要求 |

### 快手（可灵）

| 维度 | 策略 | 依据 |
|---|---|---|
| 蒸馏 | 跟进业界方案 + 自研（细节不明） | 可灵出图速度暗示有蒸馏 |
| 量化 | 推测 FP8（同样大量 H100） | 行业标配 |
| 视频 | 自研视频模型（可灵视频版）| 早于抖音推出视频生成 |
| 产品策略 | **积分/付费制**——转嫁部分成本给用户 | 可灵有明确的积分限制 |
| 差异化 | 图生视频路线（首帧确定→成本更低） | 可灵主打图生视频 |

### 腾讯（混元/即创）

| 维度 | 策略 | 依据 |
|---|---|---|
| 模型 | HunyuanVideo（13B 开源） | 2024 年底开源 |
| 蒸馏 | HunyuanVideo Turbo（待验证） | 跟进行业趋势 |
| 差异化 | 开源策略——让社区帮忙优化 | 降低自身优化成本 |
| 产品策略 | 微信生态内嵌（用社交分发覆盖成本） | 类似抖音的逻辑 |
| 硬件 | 自研"紫霄"芯片 + NVIDIA 集群 | 芯片管制对冲 |

### Stability AI（Stable Diffusion 系列）

| 维度 | 策略 | 依据 |
|---|---|---|
| 模型 | SD3（8B DiT）→ 开源社区承担推理 | 开源后优化由社区做 |
| 蒸馏 | SD Turbo / SDXL Turbo（官方蒸馏） | 论文公开 |
| 商业模式 | API 收费 + 模型授权 | 不承担终端用户推理成本 |
| 优化 | 主要靠社区（ComfyUI/Diffusers 生态） | 轻资产策略 |
| 困境 | 公司财务紧张，研发投入受限 | 2024 年裁员/资金链紧张 |

### Black Forest Labs（FLUX）

| 维度 | 策略 | 依据 |
|---|---|---|
| 模型 | FLUX.1（12B DiT） | schnell=蒸馏版 / dev=完整版 |
| 蒸馏 | LADD（潜在对抗蒸馏），4 步 | schnell 论文 |
| 商业模式 | API（通过 fal.ai/Replicate 等）+ 企业授权 | 不直接面向 C 端 |
| 优化 | guidance_scale=0（省掉 CFG 的 2× 成本） | schnell 的关键创新 |
| 开源 | schnell 开源（Apache 2.0） | 社区自行优化 |

### Midjourney

| 维度 | 策略 | 依据 |
|---|---|---|
| 模型 | 不公开（推测自研，非 SD 系） | 闭源 |
| 蒸馏 | 推测重度蒸馏（v6 出图速度极快） | 用户体感快 |
| 商业模式 | **纯订阅制（$10-60/月）**——用户承担成本 | 最简单的成本模型 |
| 优化 | 大量工程优化（Relaxed mode 低优先级） | Fast/Relaxed 分级 |
| 产品策略 | best-of-4 + upscale 分离 | 默认出 4 张低分辨率 → 选中后才 upscale |
| GPU 集群 | 自建（传闻 ~10,000+ GPU） | 规模效应 |

## 对抖音特效场景的七个启发

### 启发 1：字节的蒸馏优势是核心壁垒

字节自研了 SDXL Lightning 和 Hyper-SD，蒸馏技术是"自家的"。这意味着：
- 新基底模型出来后，字节可以最快蒸馏出轻量版
- 蒸馏质量的调优可以针对抖音特效场景定制（不用通用方案）
- **竞品（快手/腾讯）如果用外部蒸馏方案，响应速度和定制度都不如字节**

### 启发 2：FP8 + H100 是当前的"免费午餐"

FP8 量化在 H100 上几乎无损地把速度翻倍。这不需要任何产品改动，纯工程收益。**但这个红利依赖 H100 供应**——芯片管制是最大风险。产品设计师应该意识到：如果某天被迫换回 A100，所有 latency 承诺都要翻倍。

### 启发 3：产品层降本被严重低估

大多数讨论聚焦在模型层和推理层，但**产品层降本的杠杆可能最大**：
- "让用户一次成功"比"让模型跑更快" ROI 更高
- "预生成热门模板"比"优化推理框架"部署更快
- "渐进式预览"让用户自己过滤掉 80% 的无效生成

**产品设计师应该主动提出产品层降本方案，而不是等工程团队把模型优化到极致。**

### 启发 4：视频特效需要完全不同的成本策略

图片特效可以靠蒸馏+量化+batching 把单次成本压到可接受范围。视频特效即使全部优化拉满，单次成本依然是图片的 **20-50 倍**。

视频特效的降本策略必须包含产品层手段：
- 限制时长（3 秒而非 10 秒）
- 限制分辨率（480p 生成 → 超分）
- 限制频次（每天 N 次）
- 异步生成（排队/推送通知）
- 图生视频（首帧确定 → 成本更低）

### 启发 5：Midjourney 的"best-of-4 + upscale 分离"值得学习

Midjourney 的策略：默认出 4 张 256×256 低分辨率小图 → 用户选一张 → 才做高分辨率 upscale。

**低分辨率生成成本极低**（256px 的 attention 只有 1024px 的 1/16），让用户在便宜的阶段做选择，贵的阶段只跑一次。

抖音特效虽然不能让用户"挑 4 张"（太慢），但可以借鉴思路：**低成本快速预览 → 确认后才跑全流程**。

### 启发 6：端侧推理是 2-3 年后的结构性变量

当前端侧只能跑简单模型，但 NPU 性能每代涨 50-100%。2-3 年后：
- 旗舰手机可能流畅跑 SDXL 级别
- 简单风格滤镜/AI 美颜完全端侧化 → 这部分云端成本归零
- 复杂生成依然云端，但总流量减少

**产品规划要为端侧预留接口**——哪些特效未来可以迁移到端侧，现在就应该标注清楚。

### 启发 7：跨产品算力调度是字节独有优势

字节同时跑豆包（LLM）、即梦（重度视觉生成）、抖音特效（轻度视觉生成）、搜索推荐。这些产品的流量高峰不完全重叠：
- 豆包：白天办公时段高峰
- 特效：晚间娱乐时段高峰
- 搜索/推荐：全天平稳

**GPU 在产品间流转 = 提升整体利用率 20-40%**。这是纯技术公司（Stability AI、BFL）或单产品公司（Midjourney）做不到的。

| 优先级 | 手段 | 预期效果 | 实施难度 | 谁主导 |
|---|---|---|---|---|

## 还没搞清楚的部分

- 字节实际蒸馏方案是否已扩展到 FLUX/DiT 基底——Lightning 是 SDXL 时代的，新一代待确认
- 快手可灵的具体降本技术栈——公开信息很少
- 抖音特效是否已在用产品层缓存（相同输入返回缓存结果）
- MoE 在视觉模型上的落地进度——学术有方向但工业部署案例少
- 端侧推理在安卓生态的实际适配难度——碎片化问题多严重
- 视频蒸馏（CausVid 等）的真实质量——论文数字 vs 实际效果差距
- 字节自研芯片是否会影响量化策略选择
