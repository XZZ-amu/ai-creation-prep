# 算力成本：一次特效生成在后端到底花了多少 GPU·秒

> 写作说明：本文是 v2 范式的延伸——沿用 [intent-understanding/01-mechanism-v2.md](../intent-understanding/01-mechanism-v2.md) 的"画室"比喻体系，并新增两个角色（**画工**、**调度员**）。如果还没读过 v2，建议先读那篇——本文里"翻译官 / 画家 / 草稿世界 / 骨架图 / 风格手册"五个角色不再重新解释。本文聚焦"画一次画到底烧多少电"，以及"怎么烧得更省"。

---

## 一句话总结

**单次特效生成的算力成本 = 模型大小 × 推理步数 × 分辨率 × 帧数 × 候选张数。**
五个乘子里任何一个变了，成本可以差一个数量级。
**蒸馏（少几步）、缓存（复用激活）、量化（小精度）、批处理（拼车）四把工具，每把降一档；视频天然比图片贵 N 倍，N=帧数。**
对抖音特效来说，"上线哪个模型 + 走不走蒸馏 + 视频还是图片"不是技术问题，是**单次成本 × DAU**的财务账——算力是隐形产品决策。

---

## 贯穿案例

整篇文档围绕这一个抖音特效场景走完全程：

> 用户在抖音 AI 特效里上传自拍，选"皮卡丘变身"特效，**输出一段 5 秒的"我变成皮卡丘"短视频**。

每一个机制讲完，都会回到这个皮卡丘——这一关在它身上烧了多少电、能省多少。

## 画室扩编：成本视角的两个新角色

在 v2 五人组（翻译官、画家、草稿世界、骨架图、风格手册）基础上，再加两个：

| 角色 | 对应概念 | 它在干什么 |
|---|---|---|
| **画工** | 推理 step 数 / 单步 FLOPs | 画家每一笔的"工作量"——画 50 步还是 4 步、每一步多少计算 |
| **调度员** | Batch / KV cache / VAE tile / GPU 集群调度 | 决定不同用户的画作能不能拼车、缓存能不能复用——决定 GPU 利用率 |

**画家**的本事决定"这张画值不值"，**画工**和**调度员**决定"这张画烧了多少电"。
两者解耦——同一个画家，画工和调度员的策略不同，单次成本能差 5-10 倍。

## 第一关：成本由什么决定（成本公式拆解）

一次特效生成的算力账，几乎完全可以拆成五个乘子：

```
单次成本 ≈ 模型大小 × 步数 × 分辨率系数 × 帧数 × 候选张数
```

每个乘子单独看一下。

### 1a. 模型大小（参数量）

参数量决定**每一步**画家要做多少 FLOPs。粗略对照（数量级粗估）：

| 模型 | 参数量 | 类型 |
|---|---|---|
| SD 1.5 | ~860M（UNet 部分） | UNet 时代 |
| SDXL | ~3.5B 总（UNet + text encoders） | UNet 时代 |
| FLUX.1-schnell | **12B** | DiT 时代 |
| SD 3 大版 | **8B**（深度 d=38） | DiT 时代 |
| 视频模型（HunyuanVideo / CogVideoX 等） | 10B+（视频通常更大） | 时空 DiT |

参数量从 SD 1.5 到 FLUX 涨了约 14 倍——这意味着即便步数不变、分辨率不变，单步算力涨 14 倍。这是为什么"换 FLUX 算力账翻几倍"是工程同事第一反应反对的。

### 1b. 推理步数

每减一半步数，成本几乎线性减一半。这一档的进步是过去两年最显著的：

| 方法 | 步数 | 备注 |
|---|---|---|
| 经典 DDIM | 50 步 | 早年默认 |
| DPM-Solver++ | 20-30 步 | 高阶 ODE 求解器，质量基本无损 |
| LCM (Latent Consistency Models) | 4-8 步 | 一致性蒸馏 |
| SDXL Turbo / Lightning | 1-4 步 | 对抗蒸馏 |
| FLUX.1-schnell | **1-4 步**（4 步质量更稳） | 潜在对抗蒸馏 LADD |

LCM 论文（Tsinghua, 2023）证明在 latent space 做 consistency distillation 可以把 50 步压到 4-8 步、质量下降很小。SDXL Lightning（字节自己的论文，2024）用渐进式对抗蒸馏，把 1024px 文生图压到 1-8 步。FLUX-schnell 走 LADD（潜在对抗蒸馏），4 步出图，guidance_scale=0（连 CFG 都省了，又减一半）。

**画工视角**：步数从 50 减到 4 = 算力降 12 倍。**这是过去两年最大的"白送"红利**。

### 1c. 分辨率（Attention 的二次复杂度）

分辨率在 UNet 里成本是 O(H·W) 线性增长，但在 **DiT** 里因为 attention 是全局的、复杂度是 token 数的平方——分辨率 ×2，token 数 ×4，attention 成本 **×16**。

| 分辨率 | latent 大小（f=8） | DiT attention 相对成本 |
|---|---|---|
| 512×512 | 64×64 = 4096 token | 1× |
| 1024×1024 | 128×128 = 16384 token | 16× |
| 2048×2048 | 256×256 = 65536 token | 256× |

**这是为什么"高清出图"贵得不成比例**——同一个模型出 1024 vs 512，attention 部分差 16 倍。SD 3 论文里专门讨论了 token 数 vs 算力的 scaling 行为。

### 1d. 帧数（视频独占的 N 倍开销）

5 秒视频 @ 24fps = **120 帧**。视频模型用 **spatial-temporal attention**——一个 token 同时覆盖一小块画面 × 一小段时间。token 数变成 H×W×T，attention 复杂度是 (H·W·T)²。

粗略估算（数量级）：
- 单帧静态图 attention 成本 = X
- 5 秒视频 = X × 120 × 帧间 attention 系数（粗略 5-10x，因为时序 attention 不是完全二次方，有窗口/分块优化）
- 实际工程：5 秒视频 ≈ 单帧图的 **数十到上百倍**

**这是为什么"图片特效"和"视频特效"在产品定位上必须分家**——成本根本不在一个量级。

### 1e. 候选张数（best-of-N）

Best-of-N 直接 ×N。N=4 把成本拉到 4 倍。
**这是为什么抖音特效不能 best-of-N**——MJ/即梦走"出 4 张让用户挑"是合理的（用户是创作者愿意挑），抖音特效用户根本不会挑。所以抖音必须走 in-process pruning（中段剪枝）来提质量，参考 [quality-and-fallback/01-mechanism.md](../quality-and-fallback/01-mechanism.md) 里"出图中剪枝"那段。

### 回到皮卡丘案例

假设抖音特效用 **FLUX schnell 12B + 4 步 + 1024×1024 + 5 秒视频**：

```
单帧成本 ≈ 12B 参数 × 4 步 × (1024² 的 DiT attention)
          ≈ 数十 GFLOPs × 步数（粗估）
5 秒视频 ≈ 单帧成本 × 120 帧 × ~5-10 倍时空系数
        ≈ 单张图片成本 × 600-1200 倍
```

**用 GPU·秒粗估**（H100 fp8 估算，待实测）：
- 1024 单图：FLUX schnell 4 步 ≈ **0.5-1.5 GPU·秒**（H100 上）
- 5 秒视频（时空 DiT，4 步蒸馏过的视频模型）：**30-100 GPU·秒**（数量级粗估）

**注**：所有数字都是基于公开论文和博客 latency 反推的数量级，实际抖音线上数字未知（待入职查）。

## 第二关：降本工具箱

每一档都是"在不显著掉质量的前提下把某个乘子打下去"。

### 2a. 蒸馏（教练课——把步数从 50 压到 4）

v2 里讲过，speed-and-waiting 也讲过，这里只补**算力账**：

- LCM（一致性蒸馏，Tsinghua）：50→8 步，**约 6 倍加速**，质量微损
- SDXL Lightning（字节，2024）：1024px 上 1-8 步，**对抗+渐进式蒸馏**
- FLUX schnell：1-4 步，**LADD（潜在对抗蒸馏）**
- 字节自己的工具是 SDXL Lightning——值得记住，这是字节系产品的"母乳"

**画工视角**：教练课让画家从"画 50 笔细工"变成"画 4 笔写意"——单笔重要性指数级提高（错一笔很难救），但总工作量打下来了。

**比喻不准在哪**：教练课不是免费的——蒸馏过的画家**多样性会下降**（一个 prompt 出来 4 张高度相似），且对极小众风格 / OOD prompt 的鲁棒性更差。这是为什么 RLHF/DPO + 蒸馏的组合让"开箱好看"的代价是"风格趋同"。

**回到皮卡丘**：FLUX-schnell 4 步 vs 原版 FLUX-dev 30 步 = **省 7-8 倍单次成本**。这一关是抖音特效后端必走的红利。

### 2b. 缓存（DeepCache 类）

**机制**：UNet 的浅层激活在相邻去噪步之间变化很小——缓存下来，跨步复用。
- DeepCache（HKU/SenseTime, 2023）：UNet 浅层激活跨步复用，**降本约 30-50%**，画质几乎无损
- FasterDiffusion 等同思路变种

**物理类比**：画家画到第 30 步时，发现"这一笔的底色和第 28 步基本一样"，直接复制过来不重新调。

**比喻不准在哪**：缓存对 **DiT 适配性差**——DiT 的全局 attention 让"哪些层激活几乎不变"这个假设失效。目前 DeepCache 类技术主要在 UNet 上有效，FLUX/SD3 这一代 DiT 模型上的缓存方案还在演进（学术界有 Δ-DiT、Token Caching 等尝试）。

**回到皮卡丘**：如果用 SDXL 老路线，DeepCache 能在 4 步基础上再**省 30-50%**；如果用 FLUX schnell，这一关基本拿不到红利——4 步本身已经压到极限。

### 2c. 量化（FP16 / FP8 / INT8）

**机制**：把权重和/或激活的精度降低，单步 FLOPs 减半甚至 1/4。
- FP16：默认工业标准（vs FP32 节省一半）
- **FP8**：H100 上原生支持，**理论 2× 速度提升 vs FP16**——这是字节集群的关键红利
- INT8：进一步压；画质有可见损失，配合 LoRA 校准能挽回一些

**画家视角**：从用炭笔画（粗糙但快）vs 用 0.1mm 针管笔（精细但慢）——用炭笔时画家本事不变，速度更快。

**比喻不准在哪**：量化的真实复杂度在"哪些层敏感不能压、哪些层可以压"——attention 的 softmax、time embedding 这种关键层用低精度容易爆。生产上常见 **混合精度**（敏感层 FP16、其他层 FP8）。

**回到皮卡丘**：H100 上 FP8 vs FP16 ≈ **2× 加速**。这是为什么字节大量上 H100 集群——FP8 红利是 H100 独占的。

### 2d. 批处理（Batch 拼车）

**机制**：多个用户的请求拼到同一个 batch 里同时跑，GPU 利用率从 30% 涨到 80%+。

**调度员视角**：早高峰一个班车塞 4-8 个用户的画作一起送进画家工作台——画家的"单步开销"被多个用户均摊。

**比喻不准在哪**：拼车的前提是"乘客去同一个目的地"。**需求多样化（不同 LoRA、不同分辨率、不同步数、不同 ControlNet）会让 batch 散开**——一个用户带了一堆奇怪附加件，整车只能等他、或者只能他单跑。这是为什么"特效种类越多，batch 利用率越低"——产品扩特效库和后端拼车经济性是直接矛盾的。

vLLM、TensorRT-LLM 这些推理框架本来是 LLM 的，最近一年在扩散模型推理也大量借鉴（continuous batching、PagedAttention 思想）。

**回到皮卡丘**：高峰期 batch=8 拼车，单用户成本可以再降 3-4 倍；低峰期 batch=1 单跑，成本就是 1 倍。**这是为什么同一个特效用户感知成本"高峰反而便宜"——拼到了车**。

### 2e. KV Cache（DiT/Transformer 独有）

**机制**：DiT 的 attention 可以缓存 K/V 矩阵，跨步或跨帧复用。在视频场景帧间共享省得最猛——相邻帧的 K/V 大量重叠。

**比喻不准在哪**：KV cache 在 LLM 里是显存换计算，在扩散里因为去噪每一步 latent 都在变，K/V 不能完全照搬——是"近似复用"，需要工程上的精细控制。

**回到皮卡丘**：视频生成时帧间 K/V 复用是**视频降本的关键**——没这一档，视频成本根本压不下来。

### 2f. MoE（Mixture-of-Experts）

**机制**：把大模型拆成多个专家，每次只激活其中几个。**总参数巨大但单次推理 FLOPs 小**。

例子：RAPHAEL（SenseTime/HKU，2023）用 space-MoE + time-MoE，3B 参数 + 海量路径达到当时 SOTA。LLM 侧 Mixtral 8×7B、字节豆包系列大概率走 MoE。

**画家视角**：画室里其实有 8 个画家，但每张画只调出最对路的 2 个动手——墙上挂着 8 个画家的肖像（参数量），但工作台上同时只有 2 个（推理 FLOPs）。

**比喻不准在哪**：MoE 显存压力反而更大（8 个画家都得在场），调度（路由器）做不好会让某些画家闲死、某些画家累死，工程难度高。MoE 在视觉模型上还不像 LLM 那么主流——RAPHAEL、字节内部、谷歌 Imagen 3 据传是 MoE 路线，但开源模型生态主流还是 dense。

**回到皮卡丘**：如果抖音底层是 MoE 模型，**总模型容量大（视觉知识丰富）但单次推理便宜**——这是大厂自研模型 vs 开源模型的核心差异化点。**待入职查**：字节豆包视觉/即梦底层是 dense 还是 MoE。

### 2g. CPU offload + tile decode（推理框架技巧）

**机制**：显存不够时把部分层换出 CPU——慢但能跑。VAE tile decode：把大图切小块解码避免 VAE OOM。
ComfyUI、Diffusers 都支持这些 trick。ComfyUI 的"smart memory management"能让 12B 的 FLUX 在 1GB 显存上跑（极慢但能出图）。

**调度员视角**：画室空间不够大，把暂时用不到的画具搬去隔壁仓库——要用时再搬回来，慢。

**回到皮卡丘**：抖音云端 H100 集群显存大、不需要这个 trick；**端侧推理产品**（Lobe2 类、未来的"手机本地特效"）才高度依赖这一档。

## 第三关：模型档位选择（产品判断的物质基础）

### 一张表：成本 vs 质量 vs 适用场景

| 档位 | 模型 | 单次生成成本（粗估，H100 fp8） | 质量 | 适合什么 |
|---|---|---|---|---|
| 轻 | SD 1.5 + LCM | 0.1-0.3 GPU·秒 | 一般 | 纯本地推理（性能差的设备）/ 已被基本淘汰 |
| 中 | SDXL + Lightning | 0.3-1 GPU·秒 | 较好 | 抖音特效云端**性价比最优**默认档 |
| 重 | FLUX schnell | 1-3 GPU·秒 | 优秀 | 抖音特效**高质量模式** |
| 超重 | 视频模型（HunyuanVideo / CogVideoX / Wan / Sora） | **30-100 GPU·秒** | 视频原生 | 抖音特效**视频版**——每个用户都贵 |

数字标"粗估"——精确数字得看具体硬件、具体优化栈。但**数量级关系是稳定的**：图→视频跨越一个数量级。

### 为什么"算力是产品决策"

- "上线 FLUX 还是 SDXL"不是技术问题——是【单次生成成本 × 日活】决定的财务问题
- 用户感知的"快慢"和后端"花了多少 GPU·秒"**不是同一件事**——后端可以堆并发让用户感觉很快但成本巨大
- **视频特效 vs 图片特效不是"一个进阶版本"——是产品形态分家的分水岭**

## 第四关：抖音特效的算力账（产品视角）

### 1. 一次特效的成本结构（皮卡丘视频版拆解）

| 环节 | 相对成本 | 备注 |
|---|---|---|
| 用户上传自拍 → 人脸识别 + 活体 | 极低 | 检测器小模型 |
| 文字 prompt 编码（CLIP/T5/MLLM） | 低 | T5-XXL 4.7B 是大头 |
| **生成主流程**（视频去噪） | **最重的一段，>80%** | 参考第一关计算 |
| 修脸 / 修手 / 超分（每个独立） | 各几十毫秒-几秒 | 详见 quality-and-fallback |
| NSFW + 明星 + 政治敏感识别 | 中等 | 多分类器叠加 |

**总成本 ≈ 主流程 × 1.2-1.5**（兜底环节加成）。
即"主流程"决定了 80%+ 的账，但兜底环节也不是 0——加起来能到 20-50%。

### 2. 抖音的算力规模约束（拍脑袋估算）

假设：
- 抖音 DAU **1 亿**
- 特效渗透率 **5%**（保守）
- 人均 **3 次**特效/天
- → **1500 万次/天**

每次 1 GPU·秒（图片）→ 1500 万 GPU·秒/天 = ~4000 GPU·小时/天 = **~167 张卡常驻**
每次 30 GPU·秒（视频）→ ~5000 张卡常驻
每次 100 GPU·秒（重视频）→ ~17000 张卡常驻

**这是为什么"视频特效"和"图片特效"产品定位完全不同**——卡的数量差两个数量级，财务部门会直接卡死。

注：以上只是数量级粗估，实际数字受拼车、缓存、降级策略影响很大。**真实数字待入职**。

### 3. 抖音 vs 即梦的算力哲学差异

| 维度 | 即梦（创作者工具） | 抖音特效（社交内容） |
|---|---|---|
| 用户期望延迟 | 30 秒-数分钟可接受 | **1-3 秒**必须出 |
| 单次成本可承担 | 高（用户为创作买单） | 必须压（每天上千万次） |
| 多张候选 | best-of-4 是默认 | best-of-N 不可行（用户不挑） |
| 主用模型档位 | 重档为主（FLUX、视频模型） | 中档为主（SDXL+Lightning），重档为高质量模式 |

**同一份模型在两个产品上跑出来的"性价比"完全不同**——这是为什么字节内部要做"模型分档 + 产品分场景"。

## 第五关：未来趋势（一段就好）

- **蒸馏边际效益递减**——4 步以下质量明显崩，1 步路线（SDXL Turbo、Hyper-SD）质量起伏大
- **DiT 取代 UNet** 改变了所有缓存路线——DeepCache 类技术要在 DiT 上重做
- **视频模型成本是下一波关键瓶颈**——Sora 类模型每秒视频依然要数十秒推理；视频蒸馏（video LCM、CausVid）正在快速演进，但远没到图片端的成熟度
- **端侧推理（手机直接跑）**——目前不现实，Apple M4 / 高通骁龙 NPU 在追，3-5 年内可能改变格局；如果端侧成本变 0，整个抖音特效成本结构会重写
- **MoE + 推测解码**——LLM 成本曲线下来主要靠 MoE + 投机解码，视觉模型还在追

## 失败模式清单（用户视角）

- **"出图很慢"**：模型档位选过重 / 没用蒸馏 / 高峰期 batch 拼不上车 / 命中冷启动
- **"高峰期生成失败 / 排队"**：算力扩容跟不上突发流量，自动降级到轻量模型或排队
- **"同一个特效今天 1 秒明天 5 秒"**：算力调度策略——晚高峰自动切到更重的模型，或者反过来切到更轻
- **"高质量模式比标准慢 5 倍"**：步数 / 模型档位 / 是否走 best-of-N / 修补匠是否全开 的差异
- **"vip 给我开通了高质量但好像没差多少"**：要么后端没真切档（虚标），要么算力升档但用户感知不到（如步数 4 → 8 视觉差距小）
- **"视频生成总要排队但图片秒出"**：视频成本是图片的几十倍，并发能力差几十倍

## 产品设计师的落地洞察

1. **算力是隐形产品决策**——上线哪个模型、用哪种蒸馏方案、要不要走 best-of-N、视频还是图片，每一个决定背后都是【单次成本 × DAU】的财务账。**产品设计师要知道自己的设计提案在算力账上意味着什么**——同一份功能想法，"出 4 张让用户挑" vs "出 1 张+中段剪枝"，算力差 4 倍。这种判断不能等工程同事拍。

2. **视频和图片是两个完全不同的成本世界，产品设计要分家**。视频比图片贵一个数量级。"图片特效全量上、视频特效灰度上"是物理约束决定的——不是产品策略可以选的。**视频特效的 KPI、覆盖率预期、生成时长承诺，必须按"贵一个数量级"重新定**——直接套图片特效的指标会算崩账。

3. **"高质量模式"不应该让用户付费换算力**——成本应该被产品策略消化（限时高质量、特殊场景高质量、VIP 高质量但配额限制），不是简单"vip 给更贵模型"。**用户买 VIP 是买"被看见"，不是买"我多烧 5 倍 GPU"**。把算力账直接转嫁给用户的产品体验，不专业。

4. **抖音特效的成本约束反而催生了更精巧的工程**——蒸馏 + KV cache + 量化 + 拼车 + in-process pruning 的组合是"抖音特效后端"区别于即梦/竞品的核心能力。**这是壁垒不是缺陷**——成本压力倒逼出来的工程能力，是产品差异化的物质基础。

5. **算力成本的弹性应对策略——晚高峰 vs 凌晨，应该有不同的模型档位预设**。对用户透明（同一个特效入口）但成本差很大（凌晨给 FLUX、晚高峰给 SDXL+Lightning）。**做这种"对用户隐藏的算力调度"，需要产品设计师主动设计降级体验**——不能让用户感受到"今天画质明显更差"。

6. **"特效种类多样化"和"后端 batch 经济性"是矛盾的**。每加一个特效可能就多一种 LoRA / ControlNet 组合，batch 拼不上车。**产品规划特效库扩张速度时，必须把"算力调度复杂度"作为一个隐性成本算进来**——这是工程同事不会主动喊出来但实际承担的代价。

## 一张图：算力成本的五个乘子 + 七把降本工具

```
单次成本 = 模型大小 × 步数 × 分辨率 × 帧数 × 候选张数
            │       │      │        │       │
            │       │      │        │       └─ 抖音特效不能 best-of-N（in-process pruning 替代）
            │       │      │        └────────── 视频专属 N 倍开销（5 秒 = ~120 帧）
            │       │      └─────────────────── DiT attention 是 token² → 高清成本平方涨
            │       └────────────────────────── 蒸馏：50→4 步（白送 12 倍）
            └────────────────────────────────── SD1.5 → FLUX 涨 14 倍

降本工具箱：
  ┌─ 蒸馏（LCM / Lightning / schnell）         步数 ↓
  ├─ 缓存（DeepCache / 帧间 KV cache）          单步 FLOPs ↓
  ├─ 量化（FP8 / INT8）                         单步 FLOPs ↓
  ├─ 批处理（多用户拼车）                       GPU 利用率 ↑
  ├─ MoE（大模型小推理）                        单步 FLOPs ↓
  ├─ Tile decode / CPU offload                  显存压力 ↓（端侧）
  └─ 调度（高峰降档、低峰升档）                 整体 throughput ↑
```

## 还没搞清楚的部分（诚实标记）

- **抖音特效线上实际单次 GPU·秒数**——没有公开数据，待入职查
- **字节集群的 H100 vs A100 比例**——FP8 红利的覆盖面取决于这个
- **字节是否用 MoE 自研基础模型**——豆包视觉/即梦底层架构是 dense 还是 MoE 路线，公开信息几乎为零
- **蒸馏到 4 步以下的画质损失在抖音具体场景能不能接受**——1 步生成（SDXL Turbo / Hyper-SD）是否被采用、在哪些场景用，没看到公开案例
- **视频特效的真实成本是 30 秒还是 100 秒级别**——影响整个产品形态。如果接近 30 秒，可以全量上；如果接近 100 秒，永远是灰度/付费功能
- **In-process pruning 是否在生产中使用**——学术界的方向，不知道工业界推进到什么程度
- **抖音特效的"特效库扩张"vs"batch 经济性"如何权衡**——内部工程方案细节
- **DiT 上有效的缓存方案进展**——FLUX/SD3 时代的缓存技术目前落后 UNet 一代
