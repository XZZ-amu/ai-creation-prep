# 多模态对齐：图、视频、音频是怎么被一起生成的

> ⚠️ **写作来源说明**：本文写作时网络搜索工具不可用，事实层主要依赖模型训练记忆 + 一手论文 URL（论文 ID 真实，但未逐字核对内容）。**模型架构和机制层可信度高**（这些是论文里反复出现的标准事实），**具体数字、自评指标、最新版本细节请以论文原文为准**。审查时若遇到关键产品判断依据，建议打开来源 URL 核实。

> 这是 [intent-understanding/01-mechanism-v2.md](../intent-understanding/01-mechanism-v2.md) 体系的扩展。v2 讲了"用户文字怎么变成图"，本文回答的是上一层问题——**当输入不止文字、输出不止单张图时，画室里怎么协作**。所以本文沿用 v2 的画室五人组比喻，并新增一类角色（音频翻译官、参考图翻译官），其余设定不变。

---

## 一句话总结

**多模态生成不是"一个翻译官+一个画家"，是"多个翻译官同时给一个画家递话"，谁声音大谁就被听到。**
图像作输入是"递了一张已经画好的草稿让画家照着接着画"；
音频作输入是"给画家递了一份逐帧的节拍 / 嘴型时间表"；
文字、图、音频同时存在时，画家**不会平等听**——硬约束（结构、第一帧像素）压软约束（风格、prompt 语义），声音越具体越被采纳。
**多模态对齐的失败，本质是几个翻译官说的话没对上，或者画家根本没在听某一路。**

---

## 贯穿案例

整篇文档围绕这一个具体任务走完全程：

> **一张橘猫的照片 + 一段欢快的电子节拍音乐，让猫随着节拍点头、嘴里跟着节拍"哒哒"地动。**

这个例子覆盖三种典型多模态：图作为视频锚点（I2V）、音频驱动动作（音乐→节拍）、音频驱动嘴型（lip-sync 退化版——猫嘴型）。每一关讲完，都会回到这只猫。

## 画室角色升级版（基于 v2 的扩展）

v2 里画室有五人：翻译官、画家、草稿世界、骨架图、风格手册。多模态场景下角色没换，但**翻译官从一个变成多个**，画家收到的"参考资料"从一份变成多份：

| 角色 | 对应概念 | 在多模态里干什么 |
|---|---|---|
| **文字翻译官** | Text encoder（CLIP / T5 / MLLM） | 翻译用户写的 prompt（v2 已讲） |
| **图像档案员**（新增，沿用 [style-consistency](../style-consistency/01-mechanism.md) 的命名） | Image encoder（CLIP image / VAE encoder） | 把参考图压成画家能读的向量 / 直接灌进草稿世界当起点 |
| **音频翻译官**（新增） | Audio encoder（Wav2Vec2 / HuBERT / 音频特征） | 把音频按时间切成一帧一帧的特征向量，每一帧对应"这一秒该有的嘴型 / 节拍能量" |
| **画家** | UNet / DiT 主网络 | 从噪声反复去噪，生成视频每一帧。多模态下画家长出多个"耳朵"分别听不同翻译官 |
| **草稿世界** | Latent space | 视频的草稿世界比图多一个维度（时间） |
| **骨架图 / 风格手册** | ControlNet / IP-Adapter / LoRA | v2 已讲，多模态下还在 |

**每个比喻都不完全准**，每节会附"比喻不准在哪"。

## 第 1 关：图像档案员 —— 一张参考图怎么变成"视频的第一帧锚点"

### 干什么

用户给一张橘猫照片，希望视频"以这只橘猫为主角"。技术上有两条路把这张图灌进画家脑子：

1. **像素级锚点（concat / VAE 编码进 latent）**：把参考图过 VAE 压到草稿世界，**和噪声一起塞进画家工作的画布**——相当于第一帧不是从纯噪声起步，而是从"橘猫的草稿"起步。后续帧从这个起点继续去噪，画出来的猫和原图基本是同一只。
2. **语义级锚点（image embedding via cross-attention）**：把参考图过 CLIP image encoder 压成一组向量，画家像听 prompt 一样，**通过 cross-attention 听这组向量**。这条路保的是"长得像、风格像"，不保"像素级一致"。

绝大多数 I2V 模型（Stable Video Diffusion、CogVideoX-I2V、可灵 I2V、Wan I2V）**两条路都走**：第一帧灌 VAE latent 锚住像素，全程灌 image embedding 锚住语义。

### 比喻不准在哪

"递一张草稿让画家接着画"听起来温和，实际上**第一帧的 latent 是被强行钉死的**——前几帧画家几乎没自由度，只能从这个起点慢慢"放手"。所以 I2V 视频开头几帧最稳定，**越往后越容易飘**（主体长相变了、背景跑了、镜头逻辑乱了）——这是用户能感知到的"I2V 后段崩坏"。

### 回到我们的橘猫

- 参考图过 VAE 编码 → 第一帧的 latent 被这只橘猫的像素压住；
- 同一张图过 CLIP image encoder → 一组语义向量，告诉画家"主角长这样、毛是橘色、表情是这种"；
- 后续帧画家自由发挥，但 cross-attention 一直能听到"这是同一只猫"的提醒；
- **第一帧基本就是原图本身**，第 30 帧之后猫脸可能慢慢变形——这是 I2V 通病。

### 用户视角的失败现象

- **"开头很像，越到后面越不像"**——image embedding 在长视频里影响力衰减。
- **"参考图里有的细节（比如颈圈），生成视频里消失了"**——CLIP image encoder 是个语义压缩器，细小局部特征被压没了。
- **"第一帧完美，第二帧画风突变"**——第一帧用了像素锚点，第二帧只剩语义锚点，落差产生跳变。

## 第 2 关：音频翻译官 —— 一段音乐怎么变成"逐帧的动作时间表"

### 干什么

文字翻译官输出"一句话的语义矩阵"，输入是一段，输出是一段。
**音频翻译官输出的是一条时间轴**——25fps 的视频要 25 帧，音频翻译官就得给出 25 个特征向量，每一个对应"这一帧画面该是什么状态"。

具体做法（这是 EMO、Sonic、SadTalker 这类音频驱动模型的通用模式）：

1. 用 **Wav2Vec2 / HuBERT** 这种语音预训练模型把音频切成 20ms 左右的小窗，每个窗输出一个特征向量；
2. 这条向量序列**沿时间轴对齐到视频的每一帧**（重采样、插值）；
3. 画家的每一帧通过一个**音频专属的 cross-attention 层**听到对应时刻的音频特征。

### 比喻不准在哪

把音频说成"逐帧时间表"会让人以为画家精确按节拍画。实际上：

- **嘴型 / 节拍**：画家学到的是"这种音频特征 → 对应这种唇形 / 这种动作幅度"。学的是统计关联，不是物理因果。所以**唱中文 vs 唱英文**，模型见过英文训练数据多，中文嘴型就经常对不上（HeyGen、D-ID 早期版本中文唇形飘逸是常见吐槽）。
- **音乐节拍**：音频翻译官**对节拍本身不敏感**——它输出的是音色 / 能量包络，节拍要靠模型自己从能量曲线里"悟"。所以"猫随节拍点头"实际上是模型学过"音频能量峰 ≈ 动作幅度峰"这个统计关联——节奏感弱、不齐拍是常态。

### 回到我们的橘猫

- 这段欢快电子节拍 → Wav2Vec2 → 一条 30 帧/秒的特征向量序列；
- 画家在画第 12 帧时，听到"这一帧音频能量是峰值" → 倾向于让猫头处于动作幅度最大的位置（点头到底）；
- 画家在画第 18 帧时，听到"能量回落" → 猫头倾向于回到中位。

但这个机制对"猫嘴型"几乎无效——音频驱动模型的训练数据 99% 是人脸视频，它知道"这个音 → 人嘴这样动"，**不知道"这个音 → 猫嘴这样动"**。所以猫嘴大概率是按"通用张合节奏"动，不会和音乐内容关联。

### 用户视角的失败现象

- **"嘴型对不上"**：音频翻译官给的是统计关联，遇到训练数据少的语言（中文、方言）、罕见情绪（咆哮、低语）就崩。
- **"节拍是对的，但动作幅度不对"**：模型学了能量包络，没学"重拍 vs 弱拍"的音乐结构。
- **"音乐换了一首，画面没什么差别"**：音频特征里能量曲线相似的两首歌，模型给出的动作几乎一样——它对"风格"的感知比想象中弱。
- **"非人脸主体音频驱动效果很差"**：训练数据是人脸视频。猫、狗、卡通角色用音频驱动出来的嘴型 / 动作只是"按音节张合"，没有真实表情逻辑。

## 第 3 关：画家长出多个耳朵 —— 文字 / 图 / 音频同时存在时怎么仲裁

### 干什么

v2 讲过：cross-attention 是 prompt 影响图片的唯一通道。
多模态下，**画家有多个 cross-attention 层**——每一层听一路输入：

- 文字 cross-attention 层：听 prompt 矩阵；
- 图像 cross-attention 层：听参考图的 image embedding；
- 音频 cross-attention 层：听音频特征序列（每一帧画面对齐到对应音频帧）；
- 自注意力层：听自己（视频内部时空连贯性）。

每一帧画家在每个空间区域**对每一路都打一个相关度分**，分数加权后决定这一笔怎么画。

### 比喻不准在哪

"多个耳朵平等听"是错的。**模型架构里每一路 cross-attention 的权重不平等**：

- 第一帧的 VAE latent 锚点（像素级）几乎是**绝对约束**——画家在前几步去噪里没什么挣扎空间；
- 音频的影响范围是**局部的**——只影响嘴部、头部这些训练时和音频强相关的区域，不影响背景；
- 文字 prompt 的影响是**全局但弱的**——它能改风格、改语义、改氛围，但很难压过参考图给的"这是这只猫"。

### 多输入仲裁的经验顺序（公开材料 + 业界经验综合）

> **此处是综合判断，不是某一篇论文的明确结论，量化对比在公开材料里几乎没有。**

从强到弱大致是：

1. **第一帧 / VAE 像素锚点**（I2V 的参考图、视频续写的最后一帧）—— 硬约束，几乎不能违背
2. **ControlNet 类骨架信号**（pose、depth、canny）—— 形状硬约束
3. **音频驱动的局部信号**（嘴型、节拍动作）—— 局部硬约束
4. **IP-Adapter 类参考图语义**（风格、人脸特征）—— 软约束
5. **文字 prompt** —— 最软的约束，最容易被以上几路压过

所以"我给了参考图 + prompt 说'换成蓝色背景'，结果背景没变"——是 prompt 输给了参考图。
"我给了 ControlNet 骨架 + prompt 说'这个人坐着'，结果还是站着"——是 prompt 输给了骨架。

### 回到我们的橘猫

混合输入的仲裁结果（猜测）：

- 第一帧：参考图压死 → **就是原图那只橘猫**；
- 主体身份：image embedding 持续灌入 → **整段视频都是这只猫**；
- 嘴部 / 头部动作：音频特征驱动 → **跟着节拍动**；
- 背景、整体氛围：参考图 + prompt 共同决定，prompt 在这里有发挥空间；
- 如果用户加 prompt "夜晚、霓虹灯"——背景可能改一点，但猫本身（参考图锚定）几乎不变。

### 用户视角的失败现象

- **"我说什么 prompt 都没用"**：参考图 / ControlNet 太强，prompt 被压。设计师视角：要给用户一个"参考图影响强度"的旋钮（IP-Adapter scale 类似物）。
- **"嘴在对节拍，但人在原地不动"**：音频信号只 wired 到嘴部区域，全身动作得靠 prompt + 模型自己悟。这就是为什么"音频驱动 talking head"和"音频驱动全身动作"是两个不同难度的问题。
- **"加了多张参考图，模型只听第一张"**：多图作为参考时，不同模型聚合方式不同（有的取平均、有的拼接、有的串联），多图细节经常被合并成一个"平均向量"，丢了各自特征。

## 第 4 关：跨模态语义对齐的底层 —— CLIP / ImageBind 在干嘛

### 干什么

前面说"图过 CLIP image encoder、文字过 CLIP text encoder，画家都能听"——为什么这两路能放在一起被画家理解？

答案是：**它们被训练到了同一个向量空间里**。

- **CLIP**（OpenAI, 2021）：用 4 亿对"图 + 图说明"训练，让"狗的图"和"a dog"这两个不同模态的输入，**最终向量靠得很近**。这就是"图文对齐"。
- **ImageBind**（Meta, 2023）：把 CLIP 的两模态扩到六模态（图、文、音频、深度、热成像、IMU 运动数据），核心做法是**把图作为锚点**——让"狗叫的音频"和"狗的图"靠近，"狗叫的文字"和"狗的图"也靠近，于是"狗叫的音频"和"狗叫的文字"通过图作为桥梁，**间接也靠近了**。

### 比喻不准在哪

把这套机制说成"图、文、音都被翻译到同一种语言"是过度浪漫的描述：

- **对齐是统计意义上的，不是语义意义上的**——"温柔的音乐"和"温柔的画面"在向量空间里靠近，是因为它们经常在同一段视频里同时出现，**不是因为模型理解了"温柔"这个概念**。
- **对齐是单向锚定的**——ImageBind 用图当锚点训练，文 ↔ 音的对齐是间接的，所以**直接做"文字 → 音频生成"的效果显著弱于"文字 → 图像"**。
- **对齐空间里"近"≠"对应"**——CLIP 里"红苹果"和"红色"的向量是近的，但作为生成条件用，"红苹果"的向量塞给画家，画家可能画出红色的别的东西。这就是 v2 里讲的"属性飘移"在多模态层面的根因。

### 回到我们的橘猫

- 用户给的橘猫照片，过 CLIP image encoder → 一组向量 V_img；
- 假设这组向量在 CLIP 空间里和 "orange tabby cat" 这个文本向量 V_txt 靠近；
- 画家在画的时候同时听 V_img 和 V_txt，因为它们都指向同一片区域 → 画出来的还是这只橘猫；
- 但如果 V_img 和 prompt 里某个词意外靠近（比如"咖啡"——因为参考图里有杯子），prompt 里加 "在户外" 可能就被参考图悄悄压回了"室内"——**没人能看出问题出在哪**。

### 用户视角的失败现象

- **"我用文字描述音效，模型生不出对应画面"**：文 ↔ 音的对齐是间接的，比文 ↔ 图弱很多。
- **"两张参考图融合，结果是混乱的怪物"**：两张图在 CLIP 空间里方向不同，平均后落到一个谁都不像的位置。
- **"用音频驱动的视频里，画面和音频氛围不匹配"**：画面主要由 prompt + 参考图决定，音频只驱动局部动作——画面氛围 ↔ 音频情感的对齐链路并不强。

## 第 5 关：让画面里出现稳定的文字 —— SD3 / FLUX 突破在哪

### 干什么

让模型在画里写"happy birthday"，过去几年都画不对——不是字母错，就是连成一坨涂鸦。
2024 年开始 SD 3 / FLUX / DALL-E 3 突然能写对了。底层做了三件事：

1. **换翻译官**：从只用 CLIP 改成 **CLIP + T5**（甚至有的版本只用 T5）。T5 是个文本生成模型，**对字符串本身**（每个字母、字母顺序）的表示比 CLIP 强很多——CLIP 只学了"这是英文字母组成的词"，T5 学了"哪个字母在哪个位置"。
2. **架构改成 DiT（MMDiT）**：图块 token 和文字 token 在同一个 transformer 里**双向看彼此**——画家不只是查询文字翻译官，文字也"知道自己被画到哪里"。
3. **训练数据补齐**：大量带文字的图（招牌、海报、标语）+ 精确 caption。

### 比喻不准在哪

"换了翻译官就能写字"听起来像换零件，实际上**这是一次架构级的修补**：CLIP 的 77 token 上限本身就装不下复杂文本任务，T5 通常拉到 256+ token，加上 MMDiT 让文字 token 在图里有"位置感"。**T5 是必要不充分条件——光换 T5 不改架构和数据，效果有限**（这也是为什么早期一些"用 T5 的 SD 模型"出文字依然崩）。

### 回到我们的橘猫

如果想生成"橘猫戴一顶写着 BEATS 字样的耳机"：

- SD 1.5（CLIP only）：耳机上的字大概率是乱码；
- SD 3 / FLUX：BEATS 四个字母大概率正确；
- 中文（"猫"字）：FLUX 1 主版本的中文渲染不如英文，需要专门的中文版微调（部分国内模型的卖点）。

### 用户视角的失败现象

- **"英文写对了，中文崩了"**：训练数据里中文比例小，T5 对中文字符的内部表示也弱于英文。
- **"短文字写对，长文字开始崩"**：文字 token 数量超出注意力窗口的稳定区间。
- **"字写对了但风格不对（比如要手写体出印刷体）"**：T5 知道"哪个字在哪个位置"，不太管"什么字体"——字体由整体风格 prompt 决定。

## 多模态的 5 个失败点（用户视角版）

> 和 [intent-understanding/01-mechanism-v2.md](../intent-understanding/01-mechanism-v2.md) 的"5 个信息丢失点"是**两套独立的诊断框架**——v2 那 5 个是单模态文生图链路的环节（tokenization / encoder / cross-attention / CFG / sampling），本文这 5 个是**多模态特有**的失败点（多路翻译关、时序关、仲裁关、跨模态对齐关、模态特化关）。同名"翻译关"含义不同：v2 里只指文字翻译官，这里包括所有翻译官 + 图像档案员 + 音频翻译官多路并发。

整条多模态链路常见的失败现象，定位到关：

### 失败点 ① 翻译关：某一路输入压根没被听到

**现象**：
- 给了参考图，但生成视频的主角和参考图差别很大
- 加了音频，但视频画面和音频几乎没关联
- 加了 ControlNet 但 prompt 改了之后形状跟着变了

**根因**：模型架构里这一路的 cross-attention 权重不够强；或者训练时这一路的数据质量差。

### 失败点 ② 时序关：单帧对，连起来不对

**现象**：
- 每一帧单独看都正常，连起来角色脸在变 / 服装在飘
- 嘴型对得上，但表情切得不连贯
- 音乐节拍对，但动作幅度忽大忽小

**根因**：时序一致性靠**视频内部的自注意力**学，模型见过的训练视频不够多 / 不够长 / 不够多样，就学不出长程一致性。这是 I2V 后段崩坏的根因。

### 失败点 ③ 仲裁关：多路信号互相打架

**现象**：
- 参考图说"白天"，prompt 说"夜晚"，结果是个奇怪的暮色
- ControlNet 给一个站姿，prompt 说"坐着"，结果是个尴尬的半蹲
- 音频驱动嘴型，prompt 说"严肃地说"，结果是嘴在动但表情飘

**根因**：多路输入没有显式优先级，模型按训练时学到的隐式权重去调和。用户没办法精确控制"我希望参考图压 prompt"还是反过来。

### 失败点 ④ 跨模态对齐关：模态间不是真理解

**现象**：
- "悲伤的音乐"配上的画面是中性的
- "雷声"对应的画面是阴天但没有雷
- 双语 prompt 里中文部分被淡化

**根因**：多模态对齐是统计共现，不是语义理解。训练数据里没共现过的搭配，模型不会推理。

### 失败点 ⑤ 模态特化关：某一路对非典型主体失效

**现象**：
- 音频驱动人脸效果好，驱动猫脸 / 卡通角色就完全不对
- ControlNet 对真人姿态准，对动物 / 物体姿态飘
- 中文 prompt 比英文 prompt 出图差很多

**根因**：每一路输入的"理解能力"被训练数据的分布锁死。训练数据里 99% 是人脸 + 英文 + 真人姿态，超出这个范围模型就退化成"按音节张合 / 大致姿态 / 关键词堆砌"。

## 多模态的隐形旋钮（v2 没讲到的部分）

v2 讲过 prompt rewrite 是隐形黑盒。多模态场景下还有几个**用户感知不到但极大影响结果**的旋钮：

1. **参考图的预处理**：可灵 / 即梦 I2V 接到用户上传图后，会先做面部对齐 / 主体抠图 / 重 caption。"我上传的图模型实际看到的根本不是这张"——和 prompt rewrite 是同一类问题。
2. **音频的预处理**：HeyGen / D-ID 接到音频后会先做语音识别（ASR）→ 文本驱动嘴型 → 再融合音频 prosody。**所以"嘴型对不上"有时候不是音频问题，是 ASR 识别错了**。
3. **第一帧 / 最后一帧锚定策略**：有的 I2V 模型只锚第一帧（前向生成），有的锚首尾两帧（中间插值），策略不同效果差异巨大。可灵的"首尾帧"功能就是公开了这个旋钮。
4. **多 CFG 协同**：文字 CFG、图像 CFG、音频 CFG 是三个独立旋钮。一般只暴露一个"creativity"或"strength"给用户，背后映射到多个值——映射策略本身是产品决策。

---

## 产品设计师的几个落地洞察

1. **多模态不是"把更多输入塞进去"，是"在已经满当当的注意力里抢资源"**。新增一路输入意味着原有几路的影响力被稀释。产品设计第一要回答："为这次生成新增的这一路，值得让其他路退场多少？"

2. **优先级要用户可感、可调**。混合输入的"听谁的"问题没有自动最优解。把"参考图强度 / 音频强度 / prompt 强度"做成可见的旋钮，比把它们全自动化更靠谱（自动化适用于新手，资深用户必然要手动）。可灵在这件事上比即梦做得显式。

3. **音频驱动的能力边界要诚实标注**。给用户音频驱动功能时，要明确说"这个功能对真人脸最稳定，卡通 / 动物只能跟节拍动嘴"——而不是让用户上传猫照片+音频后困惑为什么效果差。

4. **I2V 的"后段崩坏"是技术天花板，要做产品兜底**。已知第 30 帧后越飘，**短视频本来就只 5-10 秒**这件事是产品级优势——抖音特效场景天然规避了这个问题。但对超过 5 秒的功能，要在 UI 上提示"长视频更容易飘"或自动切片重生成。

5. **多模态对齐失败的诊断要分路问**。不要给用户笼统的"重新生成"，而是问："参考图够清晰吗 / 音频和动作类型匹配吗 / prompt 是不是和参考图冲突？" 这是把 v2 提到的"分层诊断"扩展到多模态场景。

6. **跨模态语义对齐是上游能力，特效产品要假定它会失败**。"用音乐情绪驱动画面氛围"这种听起来很美的功能，CLIP/ImageBind 这一层的对齐还远没到位。设计要么避开这种诉求，要么用 LLM 做"音乐 → 文字 prompt"的中介转译（效果好得多）。

## 抖音视角：为什么这件事在抖音特效里特别关键

抖音 AI 特效的核心场景就是"用户给一张自己的照片 / 一段视频 + 选一个特效模板，秒出一段会动会唱的内容"——**这恰好就是多模态对齐的主战场**：

1. **照片→视频是基础技能**。抖音 AI 特效里大量"让你的照片动起来 / 让你跳科目三 / 让你和明星合拍"都是 I2V，参考图锁主体身份是命门——主角崩了用户就不会发。
2. **音频驱动是杀手级特化**。"让照片唱这首歌"是抖音生态里天然的内容母题（卡拉 OK、对口型、配音梗）。EMO 类技术（音频驱动头部 + 嘴型）落到产品上几乎是为抖音定做。
3. **特效叠加是抖音的传统强项**。在生成视频上叠贴纸 / 滤镜 / 字幕 / 转场是抖音和剪映的肌肉记忆——但和"AI 生成的视频"叠加时，**生成视频的运镜 / 主体位置是飘的**，传统特效的锚点（关键点跟踪、人脸 landmark）需要重新适配。
4. **"模板化"是抖音和即梦的边界**。即梦让用户写复杂 prompt 调出复杂结果，抖音特效让用户**只动一个变量**（自己的照片 / 自己的声音），其他全部固化。这种产品形态下，多模态对齐的健壮性比表达力重要得多——固化模板里所有失败点都得提前测过。
5. **算力限制下的多模态裁剪**。一个抖音特效要在十几秒内出片，跑全套 I2V + 音频驱动 + 后期叠加是算力大头。技术裁剪（小模型、低步数、缓存）对多模态对齐质量的影响是产品级 tradeoff——比单模态生图要复杂一档。

## 还没搞清楚的部分（诚实标记）

- **国内主流模型（可灵、即梦、Vidu）I2V 的具体条件方式**：是 concat 路 + cross-attention 双路，还是只有一路？强度怎么分配？官方资料几乎不公开，需要从工程师博客 / 技术访谈补。本文按"业界主流双路"假设来写。
- **音频驱动模型在动物 / 卡通主体上的真实效果**：文中说"训练数据 99% 是人脸所以猫嘴对不上"是合理推断，但没有量化对比。需要自己实测。
- **多输入仲裁的优先级排序**：本文给的强→弱顺序（第一帧锚点 > ControlNet > 音频局部 > IP-Adapter > prompt）是综合经验判断，**不是某篇论文的量化结论**。不同模型架构里实际优先级可能差异很大。
- **抖音 AI 特效产品里实际用的 I2V / 音频驱动模型**：是基于公开模型微调，还是字节内部模型（即梦同源）？是单模型端到端，还是流水线（先 I2V 出无音频视频 → 再音频驱动嘴型）？这是入职后才能知道的。
- **CFG 在多模态下的实际工作方式**：文 / 图 / 音三路 CFG 是各自独立施加，还是在统一去噪步内融合？不同实现差异大，本文给的是定性描述。
- **MMDiT 架构里图文双向注意力对多模态扩展的影响**：MMDiT 设计上让图 token 和文 token 互看，扩到音频是否一样有效？理论上可以，但公开材料里 audio-as-token 的 MMDiT 还较少。

```
└────────────────────────────────────────────┘
     │
     ▼
┌────────────────────────────────────────────┐
│  画家（DiT/UNet）长出多个 cross-attention  │
│  耳朵：                                     │
│   - 文字 attention 听语义                   │
│   - 图像 attention 听参考                   │
│   - 音频 attention 听时间轴（每帧对齐）      │
│   - 自注意力 听自己（时序一致性）            │
│  + 第一帧 latent 被强制压成参考图            │
│                                             │
│  ← 多路打架时按隐式优先级仲裁                │
│  ← I2V 后段越往后越飘                        │
│  ← 音频对非人脸主体退化                      │
└────────────────────────────────────────────┘
     │
     ▼
草稿世界（多了时间维度）→ VAE 解码 → 视频帧序列
```
