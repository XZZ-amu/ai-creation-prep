# 多模态对齐：抖音场景判断

---

## 1. 判断锚点

抖音特效的多模态组合不是技术炫技——**哪种组合能让普通用户最快做出"发出去有人看"的视频内容，就优先做哪种。** 核心约束是：输入门槛低（用户已有的素材）、产出门槛低（一条能发的短视频）、成本可控。

---

## 2. 核心判断

### 判断 1：抖音特效最高优先级的多模态组合是"图 → 视频"（I2V）

**理由**：
- 用户最容易获得的高质量素材是**照片**（手机相册里已有的）
- 抖音的内容形态是**视频**
- 图→视频是把"已有素材"变成"平台核心内容形态"的最短路径
- 竞品（可灵、Runway）都在这条路上发力，用户认知已建立

**对设计的影响**：
- I2V 应该是 AI 特效的第一优先级能力
- 产品形态：用户传一张照片 → 选一个特效（本质是运动/风格模板）→ 出一段 2-4 秒视频
- 技术选型：首帧条件（VAE latent concat）+ 运动模板（预设运镜/动作）

### 判断 2：第二优先级是"图 + 文字标签"，不是"图 + 自由文字"

**理由**：
- 抖音用户不会写 prompt——"图 + 自由文字描述"的交互对普通用户门槛太高
- 但"图 + 从预设选项中选一个标签"是可行的——比如传照片 + 选"变成古风""变成赛博朋克""变成油画"
- 本质是 prompt 被产品预设了，用户只做"选择"而非"输入"
- 这和纯 I2V 的区别是：I2V 只决定"怎么动"，图+标签决定"变成什么风格"

**对设计的影响**：
- prompt 永远不暴露给用户——内部用 prompt template + 用户选择的标签拼接
- 标签设计是核心产品工作——每个标签对应一套完整的 prompt + LoRA + 后处理参数
- 文字+图的冲突问题（02-problems P1）在预设标签模式下不存在——因为标签和风格本来就是配套设计的

### 判断 3：音频驱动（音乐→动作）是高价值但中优先级——等技术成熟

**理由**：
- 音频驱动视频是抖音的天然场景（抖音就是音乐+短视频平台）
- "上传一张照片 + 选一首歌 → 照片里的人随节拍动起来"——产品故事极其诱人
- 但技术现状不支持高品质产品化：节拍对齐不精确、中文唇形漂移、大幅度动作不稳定
- 音频驱动的视频生成成本远高于 I2V（每帧都要音频条件注入）

**对设计的影响**：
- 短期不做完整的"音频驱动生成"——技术不成熟 + 成本太高
- 短期可做的替代方案：AI 生成静态图/短视频 + 端侧根据音频节拍做简单动效（放大缩小、抖动、切换），成本约为零
- 中期等 EMO/Sonic 类方案在人脸说话场景成熟后，先做"照片开口说话"这个细分场景

### 判断 4：多模态冲突的解法是"产品侧消灭冲突"，不是"技术侧解决冲突"

**理由**：
- 多模态冲突（参考图风格 vs prompt 内容 vs 音频节奏）在自由创作工具里是核心难题
- 但在抖音特效里，**冲突根本不应该发生**——因为所有输入组合都是产品预设好的
- 用户不能同时上传一张油画风参考图 + 选一个"赛博朋克"标签——产品层就不应该给这种组合
- 模板设计时就把多模态条件配好（这张图用这个 prompt、配这个运动模板、配这个音乐节奏）

**对设计的影响**：
- 不需要在用户侧做"冲突仲裁 UI"（如可灵的"参考图强度滑块"）
- 模板设计工具需要支持多模态条件的联合预览——设计师在设计阶段验证组合效果
- 如果将来开放更自由的创作模式，考虑即梦式的"MLLM 理解用户意图 → 自动协调多模态输入"

### 判断 5：视频后段漂移是 I2V 的核心体验问题——用短片段 + 循环播放规避

**理由**：
- I2V 的通病是"开头几帧完美，后面越来越崩"——image embedding 影响力随时间衰减
- 对 2-3 秒的超短视频，问题不严重（前 2 秒都在"安全区"内）
- 抖音短视频天然适合循环播放——2-3 秒视频循环 3 次 = 用户感知 9 秒内容

**对设计的影响**：
- 视频特效的默认长度应该限制在 2-4 秒（技术安全区 + 用户注意力 + 成本可控）
- 设计时考虑"首尾衔接"——让视频适合循环播放（首帧和尾帧状态接近）
- 如果用户需要更长视频，用分段续写 + 接缝处理，而不是一次性生成长视频

### 判断 6：字节在多模态对齐上的独特优势是"数据量 + 自研视频模型"

**理由**：
- 抖音有全球最大的短视频数据集——图文配对、视频+音乐配对、用户行为数据
- 字节有自研视频生成模型（豆包视频底座），且有 PuLID 解决人脸一致性
- Wan 2.1 虽然是阿里开源，但字节可以基于自有数据做 fine-tune，效果可能超过开源版

**对设计的影响**：
- 应该最大化利用"抖音用户已有数据"作为多模态输入——照片库里的照片、拍过的视频、听过的音乐
- 技术评估时重点看字节自研方案和开源方案的差距——如果自研方案显著更好，值得为此承担更高的迭代成本

## 3. 产品设计建议

### 建议 1：多模态优先级路线图

| 阶段 | 组合 | 产品形态 | 技术成熟度 |
|------|------|----------|-----------|
| 短期（在做） | 图→视频 | 传照片→出短视频 | 成熟 |
| 短期（在做） | 图+标签 | 传照片+选风格→出图/视频 | 成熟 |
| 中期 | 图+音乐→视频 | 传照片+选音乐→照片随节拍动 | 半成熟（人脸说话可做） |
| 长期 | 视频+特效叠加 | 传视频→AI 在视频上叠加效果 | 待验证 |

### 建议 2：I2V 特效的标准产品结构

```
用户操作：选特效 → 传照片（或用相机拍） → 等 3-5 秒 → 看结果 → 发布
后台逻辑：照片 VAE 编码 → 拼接运动模板条件 → 云端生成 2-4 秒视频 → 质量评估 → 输出
```

关键体验指标：
- 从点击到看到结果：< 5 秒（含网络延迟）
- 视频长度：2-4 秒（可循环）
- 品质：人脸清晰 + 动作自然 + 风格鲜明

### 建议 3："音乐+照片"的低成本替代方案

在真正的音频驱动视频生成成熟之前，做一个"伪音频驱动"方案：
1. AI 生成一张风格化静态图（云端，一次性成本）
2. 端侧根据音频节拍做动效（放大/缩小/平移/抖动/粒子效果）
3. 用户感知到的是"照片在随音乐动"——但实际是端侧渲染，零边际成本
4. 这就是"AI 一次 + 端侧渲染多次"的混合方案

### 建议 4：为模板设计师建设多模态预览能力

内部工具需求：
- 设计师配置"照片模板 + prompt + 运动参数 + 音频节奏"后，能即时预览生成效果
- 支持批量测试不同类型的输入照片（正脸/侧脸/多人/宠物），验证模板鲁棒性
- 多模态冲突的 QA 在设计阶段就完成，不带到线上
